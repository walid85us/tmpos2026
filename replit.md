# Overview

This project is a multi-tenant SaaS frontend platform for small to medium-sized businesses. It provides a unified solution for business management, including Point-of-Sale (POS), employee and customer relationship management (CRM), inventory control, reporting, and marketing functionalities. The platform's core purpose is to streamline operations, offer actionable insights, and automate tasks through its scalable, customizable, and feature-rich design.

# System Architecture

## Frontend

The frontend is built using React 19, TypeScript, Vite 6, and Tailwind CSS v4.

## UI/UX Decisions

-   **Design System**: Consistent design language featuring rounded cards, a glass effect (`bg-white/80 backdrop-blur-xl`), and a primary color theme with a `ghost-border` utility.
-   **Typography**: `font-black uppercase tracking-widest` for labels.
-   **Animations**: Framer Motion for smooth UI transitions and consistent modal patterns.
-   **Notifications**: Toast notifications for user feedback.
-   **Accessibility**: Focus on semantic accessibility for forms, tables, and modals.

## Technical Implementations & Feature Specifications

-   **Authentication & Access Control**: Firebase Authentication with a 7-level hierarchical permission model.
-   **Tenant Management**: Full lifecycle management including provisioning, subscriptions, feature overrides, billing, and audit logging.
-   **Point of Sale (POS)**: Comprehensive system for cart management, diverse payment options, tax/discount calculations, customer management, repair intake, held orders, quick add stock, operator switching with RBAC, refund workflows, and warranty claims.
-   **Employee Management**: Features employee, role, time tracking, and payroll management with granular permissions.
-   **CRM (Customers)**: Enriched CRM with multi-field search, loyalty management, duplicate detection, and integrated order/invoice history.
-   **Invoices**: Full invoice lifecycle management with dynamic line items, searchable catalog, payment processing, supervisor PIN-authorized reopen, and configurable document templates.
-   **Services**: Manages a service catalog with CRUD, warranty configuration, bulk price editing, and dependency-aware delete confirmations.
-   **Repairs Module**: Full repair ticket lifecycle management with structured workflow, parts tracking, technician assignment, activity timeline, and financial summary.
-   **Reporting**: Dashboard with data visualizations.
-   **Onboarding**: Multi-step process for new tenants including plan selection, pre-configured settings, and an activation checklist.
-   **Document Template Editor**: Structured template builder for 5 document types with deterministic HTML generation, branding, and visual preview.
-   **Inventory Management**: Comprehensive system for managing various `StockItem` types (serialized, non-serialized, handset) with features like UPC/manufacturer/location tracking, min/max stock, serial number tracking, advanced filtering, stock adjustments, global movements log, inventory transfer lifecycle, stock counts, and trade-in management, all integrated with RBAC.
-   **Supply Chain Management**: Manages suppliers, purchase orders, Goods Received Notes (GRN), and Return Merchandise Authorization (RMA) with detailed tracking and financial recording, gated by RBAC.
-   **Shipping Center**: Centralized fulfillment module for managing shipments across all source documents, supporting various shipment types. Features include shipment listing with filters, detailed views, create/edit modals with package management, status transition workflows, tracking event timelines, carrier/service level configuration, cost tracking, and integration for data pre-population. Includes a provider adapter layer for external shipping carriers and a runtime mode architecture. Key features include:

    -   **Service Points & Pickup Requests (Phase 2)**: Two new operational layers inside Shipping Center, surfaced through a dedicated **Logistics** tab in the shipment detail panel (alongside Overview / Tracking & Events / Packages). **Service Points** let an operator pick a carrier-operated drop-off location for parcel handoff (UPS Access Point, FedEx Onsite, USPS PO Box, DHL ServicePoint, lockers, retail partners). **Pickup Requests** let an operator schedule the carrier to collect the parcel from the origin address. Provider/manual capability is honest: a `PROVIDER_CAPABILITIES` map declares per-provider support (`easypost`: service points + pickup; `shippo` / `shipstation`: pickup only); manual mode and unconfigured providers always return `{ servicePoints: false, pickupRequests: false }` with a visible amber lock notice naming the reason — the UI never pretends a capability exists. Every selection, request, and cancellation appends a structured event to the shipment timeline (`Service Point Selected`, `Service Point Cleared`, `Pickup Requested`, `Pickup Cancelled`) for full auditability. Sub-permissions `select_service_point`, `request_carrier_pickup`, `cancel_carrier_pickup` (parent domain `shipping`) gate the operator actions, and preview-mode safety blocks all writes via the existing `isWriteBlocked` check. Reverse-logistics integration: return shipments inherit the same Logistics tab through their linked shipment record — a return-leg shipment can use a service-point drop-off (common pattern: customer drops the return at a UPS Access Point) or a carrier pickup at the customer's address, governed by the same provider-capability and lifecycle rules. Lessons adapted from RepairDesk's ShipStation integration article: (a) treat service-point and pickup as **first-class shipment-attached records**, not loose settings (RepairDesk surfaces them as shipment context, which we mirror); (b) keep operator handoff-mode choice (direct / drop-off / pickup) **explicit and visible**, not implicit (RepairDesk shows the chosen option per shipment, which our `Direct Handoff / Selected / Not Requested / Confirmed` badges replicate); (c) preserve **full audit trail** — both selection and request actions write to the shipment timeline, matching RepairDesk's per-shipment activity log. **Not adopted from RepairDesk**: a separate "Pickup Manager" dashboard listing all pending pickups across shipments (deferred — we keep pickup actions on the shipment record itself for now to avoid creating a parallel mini-app); per-package pickup splitting (deferred — our packages array already supports this and Phase 3 batch labels will surface it natively); and ShipStation's "Stores" abstraction (we already have a multi-tenant store hierarchy, so a separate stores concept inside shipping would conflict).

    -   **Strict Validated-Only Address Readiness (Get Rates / Purchase Label / Rate Visibility)**: The address-readiness predicates `isAddressAccepted(shipment)` and `isOriginAddressAccepted(shipment)` return `true` **only** when the corresponding `addressValidation.status === 'validated'`. A `corrected` result is treated as an **intermediate state only** and does **not** unlock Get Rates, Purchase Label, the selected-rate row, or the Available Rates panel. When the provider returns a `corrected` result, `validateSingleSide` swaps the suggested/normalized address into the shipment fields (`originAddress` or `destinationAddress`) so the operator can re-validate against the cleaned address, but it does **not** set `accepted = true` on the validation result. The operator must explicitly re-run **Validate Addresses** on the corrected address and receive a `validated` status before any rate-fetch / label-purchase / rate-display surface unlocks. The combined-validation toast surfaces this explicitly when a side comes back corrected: *"Origin: corrected — re-validate the suggested address to enable rates"*. The full operator workflow is therefore: address entered → (optional) corrected/suggested → re-validate → validated → Get Rates enabled. This rule applies symmetrically to both origin and destination — if one side is `validated` and the other is only `corrected`, Get Rates remains disabled, the rate display stays hidden behind the lock notice, and Purchase Label stays blocked.
    -   **Rate Visibility Gating by Address Validation (Provider Mode)**: In provider mode, **shipping rate visibility is gated on both origin and destination address validation**. Until both `isOriginAddressAccepted(shipment)` and `isAddressAccepted(shipment)` return true, neither the **selected rate row** ("Service · carrier · service name · price · ETA" line in the Provider & Operations panel) nor the **Available Rates panel** (the list of fetched rates with Select buttons) is shown. Manual mode is **unaffected** — manually keyed carrier/service still display via the "Manual Service" row regardless of validation state, since manual mode is not provider-driven. If a rate was previously selected and an address is later edited (which clears that side's validation), the selected-rate row hides itself behind the lock notice until validation is re-run, preventing operators from acting on a stale rate against an unverified address.

Auto-sync is monotonic (only advances forward via the canonical `statusOrder`). When the shipment is still Draft, the return detail shows an amber notice directing the operator to the Shipping Center for label purchase. **Manual button suppression**: when a linked return shipment exists, the manual `Ready`, `Label Created`, `Packed`, `Dispatched`, `In Transit`, and `Delivered` transition buttons are all hidden in the return detail modal — the entire shipping leg is exclusively driven by shipment auto-sync to prevent divergence. Two delivery options: (1) "Copy Instructions" — copies formatted customer instructions (return authorization, carrier, tracking, label link, step-by-step instructions, items list) to clipboard; (2) "Open Email Draft" — opens a draft in the operator's default mail client with pre-populated subject and body addressed to the customer's email. **The email button does NOT send automatically** — it opens a draft for the operator to review and send. An amber info banner inside the modal makes this explicit ("the email button opens a draft in your default mail app — it does not send automatically. Direct send-from-server email is on the future roadmap.") and the button itself uses the `drafts` icon and "Open Email Draft" label to reinforce the affordance. Accessible via "Send Label to Customer" button in the return detail's shipment section.
    -   **Linked Shipment Cross-Navigation**: Both the `Original Shipment` (outbound) and the `Return Shipment` (reverse) numbers in the return detail modal are clickable buttons that navigate to `/shipping` with `state.openShipmentId = <shipment id>`. ShippingCenter's mount-time effect detects this state, selects the matching shipment, opens its detail panel on the overview tab, and clears the navigation state. This eliminates manual ID lookup when an operator wants to inspect tracking events, validate addresses, or purchase a label for a linked shipment.
    -   **Return Item Quantity & Price Visibility**: `ReturnPrefill.items` carries `unitPrice` and `maxQuantity` from the source record (Invoices pass `i.price` and `i.quantity` from line items). `ReturnItem` exposes the same fields. The `CreateReturnModal` items list renders each line as: item name, unit price (e.g. `@ $89.99`), original purchased quantity ("of 3 originally purchased"), an editable numeric quantity input bounded by `maxQuantity`, an optional condition badge, and a per-line refund context total (`$269.97`). Prevents over-return (qty cannot exceed `maxQuantity`) and gives the operator immediate refund-context arithmetic without leaving the modal. The detail modal also displays per-line unit price × qty totals for finalized returns.
    -   **Return Shipment Create Flow Cleanup**: The `CreateReturnShipmentModal` no longer collects carrier, service level, or tracking number at creation time — those fields are owned by the Shipping Center provider workflow (rate fetch → label purchase). The modal collects only origin (customer, editable), destination (store, editable line/city/state/zip/phone), packages, and notes. An info banner inside the modal explains: "Carrier, service level, and tracking number are set later in the Shipping Center when you fetch rates and purchase the label." Eliminates the previous double-entry confusion and prevents stale manual values from masking provider truth.
    -   **Permissions**: `returns` permission domain with sub-permissions: create_return, approve_return, receive_return, inspect_return, complete_return_disposition, cancel_return, create_return_shipment. Plan-gated to Growth and Advanced tiers.
    -   **Preview Mode**: All write actions gated by `isWriteBlocked`. Preview mode shows disabled state with "Preview Mode — Actions Disabled" indicator.
    -   **RepairDesk Article Insights Adapted**: Shipping from tickets/invoices entry points, label printing status triggers, inbound/outbound label distinction, two-way tracking sync readiness. Flat rate not adopted (not relevant to returns). ShipStation UX not copied (our architecture is provider-agnostic).
    -   **Phase 2 Forward-Compatibility**: Return model includes placeholder fields for service points (servicePointId), customs docs (customsInfo object), pickup requests (pickupRequestId), insurance claims (insuranceClaimId). These are not implemented but architecturally ready.
    -   **Phase 3 Forward-Compatibility**: Return lifecycle and statusHistory are automation-friendly (structured status transitions). Status timestamps stored for future SLA optimization and carrier scorecards. Return events compatible with future automation rules and batch processing.
    -   **Phase 2.5.1 — EasyPost Pickup Silent-Failure Fix**: Root cause: the only feedback for a pickup attempt was the page-level `providerError`/`providerSuccess` banner ~900 lines above the pickup form, so when EasyPost returned an outcome the operator could not see (most commonly test-mode pickups returning a created pickup with `pickup_rates: []`, which fell through to a "requested" persisted record with no inline cue), it looked like the button silently reverted. Fixes: (1) Added an inline `pickupAttemptResult` panel rendered directly above the Request button with structured per-step status (Create → Buy → Persist), kind (success/partial/error/info), code, providerPickupId, confirmation, cost, and dismiss button — every code path now writes to it before returning. (2) Explicit handling for the no-rates case: when `caps.pickupNeedsRatePurchase=true` and the provider returns zero rates, the created pickup is still persisted (so the operator can cancel it) and a clear "Pickup created — no rates returned" partial result is shown citing the test-mode limitation. (3) `handleRequestPickup` wrapped in `try/finally` so `setPickupSubmitting(false)` always runs and exceptions surface as a proper `NETWORK_ERROR` result with stack info. (4) `clearProviderFeedback()` now runs at the start of each attempt so stale messages do not mask the new outcome. (5) `createProviderPickup`/`buyProviderPickup`/`cancelProviderPickup` client wrappers now catch network-level throws and return them as structured `{success:false, error:{code:'NETWORK_ERROR'}}` so the caller never has to handle two failure shapes. (6) `providerId` is now passed explicitly from client to server on all three pickup endpoints (defensive — the server already falls back to active provider, but explicit IDs avoid races when the operator switches providers mid-flight). (7) Server `/api/shipping/pickup/{create,buy,cancel}` endpoints wrap the adapter calls in try/catch and log every request/response with provider id, pickup id, rate counts, and error codes, so server logs are now a usable trace for QA. Browser-side `console.log('[Pickup] …')` mirrors each step. Net effect: the button can no longer silently reset — every attempt ends in a visible structured outcome.
    -   **Phase 2.5 — Configurable Carrier Locator Settings + Provider-Specific Pickup Models**: Carrier Locator settings card became a real per-adapter form (USPS / UPS / FedEx): each row exposes Enable, Sandbox/Production select, Credential reference label, Verify button, and Docs link. Configuration is persisted to `sessionStorage` under `locator_adapters_config_v1`. Verify is honest — until the underlying adapter API call lands it records `verification_failed` with an explanatory note (no fake "verified" status). Real secrets stay server-side; the credential reference field is a per-store label only. On the pickup side, `PROVIDER_CAPABILITIES` was expanded with `supportsLivePickupBooking`, `pickupNeedsRateSelection`, `pickupNeedsRatePurchase`, `pickupNeedsProviderShipmentId`, `pickupNeedsCarrierAccount`, `pickupTestModeLimitations`. EasyPost adapter (`server/adapters/easypost.ts`) now implements real `/v2/pickups`, `/v2/pickups/{id}/buy`, `/v2/pickups/{id}/cancel`. Shippo and ShipStation adapters return honest `NOT_IMPLEMENTED` from the new optional `createPickup/buyPickup/cancelPickup` interface methods (declared in `server/types.ts`). New endpoints `/api/shipping/pickup/{create,buy,cancel}` proxy to the active adapter; `src/shipping/shippingApiClient.ts` exposes `createProviderPickup/buyProviderPickup/cancelProviderPickup`. `handleRequestPickup` rewritten: when active provider supports live booking and a `providerShipmentId` exists, it creates the pickup, auto-buys the cheapest pickup rate, and stores `status='confirmed'` with the real `providerPickupId`, cost, and confirmation number. When live booking is not supported, it records a local-only intent (`source='local_only'`, no fake confirmation number). `handleCancelPickup` calls the live cancel API when `source='live_provider'`. The `PickupRequest` type gained `providerPickupId`, `providerPickupCost/Currency`, `source`. The pickup card displays per-provider capability badges (Live booking wired vs Local-only, Rate purchase, Needs label first, Carrier account), the carrier coverage list, the audit note, and any test-mode limitations; existing records show a green "Booked live with {provider}" banner with provider id/cost when `source='live_provider'`, or an amber "Local-only request" banner otherwise.
    -   **Phase 2.4 — Honest Service-Point Search via Carrier-Specific Locator Adapters**: Earlier passes relied on a synthetic ZIP "preview" search that rewrote postal codes on a hardcoded mock list, which looked real to operators. Audit conclusion: provider aggregators (EasyPost / Shippo / ShipStation) do **not** expose a unified service-point locator API; live lookup requires direct carrier credentials per carrier. Replaced with an honest two-track flow. (1) `PROVIDER_CAPABILITIES` map expanded from `{ servicePoints, pickupRequests }` to a richer `ProviderCaps` shape including `supportsPickupCancellation`, `pickupRequiresProviderConfirmation`, `pickupMayRequireRateSelectionOrPurchase`, `pickupCarrierCoverage`, `pickupStatusModel`, and a human-readable `pickupAuditNote`. `servicePoints` is set to `false` on every provider until carrier-specific adapters are wired. The pickup card surfaces the `pickupAuditNote` as a "Provider note" banner (e.g. EasyPost: "Pickup → Buy required, carrier may charge a pickup fee, live API call not yet wired"). (2) Added a parallel `getServicePointManualEntryEligibility` evaluator alongside the existing `getServicePointEligibility`. Both share plan / permission / lifecycle / mutex checks; only the live-locator path additionally requires `caps.servicePoints`. The Logistics Service Point panel uses `spEditable = liveElig.eligible || manualElig.eligible`, so operators can always record a carrier-issued service-point reference manually even while no locator adapter is configured. (3) Service Point modal rewritten — the fake ZIP search is gone, replaced by a permanently-disabled "Live ZIP search" fieldset (clearly badged "Unavailable") and a manual-entry form that captures id, name, type, full address, phone, and selection notes; submit goes through `handleSelectServicePointManual` → shared `handleSelectServicePoint` which now accepts either eligibility track. The selected `ServicePoint` carries a new `source: 'live_locator' | 'manual' | 'preview'` field for audit, and the panel renders a "Manual" / "Live" pill next to the name. (4) Settings tab now embeds a "Carrier Locators" card under Shipping Providers listing USPS Locations, UPS Locator, and FedEx Locations adapters with required env keys and a "Connect (coming soon)" button. (5) Adapter scaffolding lives at `src/shipping/locators/` — `types.ts` (LocatorAdapter contract: `carrierId`, `carrierLabel`, `isConfigured()`, `findServicePoints({ postalCode, countryCode, serviceCode, weightLb, dimensionsIn, radiusKm, limit })`), `uspsLocator.ts` / `upsLocator.ts` / `fedexLocator.ts` (all return `{ unavailable: true, configHint }` until credentials land), and `index.ts` dispatcher exposing `findServicePoints({ carrier, ... })`, `isAnyLocatorConfigured()`, `getConfiguredCarriers()`. The dispatcher is re-exported from `src/shipping/shippingApiClient.ts` so the rest of the app calls a stable surface. Result: zero fabricated locator data reaches the operator, the lock-notice copy explains exactly why live lookup is unavailable and what to do about it, manual entry remains a fully audited path, and the locator wiring is one credentials task away from going live carrier-by-carrier.

## System Design Choices

-   **Routing**: React Router v7.
-   **State Management**: Primarily React Context API and local component state.
-   **Firebase Integration**: Firestore for backend data persistence and Firebase Authentication for user management.
-   **AI Integration**: Gemini API for AI functionalities.
-   **Server-Side Shipping API**: Express server on port 5001 handles all shipping provider operations, with credentials stored securely server-side.
-   **Configuration**: `firebase-applet-config.json` for Firebase project configuration and `firebase-blueprint.json` for Firestore schema definition.
