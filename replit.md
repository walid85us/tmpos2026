# Overview

This project is a multi-tenant SaaS frontend platform for small to medium-sized businesses. It provides a unified suite of tools including Point-of-Sale (POS), employee and customer relationship management (CRM), inventory control, reporting, and marketing. The platform aims to streamline operations, offer actionable insights, and automate business processes through a scalable, customizable, and feature-rich architecture.

# System Architecture

## Frontend

The frontend is built using React 19, TypeScript, Vite 6, and Tailwind CSS v4.

## UI/UX Decisions

-   **Design System**: Features rounded cards, a glass effect (`bg-white/80 backdrop-blur-xl`), a consistent primary color theme, and a `ghost-border` utility.
-   **Typography**: Uses `font-black uppercase tracking-widest` for labels.
-   **Animations**: Employs Framer Motion for smooth UI transitions and consistent modal patterns.
-   **Notifications**: Utilizes toast notifications for user feedback.
-   **Accessibility**: Prioritizes semantic accessibility for forms, tables, and modals.

## Technical Implementations & Feature Specifications

-   **Authentication & Access Control**: Implemented with Firebase Authentication and a 7-level hierarchical permission model.
-   **Tenant Management**: Provides full lifecycle management, including provisioning, subscriptions, feature overrides, billing, and audit logging.
-   **Business Management Modules**:
    -   **Point of Sale (POS)**: Comprehensive cart management, diverse payment options, tax/discount calculations, customer management, repair intake, held orders, quick add stock, operator switching with RBAC, refund workflows, and warranty claims.
    -   **Employee Management**: Manages employees, roles, time tracking, and payroll with granular permissions.
    -   **CRM (Customers)**: Enhanced CRM with multi-field search, loyalty programs, duplicate detection, and integrated order/invoice history.
    -   **Invoices**: Supports full invoice lifecycle with dynamic line items, searchable catalog, payment processing, and configurable document templates.
    -   **Services**: Manages a service catalog with CRUD operations, warranty configuration, and bulk price editing.
    -   **Repairs Module**: Manages the full repair ticket lifecycle, including structured workflow, parts tracking, technician assignment, activity timelines, and financial summaries.
    -   **Inventory Management**: Comprehensive system for managing various `StockItem` types (serialized, non-serialized, handset). Features include UPC/manufacturer/location tracking, min/max stock levels, serial number tracking, advanced filtering, stock adjustments, global movements log, inventory transfer lifecycle, stock counts, and trade-in management, all integrated with RBAC.
    -   **Supply Chain Management**: Manages suppliers, purchase orders, Goods Received Notes (GRN), and Return Merchandise Authorization (RMA) with detailed tracking and financial recording, gated by RBAC.
    -   **Shipping Center**: A centralized fulfillment module for managing shipments across all source documents, supporting various shipment types. Includes status transition workflows, tracking event timelines, carrier/service level configuration, cost tracking, a provider adapter layer for external shipping carriers, service points, pickup requests, and address validation. Features a unified, idempotent webhook/event processing pipeline, and tools for bulk tracking synchronization.
    -   **Returns Portal / Reverse Logistics**: A first-class module for managing returns with a structured lifecycle (Draft → Requested → Approved → ... → Completed), return reasons, requested/final resolutions, item disposition, intake/inspection workflow, and integration with return shipments.
-   **Reporting**: Features a dashboard with data visualizations.
-   **Onboarding**: A multi-step process for new tenants, covering plan selection, pre-configured settings, and an activation checklist.
-   **Document Template Editor**: A structured editor for 5 document types, offering deterministic HTML generation, branding, and visual preview.
-   **Carrier Analytics Foundation (Phase 2 + correction pass)**: Provides deterministic, truthful shipping metrics inside Shipping Center → Analytics tab. Implementation in `src/components/shipping/CarrierAnalytics.tsx`.
    -   **Mode derivation**: The Mode filter uses a `deriveMode(shipment)` helper that mirrors the operational `getShipmentMode()` in ShippingCenter (provider when `selectedRate` exists; manual when `carrier && serviceLevel` are populated without a live rate; provider otherwise). The stale UI-only `shipmentMode` field is intentionally NOT consulted, so analytics cannot drift from operational behavior.
    -   **Status-to-bucket mapping**: Dispatched and In Transit are SEPARATE buckets. `dispatchedCount = shipments where status === 'Dispatched'`. `inTransitCount = shipments where status === 'In Transit'`. We never collapse Dispatched into In Transit. The volume card explicitly renders both with a "Dispatched ≠ In Transit" affordance, plus Delivered, Exceptions, and Cancelled cards.
    -   **Pickup Analytics gating**: The pickup-analytics block is gated by BOTH the `pickup_requests` plan feature AND the `view_pickup_analytics` sub-permission (separate from operational `request_carrier_pickup` so visibility can be granted without the ability to schedule or cancel pickups). When the plan excludes pickups, a coherent "not included" state renders. When the role lacks the sub-permission, a coherent "not enabled for your role" state renders. Local-only intents are NEVER inflated into provider-confirmed counts (provider-confirmed strictly requires `status in {confirmed, completed} AND source === 'live_provider'`).
    -   **Provider-connected analytics**: When an active aggregator provider is configured (`activeProviderId !== null`), an additional read-only block surfaces provider-backed sub-counts (provider shipments / labels / delivered / dispatched / in-transit / provider-confirmed pickups) derived from the SAME already-recorded fields. We do NOT poll a live carrier dashboard, do NOT fabricate SLA grades, and do NOT imply parity across unconfigured providers. An explicit coverage chip and amber caveat call out partial coverage truthfully. The block hides entirely when no provider is configured.
    -   **Cost metrics, lifecycle timing, distributions, coverage chips**: As before — cost gated by `view_shipping_costs`, lifecycle timing requires ≥3 paired timestamps (otherwise `insufficient-data: N/3` badge — never extrapolated), distributions surface `Unknown` rows in amber, coverage chips colored green ≥90% / amber ≥50% / rose <50%.
    -   **Plan + permission gating (overall surface)**: The Analytics tab requires `carrier_analytics` plan feature (Growth + Advanced) AND `view_carrier_analytics` sub-permission. The component re-validates both internally so a stale activeTab cannot bypass either gate. `view_carrier_analytics` has `defaultLevel: 'manage'` so the schema-level fallback cannot silently grant analytics to shipping viewers; technician + sales_associate role templates also explicitly seed `false`.
-   **Carrier Locators / Shipping Provider plan & permission treatment (Phase 2 correction)**:
    -   **Plan features**: `service_points` (Carrier Locators) and the new `shipping_providers` (Shipping Provider Configuration) are independently controlled in System Owner Plans & Features (`src/owner/mockData.ts` featureMatrix). Both ship enabled for Growth + Advanced and disabled for Essential.
    -   **Settings tab gating in Shipping Center**: When `shipping_providers` is excluded from the plan, the ShippingProvidersPage block is replaced by a "not included in your current plan" state. When `service_points` is excluded, the Carrier Locators block is replaced with a similar state. Aggregator provider configuration (EasyPost / Shippo / ShipStation) and per-carrier locator adapters (USPS / FedEx / UPS / DHL / GLS) are conceptually separate and gate independently.
    -   **Sub-permissions**: `manage_shipping_settings` controls aggregator provider configuration. The new `manage_carrier_locator_settings` (`defaultLevel: 'manage'`, granted to manager / store_owner only) controls per-store locator adapter configuration. The Carrier Locators block hides with a coherent "not enabled for your role" state when the operator lacks this sub-permission. The new `view_pickup_analytics` sub-permission controls pickup-analytics visibility independently from operational pickup actions.
-   **Customs Docs and Insurance Rules — queued, not deferred**: These are NOT implemented in the Phase 2 correction pass, but per the correction-pass guidance they are no longer treated as indefinitely deferred. They are the next queued shipping improvements after this correction pass closes. When implemented they MUST be added coherently to BOTH System Owner Plans & Features (as plan-gated features) and Store Owner Permissions Matrix (as sub-permissions), matching the gating discipline used here for Carrier Analytics, Pickup Requests, Service Points / Carrier Locators, and Shipping Provider Configuration.
-   **Out of scope for Phase 2 correction pass**: The broader Addon governance / archived-addon / tenant Features tab paid-override redesign is intentionally NOT in this correction pass. It belongs to a separate commercial-platform workstream.
-   **Phase 2 — Plan-to-Permission Propagation + Provider Hard-Disable (general rule)**: A generalized rule has been formalized: plan decides whether a feature exists; role decides who uses it; role can NEVER resurrect a plan-disabled feature. This is now enforced at three layers — (1) `FEATURE_PERMISSION_DEPENDENCIES` map in `accessConfig.ts` links feature ids (`shipping_providers`, `pickup_requests`, `service_points`, `carrier_analytics`) to dependent sub-permissions, with helpers `isSubPermissionPlanAvailable`, `isPlanFeatureLiveFor`, `getFeatureGatesForSubPermission`; (2) `AccessContext.checkSubPermission` runs the plan-availability check BEFORE the system_owner/store_owner blanket allow so no role can exercise a plan-disabled capability; (3) `Employees.tsx` Permissions Matrix renders Plan-Locked rows with a truthful subtext (which feature/module is missing) and overrides the Store Owner column to Plan-Locked instead of Allowed. `planFeatures` for Growth/Advanced now includes `carrier_analytics` so the static-fallback path is consistent with the override path.
-   **Phase 2 Final Entitlement Integrity correction — linked feature permissions DISAPPEAR (general rule)**: Refined the prior plan-to-permission propagation rule. Previously plan-disabled sub-permissions rendered as "Plan-Locked" in the Store Permissions Matrix; the final rule is that they are OMITTED entirely. `Employees.tsx` now does `if (!isSubPermissionPlanAvailable(sub, tenant.plan)) return null;` — no Plan-Locked chrome, no disabled row. The matrix only displays capabilities that actually exist for the tenant's plan. The `FEATURE_PERMISSION_DEPENDENCIES` map remains the single architectural source of truth, so future feature/plan additions inherit disappearance semantics with no UI changes needed. The runtime evaluator `checkSubPermission` still short-circuits to denied for plan-locked sub-permissions, so even a stale role grant cannot resurrect a plan-disabled capability.
-   **Explicit Shipping Provider configuration permission**: Added new sub-permission `configure_shipping_provider` (defaultLevel: 'manage', store_owner / manager only). It is the dedicated permission for connecting / configuring / switching environments for / removing aggregator shipping providers (EasyPost / Shippo / ShipStation). It is plan-gated by `shipping_providers` via `FEATURE_PERMISSION_DEPENDENCIES` and is independent of (a) general `manage_shipping_settings` (origin defaults / package presets / manual carrier list / label preferences) and (b) per-action provider permissions (`fetch_shipping_rates` / `purchase_shipping_label` / `validate_shipping_address` / `sync_shipping_tracking`). `ShippingCenter.canManageProviderSettings` now reads `configure_shipping_provider`. `manage_shipping_settings` is no longer a member of the `shipping_providers` feature dependency list and remains visible/assignable independent of provider plan gating.
-   **Shipping Provider FULL deactivation on plan disable**: When `shipping_providers` is no longer live in the tenant's plan, a single plan-aware effect in `ShippingCenter.tsx` performs hard deactivation: (1) calls `shippingApi.setActiveProvider(null)` to clear the server's active-provider pointer, (2) enumerates all configured providers via `getProvidersStatus` and calls `removeProviderCredentials(id)` for each, and (3) clears every local operational state slot (active provider id, environment, provider statuses, settings panel visibility, available rates, rates panel visibility). The effect uses a `cancelled` token so any in-flight load cannot land after the plan flips and resurrect stale state. The on-mount load is also folded into the same effect so it only runs when the plan ALLOWS providers — eliminating the resurrection race entirely. If the server purge partially fails the operator sees a truthful `providerWarning` ("provider deactivated locally; some server-side credentials may not have been cleared and will be removed on the next successful sync"). When the plan re-enables the feature later, the load branch re-runs but the server has no active provider and no credentials, so the operator must explicitly reconnect — nothing silently resurrects.
-   **Shipping Provider hard-disable in `ShippingCenter.tsx`**: Renamed underlying state to `activeProviderIdRaw` and introduced a single derived `activeProviderId = planAllowsShippingProviders ? activeProviderIdRaw : null` declared immediately after the state. All existing `if (!activeProviderId)` early-returns therefore auto-inherit plan gating with no per-handler bypass. A defense-in-depth `guardShippingProviderPlan` is invoked at the top of every provider-call handler (handleFetchRates, handlePurchaseLabel, handleValidateAddress(es), handleSyncTracking, handleBulkSync, handleRequestPickup, handleConfirmPickupBuy, handleTestConnection, handleReplayEvent, handleSimulateTrackingEvent) so the operator sees a truthful "not included in your current plan" message instead of the generic "configure a provider" message. Credential store state is preserved across plan changes — only the derived id nulls out at read time. A truthful four-state `providerAvailability` tagged union (`plan_disabled | permission_denied | no_provider | configured`) drives `getProviderPrerequisiteMessage`, surfacing a distinct message per state. `CarrierAnalytics` automatically inherits this gating because it consumes the plan-aware `activeProviderId` prop.

## System Design Choices

-   **Routing**: Utilizes React Router v7.
-   **State Management**: Primarily relies on React Context API and local component state.
-   **Firebase Integration**: Leverages Firestore for backend data persistence and Firebase Authentication for user management.
-   **AI Integration**: Integrates with the Gemini API for AI functionalities.
-   **Server-Side Shipping API**: An Express server on port 5001 handles all shipping provider operations with secure credential storage.
-   **Configuration**: `firebase-applet-config.json` for Firebase project configuration and `firebase-blueprint.json` for Firestore schema definition.
