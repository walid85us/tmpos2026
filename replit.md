# Overview

This project is a multi-tenant SaaS platform frontend designed for small to medium-sized businesses, offering a comprehensive solution for business management. It integrates Point-of-Sale (POS), employee and customer relationship management (CRM), inventory control, reporting, and marketing functionalities. The platform emphasizes scalability, customization, and feature richness to streamline operations and provide valuable insights and automation.

# System Architecture

## Frontend

The frontend is built using React 19, TypeScript, Vite 6, and Tailwind CSS v4 to deliver a modern, responsive, and interactive user experience.

## UI/UX Decisions

-   **Design System**: Features a consistent design language with `rounded-[2.5rem]` or `rounded-[3rem]` cards, `bg-white/80 backdrop-blur-xl` glass effect, and a primary color theme with a `ghost-border` utility.
-   **Typography**: Employs `font-black uppercase tracking-widest` for labels.
-   **Animations**: Utilizes `motion/react` (Framer Motion) for smooth UI transitions and consistent modal patterns with `AnimatePresence`.
-   **Notifications**: Implements toast notifications for user feedback.
-   **Accessibility**: Focuses on semantic accessibility for form inputs, keyboard-navigable table rows, and modal dialogs.

## Technical Implementations & Feature Specifications

-   **Authentication & Access Control**: Leverages Firebase Authentication for platform and tenant-specific roles, supporting a 7-level hierarchical permission model with granular sub-permissions.
-   **Tenant Management**: Manages the full tenant lifecycle, including provisioning, subscriptions, feature overrides, billing, and audit logging.
-   **Point of Sale (POS)**: Comprehensive POS system covering cart management, diverse payment options, tax and discount calculations, customer management, repair intake, held orders, quick add stock, operator switching with RBAC, refund workflows, and warranty claims with supervisor PIN authorization.
-   **Employee Management**: Features employee, role, time tracking, and payroll management with granular permissions.
-   **CRM (Customers)**: Offers an enriched CRM with multi-field search, loyalty management (tiering, points), duplicate detection, and integrated order/invoice history.
-   **Invoices**: Full invoice lifecycle management with dynamic line items, searchable inventory/service catalog, payment processing (Cash, Card Terminal), supervisor PIN-authorized reopen feature, and print functionality using configurable document templates with branding.
-   **Services**: Manages a service catalog with CRUD operations, warranty configuration, bulk price editing, and dependency-aware delete confirmations.
-   **Repairs Module**: Provides full repair ticket lifecycle management with a structured workflow, device intake, parts tracking, technician assignment, activity timeline, financial summary, and integration with POS for payment collection.
-   **Reporting**: Features a dashboard with data visualizations powered by Recharts.
-   **Onboarding**: Multi-step onboarding process for new tenants, including plan selection, pre-configured settings, and an activation checklist.
-   **Document Template Editor**: A structured template builder in Settings for 5 document types, allowing deterministic HTML generation from `enabledTags`, branding management, and visual preview.
-   **Data Flow**: Utilizes `StoreLocalState.tsx` for managing shared local mock state across various entities like customers, stock, orders, invoices, services, loyalty configurations, and supply chain components.
-   **Feature Gating & Navigation Derivation**: Employs a merge-based strategy for plan feature resolution, combining static `planFeatures` with dynamic `features_data` from `sessionStorage`, supporting both `featureMatrix` and legacy `planEntry` formats.
-   **Dashboard Quick Actions**: Includes Quick Intake (repair), Print Label, Scan QR (inventory search), Add Stock, and Held Orders. Features dynamic low-stock alerts with full 3-tab inventory detail experience (Info/Movements/Edit) reused from inventory module, including adjust stock modal with increase/decrease/reason/notes (permission-gated).
-   **Inventory Management**: Comprehensive system for managing `StockItem` types (serialized, non-serialized, handset) with flags like `isRepairPart` and `isHiddenOnPOS`, UPC/manufacturer/location fields, min/max stock levels, cost price, and serial number tracking. Features advanced filtering, stock adjustment with reason tracking, global movements log, inventory transfer lifecycle (full edit mode for Draft/Sent/In Transit with caution banner, read-only after Received), stock count with re-adjust mode (Completed counts allow re-adjustment before Close, Closed counts are fully read-only), trade-in management with View Details modal, Edit modal with photo preserve/replace, camera-first capture via `navigator.mediaDevices.getUserMedia` with fallback alerts, ID photo capture and refurbishment flows, and detailed stock movement tracking for all inventory changes. Integrates repair parts and POS sales movements with RBAC gating for various actions.
-   **Supply Chain Management**: Manages suppliers, purchase orders (PO) with multi-item creation and lifecycle (Draft → Ordered → Partially Received → Received), Goods Received Notes (GRN), and Return Merchandise Authorization (RMA) with detailed tracking, resolution, and financial recording (SupplierRefundEntry). All modules are RBAC-gated.
-   **Shipping Center**: Centralized shipping module (`ShippingCenter.tsx`) for managing shipments across all source documents. Features: shipment list with status/source filters, 3-tab detail view (Overview/Tracking/Packages), create/edit modals with full package add/remove/edit UI, status transition workflow (Draft → Ready → Label Created → Packed → Dispatched → In Transit → Delivered), tracking event timeline, carrier/service level configuration, and cost tracking. Types: `customer_delivery`, `repair_return`, `store_transfer`, `rma_outbound`, `rma_return`. Sources: `invoice`, `repair`, `transfer`, `rma`. **Source Integration**: Source modules (Invoices, RepairTickets, SupplyChain/RMA, Inventory/Transfers) use `navigate('/shipping', { state: { openCreate: true, prefill: ShipmentPrefill } })` to redirect to ShippingCenter with pre-populated data including source items, addresses, and type — no inline shipment creation. ShippingCenter reads `location.state` on mount, auto-opens the create modal with prefilled fields/packages, then clears the state. Source prefill populates: sourceType, sourceId, sourceNumber, shipment type, originAddress, destinationAddress (including customer/vendor name, phone, email where available), and sourceItems (with auto-generated initial package). **Manual Source Reference Resolution**: The create modal's Source Reference Number field includes a "Resolve" button. Entering a reference number (e.g. INV-2026-001, T-1001, RMA-2026-001, TRF-2026-001) and clicking Resolve or pressing Enter triggers a lookup against the selected source type's collection. On success, form fields auto-populate from the resolved record (type, addresses, items, packages). On failure (not found or ineligible status), a red error shows the reason. Eligible statuses: Invoice=Paid/Partially Paid, Repair=Ready for Pickup/Completed, Transfer=Draft/Sent/In Transit, RMA=Pending/Shipped. Changing source type clears resolver feedback. Source-routed prefill (via navigation state) continues to work independently. **Edit Lifecycle Lock**: Shipments are fully editable (carrier, packages, addresses, notes, costs) only in Draft and Ready statuses. Once a shipment reaches Label Created, the main edit action is locked — the edit button is replaced with an "Edit Locked" indicator with a lock icon. The `openEditModal` function also enforces this at the logic level. Status transitions, tracking updates, and detail viewing remain available at all lifecycle stages. Later statuses (Packed, Dispatched, In Transit, Delivered, Exception, Cancelled) remain edit-locked. **Package Management**: Create modal shows source items summary, auto-generates initial package with contents/declared value from source items, supports add/remove package rows. Edit modal includes package editing, locked (disabled) after status reaches Label Created. **Runtime Mode Architecture**: AccessContext separates two independent flags — `isDevSession` (controls session simulation/switching) and `isPreviewModeEnabled` (controls read-only safety). DevSessionSwitcher calls `activateDevSession()` for role switching (writable); only `read_only`/`suspended` tenant scenarios additionally call `enablePreviewMode()`. ShippingCenter uses `isWriteBlocked` (derived from `isPreviewModeEnabled`) for write guards. In normal dev sessions (active/trialing stores), `isWriteBlocked=false` and all writes work. In read-only/suspended scenarios, `isWriteBlocked=true` — write handlers return early, amber "Preview Mode" banners appear, and submit buttons show "Preview Only" disabled state. RBAC (`checkPermission`/`checkSubPermission`) further gates actions by role permissions. 8 sub-permissions (`create_shipment`, `edit_shipment_pre_dispatch`, `dispatch_shipment`, `update_tracking_events`, `cancel_shipment`, `view_shipping_costs`, `manage_shipping_settings`). Dual-gated: plan entitlement (`canAccess('shipping')`) + role permission (`checkSubPermission`). Available on Growth and Advanced plans. **Edit Lifecycle Lock**: `isEditable()` returns true only for Draft/Ready statuses. Label Created and beyond shows a lock indicator; the edit button is hidden. `openEditModal` enforces the lock at logic level. Cancelled shipments show neither edit nor lock. **Provider Adapter Layer (Phase 1)**: Architecture at `src/shipping/` — `types.ts` defines `ShippingProviderAdapter` interface (validateAddress, getRates, purchaseLabel, getTracking), request/response contracts, and `ProviderError` type. `adapters/easypost.ts` implements the full EasyPost adapter with credential safety (checks `sessionStorage` for API key, returns `PROVIDER_NOT_CONFIGURED` error when missing). `providerRegistry.ts` provides factory pattern (`getProvider(name)`) and `availableProviders` list. **Extended Shipment Data Model**: `Shipment` interface includes: `addressValidation` (AddressValidationResult), `selectedRate` (ShippingRate), `label` (LabelArtifact with url/format/trackingNumber/cost/carrier/service), `providerShipmentId`, `providerTrackingId`, `providerTrackingEvents` (ProviderTrackingEvent[]), `lastTrackingSyncAt`, `ratesRetrievedAt`, `availableRates` (ShippingRate[]). **Provider UI in Detail View**: Provider & Operations section in shipment overview tab shows: address validation status badge, selected rate details, label purchase status with tracking number, provider reference ID, last tracking sync time, and action buttons — all permission-gated via 5 new sub-permissions (validate_shipping_address, fetch_shipping_rates, purchase_shipping_label, print_shipping_label, sync_shipping_tracking). Actions: Validate Address (Draft/Ready only), Get Rates (Draft/Ready), Purchase Label (Ready, requires rate selection), Open Label (opens label URL), Sync Tracking (post-dispatch). Rate shopping panel displays sorted rates with Select buttons. All actions respect `isWriteBlocked` and lifecycle locks. **Provider Config in Context**: `StoreLocalState` exposes `shippingProviderConfig` (ShippingProviderConfig | null) and `setShippingProviderConfig` for tenant-level provider settings.
-   **POS Orders Lookup**: Searchable order history with detailed invoice view for actions.
-   **POS Cart Merge**: Automatically merges identical inventory items in the cart.
-   **POS Finalize**: Creates `CompletedOrder` records and decrements stock.
-   **Stock Count Integrity**: Cart displays remaining stock with visual indicators.
-   **Store Permissions Matrix**: Interactive matrix in `EmployeesPermissionsPage` for role-domain permission configuration.
-   **Supervisor Refund Authorization**: Allows supervisor approval for refunds via PIN.
-   **UI Feedback**: Provides consistent visual feedback for successful saves.

## System Design Choices

-   **Routing**: React Router v7.
-   **State Management**: Primarily uses React Context API for global state and extensive local component state.
-   **Firebase Integration**: Leverages Firebase Firestore for backend data persistence and Firebase Authentication for user management.
-   **AI Integration**: Utilizes the Gemini API for incorporating AI functionalities.
-   **Configuration**: `firebase-applet-config.json` for Firebase project configuration and `firebase-blueprint.json` for Firestore schema definition.
