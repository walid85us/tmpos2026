# Overview

This project is a multi-tenant SaaS frontend platform for small to medium-sized businesses. It provides a unified solution for business management, including Point-of-Sale (POS), employee and customer relationship management (CRM), inventory control, reporting, and marketing functionalities. The platform's core purpose is to streamline operations, offer actionable insights, and automate tasks through its scalable, customizable, and feature-rich design.

# System Architecture

## Frontend

The frontend is built using React 19, TypeScript, Vite 6, and Tailwind CSS v4.

## UI/UX Decisions

-   **Design System**: Consistent design language featuring rounded cards, a glass effect (`bg-white/80 backdrop-blur-xl`), and a primary color theme with a `ghost-border` utility.
-   **Typography**: `font-black uppercase tracking-widest` for labels.
-   **Animations**: Framer Motion for smooth UI transitions and consistent modal patterns.
-   **Notifications**: Toast notifications for user feedback.
-   **Accessibility**: Focus on semantic accessibility for forms, tables, and modals.

## Technical Implementations & Feature Specifications

-   **Authentication & Access Control**: Firebase Authentication with a 7-level hierarchical permission model.
-   **Tenant Management**: Full lifecycle management including provisioning, subscriptions, feature overrides, billing, and audit logging.
-   **Point of Sale (POS)**: Comprehensive system for cart management, diverse payment options, tax/discount calculations, customer management, repair intake, held orders, quick add stock, operator switching with RBAC, refund workflows, and warranty claims.
-   **Employee Management**: Features employee, role, time tracking, and payroll management with granular permissions.
-   **CRM (Customers)**: Enriched CRM with multi-field search, loyalty management, duplicate detection, and integrated order/invoice history.
-   **Invoices**: Full invoice lifecycle management with dynamic line items, searchable catalog, payment processing, supervisor PIN-authorized reopen, and configurable document templates.
-   **Services**: Manages a service catalog with CRUD, warranty configuration, bulk price editing, and dependency-aware delete confirmations.
-   **Repairs Module**: Full repair ticket lifecycle management with structured workflow, parts tracking, technician assignment, activity timeline, and financial summary.
-   **Reporting**: Dashboard with data visualizations.
-   **Onboarding**: Multi-step process for new tenants including plan selection, pre-configured settings, and an activation checklist.
-   **Document Template Editor**: Structured template builder for 5 document types with deterministic HTML generation, branding, and visual preview.
-   **Inventory Management**: Comprehensive system for managing various `StockItem` types (serialized, non-serialized, handset) with features like UPC/manufacturer/location tracking, min/max stock, serial number tracking, advanced filtering, stock adjustments, global movements log, inventory transfer lifecycle, stock counts, and trade-in management, all integrated with RBAC.
-   **Supply Chain Management**: Manages suppliers, purchase orders, Goods Received Notes (GRN), and Return Merchandise Authorization (RMA) with detailed tracking and financial recording, gated by RBAC.
-   **Shipping Center**: Centralized fulfillment module for managing shipments across all source documents, supporting various shipment types. Features include shipment listing with filters, detailed views, create/edit modals with package management, status transition workflows, tracking event timelines, carrier/service level configuration, cost tracking, and integration for data pre-population. Includes a provider adapter layer for external shipping carriers and a runtime mode architecture. Key features include:
    -   **PDF-Only Primary Label**: Ensures primary label generation always results in a PDF, handling conversions if necessary.
    -   **Shipment Mode Split (Provider vs Manual)**: Dynamically computes shipment mode based on rate selection, enabling or disabling provider-backed actions accordingly.
    -   **Address Editability**: Allows editing of origin and destination addresses for specific shipment statuses.
    -   **Strict Validated-Only Address Readiness (Get Rates / Purchase Label / Rate Visibility)**: The address-readiness predicates `isAddressAccepted(shipment)` and `isOriginAddressAccepted(shipment)` return `true` **only** when the corresponding `addressValidation.status === 'validated'`. A `corrected` result is treated as an **intermediate state only** and does **not** unlock Get Rates, Purchase Label, the selected-rate row, or the Available Rates panel. When the provider returns a `corrected` result, `validateSingleSide` swaps the suggested/normalized address into the shipment fields (`originAddress` or `destinationAddress`) so the operator can re-validate against the cleaned address, but it does **not** set `accepted = true` on the validation result. The operator must explicitly re-run **Validate Addresses** on the corrected address and receive a `validated` status before any rate-fetch / label-purchase / rate-display surface unlocks. The combined-validation toast surfaces this explicitly when a side comes back corrected: *"Origin: corrected — re-validate the suggested address to enable rates"*. The full operator workflow is therefore: address entered → (optional) corrected/suggested → re-validate → validated → Get Rates enabled. This rule applies symmetrically to both origin and destination — if one side is `validated` and the other is only `corrected`, Get Rates remains disabled, the rate display stays hidden behind the lock notice, and Purchase Label stays blocked.
    -   **Rate Visibility Gating by Address Validation (Provider Mode)**: In provider mode, **shipping rate visibility is gated on both origin and destination address validation**. Until both `isOriginAddressAccepted(shipment)` and `isAddressAccepted(shipment)` return true, neither the **selected rate row** ("Service · carrier · service name · price · ETA" line in the Provider & Operations panel) nor the **Available Rates panel** (the list of fetched rates with Select buttons) is shown. In their place, a small concise amber lock notice is rendered: "Rate hidden — validate origin and/or destination address to view shipping rate." The notice names exactly which side(s) still need validation. Once both sides are accepted, the selected rate row and the Available Rates panel render normally. Manual mode is **unaffected** — manually keyed carrier/service still display via the "Manual Service" row regardless of validation state, since manual mode is not provider-driven. This gating is purely a **display** layer over the existing rate fetch/select pipeline: `getRatePrerequisites` (Get Rates button gating), `handleFetchRates`, `handleSelectRate`, `availableRates` state, `selectedRate` persistence, and `getLabelPrerequisites` (Purchase Label gating) are all unchanged. If a rate was previously selected and an address is later edited (which clears that side's validation), the selected-rate row hides itself behind the lock notice until validation is re-run, preventing operators from acting on a stale rate against an unverified address.
    -   **Combined Origin + Destination Address Validation (Operator UX)**: Although validation results are stored independently on the shipment (`Shipment.originAddressValidation` for origin, `Shipment.addressValidation` for destination), the operator-facing UX is **one coherent action**. The detail panel renders a single **"Validate Addresses"** button that calls `handleValidateAddresses(shipmentId)`. Under the hood that handler invokes the internal helper `validateSingleSide` for origin first, feeds any origin suggested-address swap forward into the destination validation call, merges both result deltas into a single `updateShipment` call, and surfaces a consolidated toast (e.g. "Addresses validated — Origin: validated • Destination: corrected & accepted"). If either side fails the composite result is surfaced as an error toast with the granular summary. `getRatePrerequisites` and `getLabelPrerequisites` independently require `isOriginAddressAccepted(shipment) && isAddressAccepted(shipment)` before fetching rates or purchasing labels — Get Rates stays blocked until both sides are accepted. Editing either address clears only that side's validation so stale validations cannot mask new typos. Two separate badges ("Origin Addr" / "Dest Addr") render the per-side validated/corrected/failed status pills for auditability. The single-side `handleValidateAddress(shipmentId, side)` remains available as a legacy programmatic path but is not exposed in the UI.
    -   **Tracking Number Visibility — Unified Render Contract**: All tracking-number rendering in the app goes through a single shared component `src/components/shared/TrackingNumber.tsx`. This is the **sole render path** for carrier tracking values in the Shipping Center list chip, Shipping Center detail row ("Provider & Operations" panel), Returns Portal linked-shipment chip, and any future surface that needs to display tracking. The component enforces four hard invariants: (1) the full provider-returned tracking value is rendered **verbatim** — no masking, no truncation, no slice/substring, no padding, no normalization; (2) there is **zero carrier conditional** — UPS, FedEx, USPS, DHL, and every other carrier share the identical code path, so per-carrier masking is impossible by construction; (3) the value is always `font-mono select-all break-all` so the operator can copy the full carrier reference with one gesture, plus an explicit copy-to-clipboard icon button when `copyable` is set; (4) permission safety is enforced **upstream** by the caller (`canAccess('shipping')`, `canSyncTracking`, returns RBAC) — the component assumes the current operator is already authorized to see the value and never gates or obscures on its own. The Bulk Sync eligibility list and results tables likewise render full tracking with `break-all` wrapping. Note on EasyPost test mode: when a UPS label is purchased in EasyPost test mode, EasyPost returns a synthetic tracking code that contains literal `X` characters (e.g. `1ZXXXXXXXXXXXXXXXX`). Those `X`s are the provider's returned value — not app-side masking — and are shown verbatim just like any other tracking string. In production mode with a live `EASYPOST_API_KEY`, UPS tracking arrives as a real `1Z...` code and renders through the same shared component.
    -   **Unified Provider Status Source of Truth**: Manages and displays shipping provider statuses consistently across the UI.
    -   **Webhook/Event Processing Pipeline**: A unified pipeline for all shipping provider webhooks and tracking syncs, featuring idempotency, status mapping, status progression awareness, provider-specific parsers, a durable webhook audit log, and security measures (HMAC-SHA256 signature verification).
    -   **Bulk Sync / Reconciliation**: A secondary recovery tool for bulk tracking synchronization of eligible provider-mode shipments, featuring eligibility scoping, batch processing, and detailed result reporting.
    -   **Phone Prerequisites & Normalization**: `getLabelPrerequisites()` checks for phone on **both origin (shipper) and destination (recipient)** for UPS/FedEx. FedEx requires both — missing either triggers `PHONENUMBER.EMPTY`. Validation requires ≥10 digits. All three adapters (EasyPost, Shippo, ShipStation) normalize phones at the adapter boundary: strip non-digits, reject <10, prepend country code 1 for 10-digit US numbers. Phone source-of-truth flows: UI state → shipment object → `handlePurchaseLabel` → server → adapter `normalizePhone()` → provider API. Server-side `[purchase-label]` and `[EasyPost]` logs trace phone values at each boundary. Seed data and `STORE_ADDRESS` use proper 10-digit phones.

-   **Returns Portal / Reverse Logistics (Phase 2)**: First-class returns management module with its own dedicated route (`/returns`), sidebar entry, and permission domain (`returns`). Key architecture:
    -   **Return Domain Model**: `Return` interface with full source linkage (invoice, repair, shipment, RMA, walk-in), customer linkage, item-level detail with condition tracking, and audit trail via `statusHistory`. Forward-compatible fields for service points (`servicePointId`), customs docs (`customsInfo`), pickup requests (`pickupRequestId`), and insurance (`insuranceClaimId`).
    -   **Return Lifecycle**: Draft → Requested → Approved → Label Created → In Transit → Received → Inspecting → Completed. Also supports Rejected and Cancelled terminal states. Each transition is recorded in `statusHistory` with timestamp, performer, and notes.
    -   **Return Reasons**: Structured selection — defective, wrong_item, not_as_described, damaged_in_transit, customer_changed_mind, warranty_claim, repair_return, exchange_request, missing_parts, other.
    -   **Requested Resolution / Final Resolution**: refund, exchange, repair, store_credit, inspection_only, send_back, dispose. Requested resolution captured at creation; final resolution captured at disposition — allows divergence.
    -   **Item Disposition**: restock, refurbish, dispose, return_to_vendor, send_back_to_customer, warranty_replacement. Per-item disposition supported via `ReturnItem.disposition`.
    -   **Intake & Inspection**: Structured intake workflow — received timestamp/person, item-level condition assessment, inspection notes, inspection completion timestamp/person. Moves return through Received → Inspecting states.
    -   **Disposition Completion**: Final resolution, disposition, financial amounts (refund, store credit, restocking fee), disposition notes. Completes the return lifecycle.
    -   **Source Reference Number Resolution**: Return creation modal (`CreateReturnModal`) supports first-class source reference resolution. Operator selects source type (invoice, repair, shipment, RMA) and enters a reference number, then clicks "Resolve" to auto-populate customer, items, source linkage, and original shipment ID from the source record. Same UX pattern as ShippingCenter's source resolution. Eligible statuses: Invoice (Paid/Partially Paid), Repair (Completed/Delivered/Ready for Pickup), Shipment (Delivered outbound only), RMA (any). Walk-in source type skips resolution. Source reference number is prominently displayed in return detail view (font-mono teal text) and in the returns list table.
    -   **Source Entry Points**: Returns can be initiated from invoices (Paid status, `Invoices.tsx`), repair tickets (Completed/Delivered status, `RepairTickets.tsx`), and shipments (Delivered status, `ShippingCenter.tsx`). Each entry point uses `navigate('/returns', { state: { openCreate: true, prefill } })` with a `ReturnPrefill` object that pre-populates customer, source, items, and reason. Walk-in and RMA source types also supported via direct creation.
    -   **Shipment Integration**: Full return shipment creation flow via `CreateReturnShipmentModal` in `ReturnsPortal.tsx`. Creates a `Shipment` with `returnInfo.isReturn = true`, source linked to the return. Origin = customer address (editable), destination = store warehouse. Return shipments visible in Shipping Center with a teal "Return" badge. `originalShipmentId` links to the outbound shipment; `returnShipmentId` links to the return shipment. Provider/manual mode rules preserved — return shipments follow the same Shipping Center discipline as outbound shipments (rates, label purchase, tracking sync all work from the Shipping Center after creation).
    -   **Return/Shipment Lifecycle Synchronization — Precise 1:1 Mapping**: The return status mirrors the linked shipment status exactly, without collapsing intermediate steps. `ReturnStatus` includes `Ready`, `Label Created`, `Packed`, `Dispatched`, `In Transit`, `Delivered` so every shipment shipping-leg state has a matching return state. The auto-sync map is strictly 1:1: Shipment `Ready`→Return `Ready`; `Label Created`→`Label Created`; `Packed`→`Packed`; `Dispatched`→`Dispatched`; `In Transit`→`In Transit`; `Delivered`→`Delivered`. Packed is **not** collapsed into Label Created, and Dispatched is **not** collapsed into In Transit — the return transitions to `In Transit` only when the shipment itself reports `In Transit`, which is driven by actual carrier/provider tracking events (webhook or bulk sync), not by local dispatch actions. Auto-sync is monotonic (only advances forward via the canonical `statusOrder`). When the shipment is still Draft, the return detail shows an amber notice directing the operator to the Shipping Center for label purchase. **Manual button suppression**: when a linked return shipment exists, the manual `Ready`, `Label Created`, `Packed`, `Dispatched`, `In Transit`, and `Delivered` transition buttons are all hidden in the return detail modal — the entire shipping leg is exclusively driven by shipment auto-sync to prevent divergence. An indigo info banner explains "Shipping-leg statuses are auto-synced from the linked shipment". When no linked shipment exists (rare manual-only workflow), all manual buttons remain available.
    -   **Delivered (Awaiting Receipt) State**: The `Delivered` return status is a distinct intermediate state between `In Transit` and `Received`, indicating the package has physically arrived at the warehouse but has not yet been intaken/inspected by staff. Visually rendered with the emerald `home_storage` badge in `RETURN_STATUS_CONFIG`. The detail modal shows a clear emerald notice: "Package delivered to your facility — awaiting intake. Click Received below to begin inspection." The `Received` transition is enabled when the return is in `Delivered` status (or, for legacy returns without a linked shipment, when the linked shipment shows Delivered). The `Approved` status does not offer a direct path to `Received` — the return must go through the full shipping lifecycle (Label Created → In Transit → Delivered → Received).
    -   **Return Label Delivery to Customer**: `LabelDeliveryModal` (titled "Share Return Label") provides an operator path to share the return label with the customer. Available when the linked shipment has a label, tracking number, or label URL. Two delivery options: (1) "Copy Instructions" — copies formatted customer instructions (return authorization, carrier, tracking, label link, step-by-step instructions, items list) to clipboard; (2) "Open Email Draft" — opens a draft in the operator's default mail client with pre-populated subject and body addressed to the customer's email. **The email button does NOT send automatically** — it opens a draft for the operator to review and send. An amber info banner inside the modal makes this explicit ("the email button opens a draft in your default mail app — it does not send automatically. Direct send-from-server email is on the future roadmap.") and the button itself uses the `drafts` icon and "Open Email Draft" label to reinforce the affordance. Accessible via "Send Label to Customer" button in the return detail's shipment section.
    -   **Linked Shipment Cross-Navigation**: Both the `Original Shipment` (outbound) and the `Return Shipment` (reverse) numbers in the return detail modal are clickable buttons that navigate to `/shipping` with `state.openShipmentId = <shipment id>`. ShippingCenter's mount-time effect detects this state, selects the matching shipment, opens its detail panel on the overview tab, and clears the navigation state. This eliminates manual ID lookup when an operator wants to inspect tracking events, validate addresses, or purchase a label for a linked shipment.
    -   **Return Item Quantity & Price Visibility**: `ReturnPrefill.items` carries `unitPrice` and `maxQuantity` from the source record (Invoices pass `i.price` and `i.quantity` from line items). `ReturnItem` exposes the same fields. The `CreateReturnModal` items list renders each line as: item name, unit price (e.g. `@ $89.99`), original purchased quantity ("of 3 originally purchased"), an editable numeric quantity input bounded by `maxQuantity`, an optional condition badge, and a per-line refund context total (`$269.97`). Prevents over-return (qty cannot exceed `maxQuantity`) and gives the operator immediate refund-context arithmetic without leaving the modal. The detail modal also displays per-line unit price × qty totals for finalized returns.
    -   **Return Shipment Create Flow Cleanup**: The `CreateReturnShipmentModal` no longer collects carrier, service level, or tracking number at creation time — those fields are owned by the Shipping Center provider workflow (rate fetch → label purchase). The modal collects only origin (customer, editable), destination (store, editable line/city/state/zip/phone), packages, and notes. An info banner inside the modal explains: "Carrier, service level, and tracking number are set later in the Shipping Center when you fetch rates and purchase the label." Eliminates the previous double-entry confusion and prevents stale manual values from masking provider truth.
    -   **Permissions**: `returns` permission domain with sub-permissions: create_return, approve_return, receive_return, inspect_return, complete_return_disposition, cancel_return, create_return_shipment. Plan-gated to Growth and Advanced tiers.
    -   **Preview Mode**: All write actions gated by `isWriteBlocked`. Preview mode shows disabled state with "Preview Mode — Actions Disabled" indicator.
    -   **RepairDesk Article Insights Adapted**: Shipping from tickets/invoices entry points, label printing status triggers, inbound/outbound label distinction, two-way tracking sync readiness. Flat rate not adopted (not relevant to returns). ShipStation UX not copied (our architecture is provider-agnostic).
    -   **Phase 2 Forward-Compatibility**: Return model includes placeholder fields for service points (servicePointId), customs docs (customsInfo object), pickup requests (pickupRequestId), insurance claims (insuranceClaimId). These are not implemented but architecturally ready.
    -   **Phase 3 Forward-Compatibility**: Return lifecycle and statusHistory are automation-friendly (structured status transitions). Status timestamps stored for future SLA optimization and carrier scorecards. Return events compatible with future automation rules and batch processing.

## System Design Choices

-   **Routing**: React Router v7.
-   **State Management**: Primarily React Context API and local component state.
-   **Firebase Integration**: Firestore for backend data persistence and Firebase Authentication for user management.
-   **AI Integration**: Gemini API for AI functionalities.
-   **Server-Side Shipping API**: Express server on port 5001 handles all shipping provider operations, with credentials stored securely server-side.
-   **Configuration**: `firebase-applet-config.json` for Firebase project configuration and `firebase-blueprint.json` for Firestore schema definition.
