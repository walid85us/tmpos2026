# Overview

This project is a multi-tenant SaaS frontend platform designed to provide small to medium-sized businesses with a comprehensive suite of tools for Point-of-Sale (POS), employee and customer relationship management (CRM), inventory control, reporting, and marketing. Its primary purpose is to streamline business operations, offer actionable insights, and automate processes through a scalable, customizable, and feature-rich architecture.

# System Architecture

## Frontend

The frontend is built using React 19, TypeScript, Vite 6, and Tailwind CSS v4.

## UI/UX Decisions

-   **Design System**: Employs rounded cards, a glass effect, a consistent primary color theme, and a `ghost-border` utility.
-   **Typography**: Uses `font-black uppercase tracking-widest` for labels.
-   **Animations**: Utilizes Framer Motion for smooth UI transitions and consistent modal patterns.
-   **Notifications**: Implements toast notifications for user feedback.
-   **Accessibility**: Prioritizes semantic accessibility for forms, tables, and modals.

## Technical Implementations & Feature Specifications

-   **Authentication & Access Control**: Implemented with Firebase Authentication and a 7-level hierarchical permission model.
-   **Tenant Management**: Full lifecycle management, including provisioning, subscriptions, feature overrides, billing, and audit logging.
-   **Business Management Modules**:
    -   **Point of Sale (POS)**: Comprehensive cart management, diverse payment options, tax/discount calculations, customer management, repair intake, and robust refund/warranty workflows.
    -   **Employee Management**: Manages employees, roles, time tracking, and payroll with granular permissions.
    -   **CRM (Customers)**: Enhanced CRM with multi-field search, loyalty programs, duplicate detection, and integrated order/invoice history.
    -   **Invoices**: Supports full invoice lifecycle with dynamic line items, searchable catalog, payment processing, and configurable document templates.
    -   **Services**: Manages a service catalog with CRUD operations, warranty configuration, and bulk price editing.
    -   **Repairs Module**: Manages the full repair ticket lifecycle, including structured workflow, parts tracking, technician assignment, and financial summaries.
    -   **Inventory Management**: Comprehensive system for managing various `StockItem` types, including UPC/manufacturer/location tracking, min/max stock levels, serial number tracking, stock adjustments, inventory transfers, and trade-in management, all integrated with RBAC.
    -   **Supply Chain Management**: Manages suppliers, purchase orders, Goods Received Notes (GRN), and Return Merchandise Authorization (RMA) with detailed tracking and financial recording, gated by RBAC.
    -   **Shipping Center**: A centralized fulfillment module for managing shipments across all source documents, supporting various shipment types, status transitions, tracking event timelines, carrier configuration, cost tracking, and a provider adapter layer.
    -   **Returns Portal / Reverse Logistics**: A module for managing returns with a structured lifecycle, reasons, resolutions, item disposition, intake/inspection, and integration with return shipments.
-   **Reporting**: Features a dashboard with data visualizations.
-   **Onboarding**: A multi-step process for new tenants, covering plan selection, pre-configured settings, and an activation checklist.
-   **Document Template Editor**: A structured editor for 5 document types, offering deterministic HTML generation, branding, and visual preview.

## System Design Choices

-   **Routing**: Utilizes React Router v7.
-   **State Management**: Primarily relies on React Context API and local component state.
-   **Firebase Integration**: Leverages Firestore for backend data persistence and Firebase Authentication for user management.
-   **AI Integration**: Integrates with the Gemini API for AI functionalities.
-   **Server-Side Shipping API**: An Express server handles all shipping provider operations with secure credential storage.
-   **Configuration**: Uses `firebase-applet-config.json` for Firebase project configuration and `firebase-blueprint.json` for Firestore schema definition.
-   **Commercial Controls**: Implements a system for managing add-ons and feature entitlements with commercial governance, tenant overrides, pricing rules, and a detailed audit trail. This includes a robust internal invoice workflow for managing payments without external processors, activation modes for feature grants (after_payment or immediate), and clear distinctions for trial, paid override, and add-on statuses.
-   **Add-on Plan Inclusion Source-of-Truth Rule (locked)**:
    -   **Plans & Features Matrix is the source of truth for included-by-plan access.** Every Add-on Catalog item that is linked to a feature must use the existing `linkedFeatureId` of a feature row in `featureMatrix`. No duplicate feature rows are created when an add-on is registered. The Add-on Catalog cannot contradict the Plans & Features Matrix.
    -   **Compatible Plans locking (Add-on Catalog editor).** In the Add-on edit modal, plans where the linked feature is included by plan are rendered as **locked** (lock icon, emerald background, disabled, helper text: "Locked plans include the linked feature in the Plans & Features Matrix. To change inclusion for those plans, edit the Plans & Features Matrix."). Locked plans cannot be unchecked from the Add-on Catalog. On save, locked plans are auto-merged into `compatiblePlans` so the persisted record can never opt out of an included plan. Non-included plans remain freely toggleable as optional add-on availability.
    -   **Tenant Features tab label rules.**
        -   **Plan includes the linked feature** → primary pill `Included by Plan`, secondary emerald pill `Add-on` (when the catalog Add-on is `governanceStatus === 'active'`), and **no** Trial / Paid Override actions (the feature is already included).
        -   **Plan does not include the linked feature, tenant's plan IS in `compatiblePlans`** → primary pill `Not in Plan`, secondary emerald pill `Add-on`, with `Trial` and `Paid Override` actions enabled.
        -   **Plan does not include the linked feature, tenant's plan is NOT in `compatiblePlans`** → the entire row is hidden. No `Not in Plan`, no `Add-on`, no Trial / Paid Override, no `Add-on available`, no `Plan upgrade required`.
        -   **Disabled or archived catalog Add-ons** are treated as ineligible: derived rows do not surface as available on any tenant, and an in-plan feature whose linked Add-on is disabled/archived shows `Included by Plan` only (no active "Add-on" badge).
    -   **Compatible Plans edits take effect immediately.** Removing a tenant's plan from an Add-on's `compatiblePlans` (where allowed — i.e. the plan is not locked) hides every add-on-derived row for that tenant on the next render. Adding a plan immediately surfaces the row with `Not in Plan` + `Add-on` and `Trial` / `Paid Override` actions where eligible.
    -   **Historical state handling.** Existing trial / pending payment / paid override / revoked records on a feature whose linked Add-on is no longer eligible for the tenant are hidden from the active Tenant Features list (both the active grid and the Override History list). The historical record itself is preserved in the SaaS Subscription Invoices section of the Billing tab and in the Commercial Audit log.
    -   **Resolver-side enforcement.** `resolveTenantFeature` strips Add-on attribution (`addOn`, `defaultPrice`) for plan-incompat tenants on `trial_active`, `paid_active`, `pending_payment`, and `disabled_by_plan` branches; `type === 'addon'` overrides downgrade to `enabled_by_paid_override` so no row's primary pill ever reads "Add-on" when the Add-on is plan-incompatible. Grant handlers (`handleEnableTrial`, `handleEnablePaidOverride`) only persist `addOnId` on the override + invoice + audit when the Add-on is currently eligible for the tenant.
    -   **The labels `Add-on available` and `Plan upgrade required` are not used anywhere** on the Tenant Features tab. The only allowed label set is: `Included by Plan`, `Add-on`, `Not in Plan`, `Trial`, `Paid Override`, `Pending Payment`, `Revoked`, `Trial Expired`, `Not Available`, `In Development`, plus `Custom Price` where applicable.
    -   The `Custom Price` pill (rendered separately below the price line) is independent of the add-on label and only appears when the tenant-specific price diverges from the catalog price.
    -   The Tenant Features tab is a commercial-entitlement surface only — it must NOT render permission chips, locked sub-permission badges, or any "Permissions Impact" block. Role-permission detail lives exclusively in the Store Permissions Matrix, which follows resolved entitlement (in-plan / active trial / active paid override expose linked permissions; pending payment / cancelled-before-activation / incompatible / disabled / archived do not).
-   **Add-on Governance & Archive Protection Rule (locked)**:
    -   **Three governance states.** Every Add-on Catalog item has a `governanceStatus` of `active`, `disabled`, or `archived`. There is no `draft` governance state — add-ons are commercially "live" the moment they are created (defaulting to `active`).
    -   **Lifecycle (PM) is a separate field.** PM `lifecycle` (`draft`, `planned`, `in_development`, `active`, `deprecated`, `archived`) is an internal product-management classification and is edited independently in the Add-on form. It does not gate tenant visibility — only `governanceStatus` does. The two fields are mirrored only when archiving/restoring (archive forces `lifecycle: 'archived'`; restoring an archived add-on flips PM lifecycle back to `active`).
    -   **Default on create.** New add-ons default to `governanceStatus: 'active'` and `lifecycle: 'active'`.
    -   **Linked Feature is optional.** The Add-on form's Linked Feature dropdown allows `— none (standalone capability) —`. Picking an existing feature links the add-on to it (so the matrix's plan-inclusion toggles drive `included_by_plan`). Leaving it as None turns the add-on into a **standalone capability**.
    -   **Standalone capability registration (no duplicates).** When the System Owner saves an add-on with Linked Feature = None, `saveAddOn` auto-registers a capability row in `featureMatrix` using a stable key `cap_<addonId>` (slug-safe, derived once from the add-on id and reused on every subsequent edit), with `source: 'custom'`, `lifecycle: 'implemented'`, and all `planAvailability` initialized to `false`. The add-on's `linkedFeatureId` is then set to that capability key. Re-saving or editing the add-on never duplicates the row — the existing key is reused. Registration emits a `feature_registered_for_addon` audit entry. The Add-on modal also still exposes an inline "+ New" register action when the editor wants to name the linked feature explicitly. The Save button is no longer gated on Linked Feature being chosen.
    -   **Tenant visibility.** `resolveTenantFeature` only requires plan inclusion (or a tenant override) to return `included_by_plan`. `isAddOnOfferable` returns `true` only when `governanceStatus === 'active'`, so `listAvailableAddOnsForTenant` excludes disabled and archived add-ons. The Tenant Features tab hides feature rows whose linked add-on is the only path to availability AND that add-on is not `active` AND not in `compatiblePlans` for the tenant's plan.
    -   **Archive dependency categories.** Archive is **blocked** while ANY of the following are true: (1) the linked feature/capability is included by plan in any active plan in the Plans & Features Matrix; (2) the add-on has any `compatiblePlans` selected; (3) there are non-revoked tenant overrides referencing the add-on (by `addOnId`) or its linked feature (by `featureId`) of type `trial`, `overridden`, `paid_override`, `addon`, or `pending_payment`.
    -   **In-app blocked-archive message.** Attempting to archive a blocked add-on (via the Archive governance button OR the modal Governance Status dropdown) opens a red Archive-Blocked modal listing each dependency category with names and counts, and offers only a "Got it" dismissal. The transition is refused and `addon_archive_blocked` is pushed to the audit trail with a structured reason string. Once all dependencies are cleared, the same flow opens the standard Archive confirmation.
    -   **Audit trail.** All add-on transitions are auditable via `commercialAuditLog` with actions: `addon_created`, `addon_updated`, `addon_default_price_changed`, `addon_status_changed`, `addon_archive_blocked`, and `feature_registered_for_addon`.
-   **Add-on Runtime Linkage & Delete/Archive Matrix Behavior (locked)**:
    -   **Linked Feature runtime meaning.** The Linked Feature dropdown in the Add-on form selects the single feature row that the add-on grants at runtime. Picking an existing feature row routes the add-on through that row's plan-availability matrix and `resolveTenantFeature` branch. Picking `— none (standalone capability) —` causes `saveAddOn` to register a stable capability row `cap_<addonId>` (slug-safe, derived from the add-on id, never duplicated on re-save) and link the add-on to it. The dropdown surfaces inline helper text describing this runtime meaning so editors are never surprised about what they wired up.
    -   **Parent-feature warning + sub-feature suggestions.** Selecting a parent feature (e.g. `feature_shipping`) is almost always a misconfiguration — entitlements should attach to a specific sub-feature instead. The modal renders a yellow warning whenever a parent id from `PARENT_FEATURE_IDS` is chosen, and proposes one-click sub-feature suggestions from `SUB_FEATURE_HINTS` that exist in `featuresData` and are `lifecycle === 'implemented'`. The audit action `addon_linked_feature_changed` is emitted whenever an existing add-on's linkage is changed in `saveAddOn`.
    -   **Delete vs Archive policy.** Archive preserves history (billing/audit) and is the recommended path. Delete is permanent and is BLOCKED while any of these dependency categories exist: (1) the linked feature/capability is included by plan in any active plan; (2) the add-on has any `compatiblePlans` selected; (3) there is an active tenant trial referencing the add-on or linked feature/capability; (4) there is an active paid override referencing it; (5) there is a pending payment referencing it; (6) there is an open / overdue / pending internal SaaS subscription invoice referencing it. The Delete button opens a Delete-Blocked modal listing each present category and emits `addon_delete_blocked` with a structured reason string.
    -   **Standalone capability cleanup on delete.** When delete proceeds and the add-on's `linkedFeatureId` is a standalone `cap_<addonId>` row that no other add-on links to (and that row is `source: 'custom'`, `lifecycle: 'implemented'`), the row is removed from the matrix alongside the catalog record and `standalone_capability_row_deleted` is emitted. When the add-on is linked to a pre-existing feature row instead, the row is preserved (other plans/permissions may use it) and only the catalog record is removed. The successful `addon_deleted` audit entry's `note` describes what was deleted vs preserved so reviewers can reconstruct the action.
    -   **Archived standalone capability rows.** When an add-on is transitioned to `archived` and a `cap_<addonId>` row exists in the matrix, that row stays in the Plans & Features Matrix as the System Owner's source-of-truth record. It renders read-only: plan toggles are disabled (amber lock icon), the lifecycle dropdown is disabled, an "Add-on Archived" badge appears, and helper text reads "This add-on is archived. Its generated capability row is read-only — restore the add-on from the Add-on Catalog to re-enable editing." The lock event is emitted exactly once at archive time as `standalone_capability_row_archive_locked` (from both the governance buttons and the modal Status dropdown). Restoring the owning add-on re-enables editing.
    -   **Handler-level protection (defense in depth).** The matrix mutation handlers `toggleFeature`, `changeLifecycle`, and `removeFeature` short-circuit and ignore any request whose target is an archived standalone-cap row, regardless of UI state. A stale UI, bulk path, or future programmatic caller cannot mutate `planAvailability`, `lifecycle`, or remove the row by going around the disabled controls — `isArchivedStandaloneCapRow(featureId)` is the single source of truth and returns true only for `cap_<addonId>` rows whose owning add-on is currently `governanceStatus === 'archived'`.
    -   **Linked-feature add-on non-regression.** Archiving an add-on that links to a pre-existing feature row (not a generated `cap_<addonId>` row) does NOT lock or hide the underlying feature row. The row is shared and other plans / Store Permissions Matrix entries still own it; only the active "Add-on" badge is suppressed for that linkage per the Tenant Features tab label rules.
    -   **Show archived/deprecated filter.** The Plans & Features Matrix Features tab hides archived/deprecated rows by default — but archived standalone-cap rows are ALWAYS visible regardless of this toggle, because they are the System Owner's audit-clarity record. The "Show archived/deprecated" toggle re-introduces all other archived/deprecated rows for restore workflows.
    -   **New audit actions.** `addon_deleted`, `addon_delete_blocked`, `addon_linked_feature_changed`, `standalone_capability_row_deleted`, `standalone_capability_row_archive_locked` extend `CommercialAuditAction` and follow the same actor / target / reason conventions as the rest of the commercial audit log.
-   **Add-on Implementation Readiness (locked)**:
    -   **Why.** Catalog add-ons frequently exist as commercial placeholders or partial implementations that aren't actually wired into runtime behavior. Granting a Trial / Paid Override on a placeholder gives the tenant a paid entitlement against a feature that doesn't exist yet. The Implementation Readiness model makes this risk explicit and gates tenant grants accordingly.
    -   **Readiness model.** Each add-on has a `readinessStatus` of one of: `runtime_backed` (linked to implemented capability with full runtime backing — safe), `partially_backed` (capability exists but one or more runtime backing items are missing — warn on grant), `parent_feature_linked` (linked to a broad parent module — may unlock more than intended, warn on grant), `implementation_required` (catalog placeholder — real app functionality must be built before tenant grant; BLOCKED unless `allowManualPresaleGrant`), `commercial_placeholder` (roadmap / pre-sale listing only — BLOCKED unless `allowManualPresaleGrant`). The status is stored on the AddOn record and is editable from the Add-on Create/Edit modal. `deriveSuggestedReadiness(addon)` computes a suggestion from `linkedFeatureId` + `KNOWN_CAPABILITY_REGISTRY` + the runtime checklist; the modal surfaces a one-click "Use suggested" button so System Owners can opt into the suggestion or override it (override emits `addon_readiness_overridden`).
    -   **Runtime Backing Checklist.** Each add-on carries a 9-item checklist (`capability_key`, `plan_matrix_row`, `runtime_ui_surface`, `entitlement_check`, `permission_dependency`, `data_model`, `audit_behavior`, `billing_behavior`, `documentation`), each item is `complete | missing | not_required | unknown`. The modal renders the checklist as a compact 4-way select per item. The seed catalog ships sensible per-add-on defaults (e.g. API → all complete, Multi-Store / White-Label → core plumbing complete + UI/data missing, SMS / Priority Support → commercial placeholder).
    -   **Known Capability Registry.** `KNOWN_CAPABILITY_REGISTRY` in `src/owner/mockData.ts` is a 28-entry map of `{ featureId, displayName, parentModule, isParentFeature, runtimeSurfaceExists, permissionIds, planMatrixRowExists, notes }` covering the implemented surfaces of the app (Plans & Features matrix rows, Inventory, POS, Repair, Customers, Reports, Shipping sub-features, etc.). It is the single source of truth used by readiness derivation and by the parent-feature warning. Helpers `isKnownImplementedCapability(featureId)` and `isParentFeatureCapability(featureId)` are exposed.
    -   **Grant Safety on Tenant Features tab.** `getAddOnGrantSafety(addon)` returns `{ allowed, requiresWarning, warningText, blockReason, shortBadge }` and is consulted on the Tenant Detail Features tab for every row whose linked add-on isn't `runtime_backed`. The rule is: `runtime_backed` → allowed (no warning); `partially_backed` → allowed + amber warning + "Runtime Not Connected" chip; `parent_feature_linked` → allowed + amber warning + "Parent Feature Warning" chip; `implementation_required` → BLOCKED unless `allowManualPresaleGrant === true`; `commercial_placeholder` → BLOCKED unless `allowManualPresaleGrant === true`. Blocked rows replace the Trial / Paid Override / Re-trial / Re-grant Paid buttons with a compact rose chip carrying the block reason in the tooltip. Allowed-with-warning rows render the buttons in amber and surface the short badge alongside, with the warning in the tooltip. Grant safety is **additive** to existing eligibility (governance status, plan compatibility, lifecycle, prior overrides) — none of those rules are changed.
    -   **Manual / Presale Grant.** When `readinessStatus` is `implementation_required` or `commercial_placeholder`, the modal exposes an `Allow manual / presale tenant grant` checkbox that flips `allowManualPresaleGrant`. This unblocks tenant grant for that single add-on, with the warning still surfaced on the Tenant Features tab. Toggling emits `addon_manual_presale_allowed`.
    -   **Parent-feature acknowledgement.** When the linked feature is in `PARENT_FEATURE_IDS`, the modal also exposes an `Acknowledge parent feature link` checkbox that flips `parentLinkAcknowledged`. The catalog still warns other System Owners on the Tenant grant screen via `getAddOnGrantSafety` (the acknowledgement does not silence the warning), but the read flag is preserved on the record. Toggling emits `addon_parent_link_acknowledged`.
    -   **Implementation Brief generator.** For any add-on whose readiness is not `runtime_backed`, the modal renders a `Generate Brief` button. `generateImplementationBrief(addon, linkedFeatureName)` in `src/owner/readiness.ts` produces a structured Markdown brief covering Add-on Identity, Linked Capability, Intended Tenant / User, Business Goal, Current Readiness, Runtime Backing Checklist, Missing or Unknown Items, Proposed UI Surfaces, Required Data Model, Required Permissions, Required Plan / Entitlement Gating, Required Tenant Visibility Behavior, Required Billing Behavior (internal SaaS invoice only — no external Stripe / customer storefront billing), Required Audit Behavior, a Manual QA Checklist, and a Replit Implementation Prompt Starter that the System Owner can paste into a follow-up build prompt. Copying the brief emits `addon_implementation_brief_generated` with the byte length so the trail captures the export.
    -   **Catalog readiness badge.** Each Add-on Catalog card renders a compact readiness badge alongside the lifecycle / governance badges, color-coded by status (emerald = runtime_backed, amber = partially_backed, orange = parent_feature_linked, rose = implementation_required, slate = commercial_placeholder). Tooltip carries the long description. `READINESS_LABELS / READINESS_DESCRIPTIONS / READINESS_BADGE_STYLES` in `src/owner/readiness.ts` are the single source of truth for badge copy + styling.
    -   **New audit actions.** `addon_readiness_changed`, `addon_runtime_checklist_updated` (per-checklist-item diff captured in the note), `addon_manual_presale_allowed`, `addon_implementation_brief_generated`, `addon_parent_link_acknowledged`, `addon_readiness_overridden` extend `CommercialAuditAction` and are emitted from `saveAddOn` whenever the corresponding field changes (diff-driven). New add-ons emit an initial `addon_readiness_changed` capturing the Day-0 readiness assumption.
    -   **Out of scope / non-regression.** Readiness governance is purely an authoring + gating layer — it does NOT generate real app functionality, does not call any AI service, does not touch Stripe / external payment processors / customer-facing storefront billing, does not change the Shipping module behavior, does not change the Plans & Features Matrix source-of-truth, does not change paid-override invoice flow, does not change archive/delete add-on rules, and does not change the Store Permissions Matrix. The Tenant Features tab continues to be a commercial entitlement surface only; readiness chips are compact and tooltip-driven, not verbose Permissions Impact blocks.
