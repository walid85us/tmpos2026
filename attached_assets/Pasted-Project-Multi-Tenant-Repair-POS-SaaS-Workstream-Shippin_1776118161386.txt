MANDATORY CORRECTIONS
--------------------------------------------------

A) SHIPPING PROVIDERS MUST LIVE UNDER SHIPPING CENTER
This is non-negotiable.

Required:
- Shipping Providers must be surfaced under the Shipping section / Shipping Center itself
- Not merely discoverable through the generic Settings area
- Implement a real native Shipping Center settings subsection / child route / operational settings panel for providers
- Governance/security may still use the correct admin architecture underneath, but operational ownership must visibly belong to Shipping Center
- The user must be able to stay in Shipping Center context to manage providers
- Do not satisfy this with only a deep link to general Settings unless the final experience is visibly a Shipping subsection/child context

This must be materially true in the UI, not only conceptually true in the code.

B) PDF-FIRST LABEL ACTION MUST BE HONEST AND CORRECT
Required:
- The main label action must open a real PDF artifact when PDF is available
- If provider/test mode returns only PNG or the app only has PNG for that shipment, the UI must not present that action as “Open PDF Label”
- Main action labeling must match the actual artifact type
- Prefer PDF-first when both are available
- Keep thermal-label readiness in mind
- Do not leave a misleading PDF label that opens PNG

C) HARDEN POST-DISPATCH LIFECYCLE GOVERNANCE
This is the most important business-rule correction.

Required lifecycle rule:
1. Before carrier acceptance/scan:
- store can prepare shipment
- select rate
- purchase label
- pack
- dispatch
- cancel only if allowed and only before carrier acceptance/scan

2. After the shipment is Dispatched and the carrier/provider has accepted/scanned it:
- store must NOT manually change shipping statuses such as:
  - In Transit
  - Delivered
  - Exception
  - Return-related future statuses if any
- these statuses must come from provider/carrier event history
- shipment lifecycle after dispatch must be provider-driven, not manually store-driven

3. Cancellation rule:
- cancellation/void is only allowed before the first provider/carrier acceptance/scan event
- once scanned/accepted by carrier, cancellation must be blocked
- if future refund/void logic exists, keep it coherent and do not fake cancellation success

Implement this with real lifecycle-aware gating, not cosmetic hiding only.

D) PROVIDER-DRIVEN POST-DISPATCH STATUS CONTROL
Required:
- after Dispatched, provider/carrier events should be the authoritative source for later shipment states
- if a provider event indicates in transit, delivered, or exception, the shipment should map/update accordingly
- if no provider event exists yet, the UI should show that clearly rather than allowing manual store-side progression
- manual notes/events may still exist, but they must not masquerade as authoritative carrier progression

E) TEST MODE VS LIVE MODE OPERATIONAL CLARITY
Required:
- in test mode, explain clearly when live carrier scans are not naturally available
- provide a truthful, test-safe provider-event validation mechanism if needed
- distinguish between:
  - manual event
  - provider event
  - test-provider simulated event
  - no provider event yet
  - carrier scan not yet received
- do not mislead the user about whether a real carrier event has occurred
