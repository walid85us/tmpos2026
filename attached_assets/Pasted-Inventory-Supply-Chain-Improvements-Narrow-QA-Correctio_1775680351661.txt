Remaining corrections to implement

A) Transfer editability before received
1. Current issue
- transfers are still not truly editable before received/finalized

2. Required behavior
- as long as a transfer is not Received/finalized, it must remain editable where operationally appropriate
- make this practical and explicit, not cosmetic
- user should be able to edit the relevant transfer details and line items through the intended edit flow before final receipt
- preserve lifecycle guardrails, but do not lock the transfer too early
- if some fields should become read-only after Send or In Transit, define that clearly and keep the remaining editable fields operational

3. UX requirement
- keep the edit flow clean and modal/drawer/form-based, not awkward inline corruption under the list

B) Completed inventory count adjustment closure
1. Current issue
- practical adjustment closure still fails once count status is Completed

2. Required behavior
- a completed count with discrepancies must still support a clear operational closure path
- user must be able to:
  - review the discrepancy outcome
  - see what was adjusted
  - complete/confirm any remaining closure steps in a practical way
- do not leave Completed as a dead-end where the user can see discrepancy but cannot finalize the adjustment outcome properly

3. Desired model
- keep count completion operationally meaningful:
  - in-progress count
  - review/confirm
  - completed with discrepancy results
  - visible adjustment outcome / closure summary
- if a dedicated close-out modal or action is needed, add it cleanly

C) Trade-in ID preview and camera behavior
1. Current issue
- uploaded ID preview is cropped / partial
- Camera action should connect to the device camera when available

2. Required behavior
- uploaded/captured ID preview must show the full image within the preview area
- use a fit/contain-style preview behavior instead of clipping the image
- keep the preview readable and operational

3. Camera behavior
- clicking Camera must invoke device camera capture when the device/browser supports it
- preserve desktop/file-picker fallback when direct camera is unavailable
- do not regress the working upload path

D) RMA edit flow UX
1. Current issue
- when clicking Edit on an RMA, edit behavior still appears under the RMA listing
- user wants this in popup/modal/drawer form

2. Required behavior
- if an RMA is editable (such as Pending / not shipped), clicking Edit must open a proper popup/modal/drawer
- do not expand edit UI awkwardly inline under the listing
- keep the edit experience consistent with the rest of the app’s detail/edit pattern

E) Dashboard low-stock action popup
1. Current issue
- current behavior still routes away to Inventory and opens stacked detail/adjust flows
- desired behavior is a true in-dashboard popup/action menu

2. Required behavior
- clicking a low-stock item on Dashboard must open a compact popup/modal/action sheet without leaving Dashboard
- the popup must offer:
  - View Stock Details
  - Adjust Stock
- this must happen on the Dashboard surface itself
- do not redirect to Inventory automatically for this action

3. Permission-aware behavior
- if user has permission to adjust stock:
  - show both View Stock Details and Adjust Stock
- if user does not have permission to adjust stock:
  - show only allowed options or a clean blocked state
- do not show misleading actions

4. View Stock Details behavior

- keep it clean and consistent

5. Adjust Stock behavior
- from the popup, choosing Adjust Stock must open the proper stock-adjust flow directly if authorized
- do not stack broken overlapping windows
- keep modal layering clean and intentional
