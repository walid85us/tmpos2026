CURRENT QA HOLD ITEMS
=====================

A) Domains design / UI
User feedback:

* Domains works but does not yet meet the desired modern SaaS UI vision.

Correction scope:

* Do a small polish only:

  * improve visual hierarchy where simple
  * make lifecycle tabs/cards feel more clickable
  * clarify active states
  * improve empty/help text
* Do not do a full redesign now.
* Full modern SaaS UI redesign can be a separate design-system / Figma-led future pass.

B) Domain lifecycle tabs not clickable
Observed:

* Lifecycle tabs show but are not clickable.

Expected:

* Every lifecycle tab should be clickable.
* Clicking a lifecycle tab applies the corresponding lifecycle filter.
* The selected tab is visibly active.
* The visible list count matches the selected tab count.
* Clear All resets the lifecycle tab to All.
* No invisible filters.

C) Add Domain subdomain/root domain logic
Observed:

* Add Domain works, but subdomain handling is too generic.
* User expects the subdomain name to be editable while the main/root domain is locked/selected.

Expected:
When Kind = Root/Main Domain:

* User enters root domain, e.g. storename.com.
* This root domain becomes manageable in Domains.
* It can be used as the base for subdomains.
* It should have its own lifecycle/readiness/SSL status.

When Kind = Subdomain:

* User selects an existing root/main domain from a dropdown.
* Root/main domain field is locked/read-only.
* User enters only the subdomain label, e.g. portal, app, support.
* The full hostname is generated as:
  subdomainLabel + "." + selectedRootDomain
  Example:
  portal + storename.com = portal.storename.com
* User should not manually edit the locked root domain in the subdomain flow.
* Validate duplicate full hostname.
* Validate subdomain label format.
* Show live preview of full hostname before save.

If no root/main domain exists:

* Show helpful empty state:
  “Create a root domain first before adding subdomains.”
* Disable subdomain save until root domain is available.

D) Main/root domain management inside Domains
Expected:

* Domains page should clearly distinguish:

  * Root/Main Domains
  * Subdomains
* Root domains should be visible/manageable in the same Domains page.
* Detail drawer should show whether the domain is root/main or subdomain.
* Subdomain detail should show parent/root domain.
* Root domain detail should show related subdomains if available.
* No real DNS/SSL automation.

Use the existing data model if possible.
If a new optional field is needed, keep it backward-compatible:

* parentDomainId?
* rootDomain?
* domainRole?: root | subdomain
  or equivalent.

Do not break existing domain records.

E) Platform Settings default-baseline maintenance
Observed:

* Reset to default works, but there is no place/process to maintain default settings.

Expected:
Add a safe “Default Baseline” governance process.

Preferred implementation:

* Add a “Default Baseline” panel or mode inside Platform Settings.
* It shows registry default values.
* It explains:
  “Defaults are the baseline used by Reset to Default.”
* Allow authorized users to update default baseline values only if safe and low-risk.

If editable:

* Gate by edit_platform_settings or Manage/Full threshold if available.
* Show Review Default Change modal.
* Show old default → new default diff.
* Require high-risk acknowledgment if the setting is high-risk.
* Audit:
  platform_setting_default_updated
  or existing safe equivalent.
* Store default overrides separately from current settings, e.g.
  platform_settings_defaults_v1
* Reset to Default should use the maintained default baseline:
  default override if present, otherwise registry default.

If this is too large/risky:

* Implement read-only Default Baseline panel now.
* Add visible “Default baseline editing is future” explanation.
* But preferred is a lightweight editable baseline if safe.

Do not change runtime enforcement.
Do not claim defaults are externally enforced.

F) Command Center domain deep-linking
Observed:

* Command Center domain issue click opens Domains page only.
  Expected:
* It should open the relevant domain record or at least a visible filtered domain context.

Required:

* Command Center domain signal should deep-link to Domains with:

  * domain id, or
  * tenant id + domain filter, or
  * issue filter if exact domain id is unavailable
* DomainsPage should read URL params once, apply visible filter/open drawer, then clear sticky URL if that is the app pattern.
* If domain id is invalid/stale:

  * show dismissible not-found notice
  * do not silently open wrong domain
* No invisible filters.
