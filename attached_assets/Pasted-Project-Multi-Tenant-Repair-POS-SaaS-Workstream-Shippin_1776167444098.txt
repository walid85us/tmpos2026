MANDATORY CORRECTIONS
--------------------------------------------------

A) FINALIZE SHIPPING PROVIDERS UNDER SHIPPING CENTER
Required:
- Make provider management fully native under Shipping Center as a Shipping > Settings > Providers subsection / child route / clearly owned panel
- This should be the primary home for provider management in the user experience
- Governance/security/permissions must remain correct
- Avoid duplicate competing sources of truth
- The older generic settings entry, if it remains, should defer to or route into the Shipping-owned experience

B) PACKED STATUS GATING AFTER LABEL PURCHASE
Required:
- Packed must not be enabled before label purchase succeeds
- Shipment flow should enforce:
  - Ready / Prepared
  - Purchase Label
  - then Packed becomes available
- Do not allow Packed as a premature operational status before the shipment has a purchased label
- Preserve other lifecycle controls already implemented

C) CLEAN PDF-ONLY PRIMARY LABEL HANDLING
Required:
- The main label artifact/action for the operational workflow must use PDF only
- Do not present or open PNG as the primary operational label output
- If provider returns multiple formats, choose PDF for the main artifact/action
- UI labeling must match reality
- Preserve thermal-label readiness and operational usability

D) FULL TRACKING NUMBER VISIBILITY
Required:
- Show the full tracking number to authorized operational users
- Remove truncation/masking in all relevant shipment detail/list surfaces where operators need the full value
- Keep lock behavior after label purchase
- Preserve permission safety for unauthorized users

E) HARD BLOCK CANCELLATION AFTER CARRIER ACCEPTANCE / PROVIDER PROGRESSION
This is critical.

Required:
- Cancellation is allowed only before carrier acceptance/first scan/provider progression
- After the shipment has provider evidence of acceptance/scan/progression, cancellation must be blocked
- Delivered must absolutely block cancellation
- Any provider-fed state implying handoff/progression must block cancellation
- Implement this with lifecycle-aware enforcement, not only cosmetic UI hiding

At minimum, cancellation must be blocked after provider states such as:
- accepted / first scan / carrier possession
- in transit
- delivered
- exception states where cancellation no longer makes operational sense

F) LIGHTWEIGHT DESTINATION ADDRESS AUTOCOMPLETE
Required:
- Add destination-address autocomplete/suggestion behavior for the shipment destination address entry flow
- When the user starts typing an address, show a suggestion list
- When the user selects a suggestion, prefill the related address fields such as:
  - address line
  - city
  - state/region
  - postal code
  - other relevant fields if available
- Keep the implementation lightweight and operational
- Do not redesign the whole shipping form
- Do not overbuild an entire maps product
- The goal is to reduce data-entry friction and improve address correctness

If no live autocomplete provider is configured, implement the cleanest architecture-ready path possible without fake success, and make the UX truthful.
