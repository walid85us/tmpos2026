PART A — PLATFORM OPERATIONS COMMAND CENTER
==================================================

Create a new Command Center view inside Platform Operations & Security.

Recommended location:
- Platform Operations & Security → Overview
or
- Platform Operations & Security → Command Center

This should become the “beating heart” of System Owner operations.

Required sections:

1) Platform Health Overview

Show cards such as:
- Open critical support cases
- Overdue support cases
- Elevated audit/security events
- Tenants needing attention
- Domain issues summary
- Pending platform actions
- Add-on readiness warnings
- Billing/add-on pending payment summary if safe
- Recent operational activity

Important:
Do not fake data.
Use existing mock/session/local data sources where available.
If data is not available, show truthful empty state or “No active issues”.

2) Needs Attention Queue

Create a prioritized queue of items that need System Owner attention.

Item types may include:
- critical support case
- overdue support case
- high-risk audit event
- failed domain verification
- pending domain verification
- platform security note
- add-on readiness warning
- pending invoice/payment if already available
- team governance warning if data exists

Each item should show:
- priority
- type
- tenant if applicable
- title
- short reason
- age / due indicator if applicable
- action button / link

Priority levels:
- critical
- high
- medium
- low

3) Tenant Risk Summary

Create a tenant risk table or card group.

Risk signals may include:
- open support cases
- critical/high support cases
- recent warning/critical audit events
- failed/pending domains
- pending billing/add-on payment if safe
- add-on readiness issues if applicable
- recent platform notes

Risk status:
- Healthy
- Watch
- At Risk
- Critical

Do not invent risk.
Use transparent derived scoring from available data.

Show a small explanation:
“Risk is derived from support, audit, domain, and operational signals available in this system.”

4) Workflow Health Summary

Show operational workflow health:
- support case workload
- audit/security workload
- domain workload
- recent settings/team/domain/support changes
- unresolved security notes

This is not real infrastructure monitoring.
Label clearly:
“Operational workflow health — not infrastructure uptime monitoring.”

5) Quick Actions

Add quick actions:
- Create Support Case
- Open Audit & Security
- Open Support Tools
- Open Domains
- Open Platform Settings
- Open Team Management

No destructive actions directly from Command Center in this pass.

PART B — AUDIT & SECURITY UPGRADE
==================================================

Enhance the existing Audit & Security surface.

Do not replace it with a superficial dashboard.
Make it operational.

Required additions:

1) Saved Audit Views

Add saved views such as:
- Critical / Warning Events
- Add-on Governance
- Billing / Invoice
- Support Activity
- Domain Activity
- Team Management
- Platform Settings
- Security Notes

Saved views may be predefined in this pass.
Custom saved views can be future scope unless easy and safe.

2) CSV Export

Add export CSV action for the currently filtered audit view.

Export should include safe fields:
- timestamp
- severity
- category
- action
- actor
- tenant
- target
- status/result
- note/summary

Do not include secrets, API keys, raw payloads, or sensitive server data.

3) Related Event Timeline

When opening an audit event detail drawer, show related events when possible:
- same tenant
- same target/entity
- same actor
- same support case/domain/add-on/invoice if known

If no related events:
- show “No related events found.”

4) Actor Profile Drawer / Panel

From an audit row, allow opening actor context if available.

Show:
- actor name/email if available
- actor role if available
- recent actions
- severity breakdown
- categories touched
- last activity

Do not overclaim auth/session information.
If unavailable, show Unknown / Not tracked.

5) High-Risk Event Flags

Add high-risk flags to audit events derived from:
- critical severity
- security category
- team role/status changes
- archive/delete blocked events
- billing/payment activation changes
- domain failed/disabled
- support critical escalations

Flag label:
- High Risk
- Needs Review
- Critical

Use clear, deterministic rules.

6) Create Support Case from Audit Event

Allow System Owner to create a support case from an audit/security event.

Required:
- prefill tenant if available
- prefill title from audit action
- prefill category based on audit category
- prefill severity based on audit severity
- attach reference to audit event
- audit that the support case was created from an audit event

If support case already exists for the same event, prevent duplicates or show a warning.

7) Security Notes Upgrade

Existing security notes should remain.
Enhance if needed:
- associate note with audit event if created from event detail
- show note timeline
- audit note add/delete

PART C — SUPPORT TOOLS UPGRADE
==================================================

Enhance Support Tools into a more mature internal support operations surface.

Required additions:

1) SLA Timers

Add support case SLA metadata.

SLA fields:
- firstResponseDueAt
- resolutionDueAt
- firstRespondedAt
- resolvedAt
- slaStatus:
  - on_track
  - at_risk
  - overdue
  - met
  - missed
  - paused
  - unknown

Derived behavior:
- if case is open and due time is near, show At Risk
- if due time has passed, show Overdue
- if resolved before due time, show Met
- if resolved after due time, show Missed
- if due fields unavailable, show Unknown

Do not fake time data.
Use seeded/mock due dates where practical.

2) Saved Support Views

Add views:
- All Open
- Critical
- Overdue
- Waiting on Tenant
- Waiting on Internal Team
- Recently Updated
- Resolved / Closed
- My Assigned Cases if assigned user data exists

3) Macros / Response Templates

Add internal support macros/templates.

Examples:
- Ask tenant for more information
- Domain verification instructions
- Billing follow-up
- Shipping provider credential issue
- Security review acknowledgment

This pass should not send emails.
Macros should insert text into internal note or response draft area only.

Label clearly:
“Internal template only — no external message sent.”

4) Case Escalation

Add escalation action:
- Escalate case
- De-escalate case
- escalationReason
- escalatedAt
- escalatedBy

Escalated cases should appear in Needs Attention queue.

5) Related Entities Panel

In support case detail, show related records where available:
- tenant
- audit events
- domain records
- invoices/pending payments if available
- add-ons/features if referenced
- shipping/provider issues if referenced
- security notes if linked

If unavailable:
- show empty state truthfully.

6) Tenant Health Panel

Add tenant health panel inside Support Tools or support case detail.

Show:
- open cases count
- critical/high cases
- recent audit warnings
- failed/pending domains
- pending payments if available
- recent add-on readiness warnings if available

Use same risk logic as Command Center where possible.

7) Case Timeline Improvements

Case detail should show a timeline containing:
- case created
- status changes
- severity changes
- assignment changes
- notes
- escalations
- audit-linked events
- resolution/closure

8) Create Case from Command Center / Audit

Support case creation should work from:
- Support Tools directly
- Audit event detail
- Command Center Needs Attention item if applicable
