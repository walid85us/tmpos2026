Remaining corrections to implement

A) Inventory item edit and operational enforcement
1. Type editability
- in the inventory edit screen, item type must be maintainable where appropriate
- allow changing type in a controlled way if the current item state allows it cleanly
- if there are guardrails required for serialized vs non-serialized conversion, implement them clearly rather than silently blocking
- do not leave type effectively frozen without explanation

2. Hide-from-POS must actually be enforced
- if an inventory item is marked hide-from-POS, it must not appear in POS sellable item selection/search/add flows
- audit all relevant POS entry points and product pickers
- keep this coherent with existing invoice→POS and inventory flows

3. Repair part flag must actually be enforced
- if an inventory item is marked not a repair part, it must not appear in repair-parts selection/linkage flows
- audit all repair part pickers and ticket part-add flows
- do not let the flag exist only cosmetically

B) Purchase Order workflow refinement
1. PO details UX
- clicking View Details or a PO reference should open a proper popup/modal/drawer, not expand awkwardly inline under the list

2. Multi-item same-vendor PO creation
- when creating a PO, user must be able to add multiple inventory items from the same vendor into one PO cleanly

3. Vendor assignment on inventory items
- add a practical vendor assignment model to inventory items
- inventory item can have:
  - one or more vendor links if the current architecture supports it cleanly, or
  - at minimum a primary/default vendor
- there must also be a valid generic/no-vendor option

4. PO item picker behavior
- when a vendor is chosen in PO creation:
  - the item picker should prioritize/filter items linked to that vendor
- generic items may still be selectable if your final model supports them intentionally

5. Supplier maintenance
- supplier records must be editable after creation
- do not leave suppliers create-only

C) Transfer workflow refinement
1. Transfer details UX
- Transfer # must be clickable
- clicking it must open a transfer detail popup/modal/drawer

2. Transfer editability
- as long as the transfer is not received/finalized, it should remain editable where operationally appropriate

3. Confirmation flows
- any transfer action button should show a confirmation message/modal before committing the action

4. Receive reconciliation
- clicking Receive must open a reconciliation popup/modal
- user should be able to review the transfer and confirm what was actually received
- receiving should not be a blind one-click state change

5. Store locations master data
- when creating a transfer, From and To must come from a managed location/store dropdown
- make sure this location/store maintenance process exists and is ready for store owner or authorized users
- keep it coherent with current multi-store/store-scoped architecture

D) Inventory count / discrepancy handling refinement
1. Improve the count experience materially
- current count/adjustment section needs stronger operational clarity

2. Add discrepancy management visibility
- there must be a clear place to review:
  - counted items
  - remaining items
  - discrepancies
  - adjustment outcome
- do not leave discrepancy logic hidden or too shallow

3. Align with strong inventory-count workflows
- make the count flow feel like a real stock-count process, not just a small adjustment helper

E) Trade-in / refurbishment / RMA refinement
1. Trade-in improvements
- when creating a new trade-in, allow selecting walk-in customer
- allow taking/uploading photo of ID
- add resale price field that moves with the item when sent into stock
- Send to Refurb must open a confirmation window and include a note/comments field describing what needs refurbishment

2. Refurbishment improvements
- refurbishment items should be clickable to open a detail popup/modal with comments/details
- before completing refurbishment, allow re-evaluating resale price
- clicking Complete must open a confirmation popup before sending the item to stock
- completion/release-to-stock should be permission-aware

3. Add needed inventory sub-permissions for these areas
- add sub-permissions under Inventory for at minimum:
  - manage_trade_ins
  - manage_refurbishment
  - manage_stock_counts
  - manage_transfers (already exists, keep and refine as needed)
- wire them into actual UI gating, not just definitions

4. RMA refinement
- when creating an RMA and selecting a supplier:
  - only PO references from that supplier should be listed
  - items from that PO should populate automatically
  - each item line should support a reason field
- in RMA Management:
  - RMA # must be clickable and open RMA details in popup/modal/drawer
  - if RMA is still pending and not shipped, details should remain editable
  - Ship action must show confirmation
  - Refunded action must open a popup to confirm refund amount
  - that refund should be reflected back into store operational account/revenue logic in a coherent preview-safe way, without turning this into a full accounting subsystem
  - Replaced action must open a popup to confirm replacement items
  - once confirmed, there should be an option to send replacement items back into stock appropriately
