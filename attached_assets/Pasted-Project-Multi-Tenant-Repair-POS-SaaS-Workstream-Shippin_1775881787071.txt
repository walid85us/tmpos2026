--------------------------------------------------
WORKSTREAM GOAL
--------------------------------------------------

Create a proper Shipping Provider Configuration and Provider Selection capability so the system can operationally support multiple carriers/providers.

This workstream must add a clean, secure, native configuration area where authorized users can:
- choose the active shipping provider
- configure provider credentials/settings
- support EasyPost, Shippo, and ShipStation
- test or validate provider configuration safely
- make Shipping Center and provider actions use the selected provider coherently

This is the missing operational layer needed before the provider actions can be fully QA’d end to end.

--------------------------------------------------
MANDATORY ARCHITECTURE REQUIREMENTS
--------------------------------------------------

1) CENTRALIZED PROVIDER CONFIGURATION
Add a proper centralized configuration surface in the correct existing admin/system-owner/settings area of the app.

This must feel native to the current product.
Do not place provider setup in an arbitrary disconnected screen.

2) ACTIVE PROVIDER SELECTION
Support selecting one active shipping provider for system operation.
The system must clearly know which provider is currently active.

Required behaviors:
- active provider selection must persist correctly
- provider-dependent shipping actions must use the selected provider
- switching providers must not corrupt existing shipment data
- historical shipment/provider metadata must remain intact

3) PROVIDER-SPECIFIC CREDENTIAL CONFIGURATION
Support the appropriate configuration fields per provider, in a secure and scalable structure.

Examples of what may be needed depending on provider:
- API key
- API secret
- account/store identifier
- environment/mode if appropriate
- webhook-related identifiers only if your current architecture needs placeholders

Do not invent fake credentials.
Do not hardcode secrets into the codebase.

4) SAFE SECRET HANDLING
Use the safest pattern available within the current architecture.

Requirements:
- do not expose full secrets after save
- mask sensitive values in the UI after initial entry
- clearly distinguish between “configured” and “not configured”
- allow update/replacement of credentials without displaying full stored secret values
- do not leak sensitive values into logs, banners, or list rows

5) PROVIDER CONNECTION / VALIDATION PATH
Add a safe “Test Connection” / “Validate Provider Config” capability.

Required behaviors:
- runs against the currently entered/saved provider config
- shows clear success/failure messaging
- does not fake success
- gives operationally useful feedback when config is incomplete or invalid
- does not create shipments or labels as part of a connection test

6) SHIPPING CENTER INTEGRATION
Shipping Center provider operations must use the configured active provider.

Required behaviors:
- if no provider is configured, provider operations remain safely unavailable or error clearly
- if a provider is configured, provider actions use that provider path
- provider section in shipment detail should reflect the active/used provider coherently
- switching active provider should affect new provider operations, not corrupt historical shipment/provider references
