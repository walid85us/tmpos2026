PRIMARY GOAL
============

Transform Support Tools into a mature support operations workspace.

The System Owner / Support Admin should quickly understand:

* Which support cases require triage?
* Which cases are unassigned?
* Which cases are at risk of SLA breach?
* Which cases already breached response or resolution targets?
* Which cases are waiting on escalation acknowledgement?
* Which cases have pending de-escalation requests?
* Which queues are overloaded?
* Which macros are available and safe to use?
* Which macro was inserted and where?
* Which actions are draft/internal only?
* What is current queue health?
* What is real current app/session data vs future automation?

Use deterministic, transparent, rule-based support operations logic.

Do not implement real AI/ML.
Do not implement real external messaging.
Do not send emails.
Do not send notifications.
Do not pretend macros send to customers.
Macros should insert internal note/draft only unless an existing accepted behavior says otherwise.

==================================================
PART F — SLA BREACH / AT-RISK LOGIC
===================================

Use deterministic SLA logic.

Rules:

* calculate based on case createdAt / updatedAt / first response if available
* if first response timestamp unavailable, use internal note/customer response proxies only if currently meaningful
* be transparent about assumptions
* resolution SLA should remain active until case closed/resolved
* closed/resolved cases should show completed / not active
* escalated cases may increase priority/attention but should not fake SLA policy change unless specified

Add:

* SLA At Risk if remaining time below threshold, e.g. below 25%
* SLA Breached if target exceeded
* Missing timestamp = “insufficient data” not fake breach
