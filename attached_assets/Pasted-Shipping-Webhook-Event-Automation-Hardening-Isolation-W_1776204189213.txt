==================================================
MANDATORY CORE IMPLEMENTATION OBJECTIVES
==================================================

1) WEBHOOK HARDENING
Strengthen provider webhook ingestion and event handling.

Required:
- secure provider webhook entry points
- signature verification where provider supports it
- idempotency protection so the same event cannot corrupt shipment state twice
- event deduplication
- retry-safe processing
- event ordering awareness where possible
- durable event persistence with timestamps/source/provider metadata
- no destructive overwrite of shipment history

2) PROVIDER-DRIVEN LIFECYCLE AUTOMATION
After dispatch and provider progression, shipment lifecycle must update automatically and coherently from provider events.

Required:
- provider events can drive shipment state progression such as:
  - accepted / scanned
  - in transit
  - exception
  - delivered
  - returned-related future-ready handling
- provider-driven states must map safely into internal shipment lifecycle
- manual notes/events must not override authoritative provider progression
- shipment list, detail view, and tracking timeline must stay synchronized

3) EVENT REPLAY / RE-SYNC / RECOVERY TOOLING
Operational systems need recovery paths.

Required:
- add safe tooling for:
  - replay webhook event processing
  - re-sync tracking/events from provider
  - recover from failed or missed webhook ingestion
- this must be operator/admin-aware and permission-gated
- do not create dangerous uncontrolled replay behavior
- event recovery tools must be honest, logged, and auditable

4) EVENT AUDITABILITY
Required:
- every provider event must preserve:
  - provider source
  - provider event identifier if available
  - shipment linkage
  - received timestamp
  - processed timestamp if different
  - resulting mapped shipment state if applicable
  - processing result / success / ignored / duplicate / failed status
- timeline/history must remain understandable to operators

5) FAILURE HANDLING / OPERATIONAL TRUST
Required:
- failed webhook or sync events must not silently disappear
- surface operationally useful error/retry states where appropriate
- do not fake successful lifecycle progression
- do not leave shipment state half-updated

==================================================
NON-NEGOTIABLE IMPLEMENTATION REQUIREMENTS
==================================================

1. Unified provider event processing pipeline
Required:
- webhook events, sync events, replayed events, and manual recovery flows must all use a coherent processing path
- avoid fragmented one-off event logic
- centralize event-to-shipment-state mapping where practical

2. Idempotent shipment event processing
Required:
- duplicate provider events must not produce duplicate lifecycle transitions or corrupt shipment history
- processing the same event twice must be safe

3. Controlled operator/admin recovery tools
Required:
- add clear recovery actions where appropriate such as:
  - Re-sync Tracking
  - Replay Event
  - Refresh Provider State
- these must be permission-aware
- they must not be exposed casually to unauthorized roles

4. Clear distinction between event sources
Required:
- UI/timeline must continue to distinguish:
  - provider events
  - manual events
  - test-provider events
  - replay/recovered events if surfaced
- operators should understand what changed the shipment and why

5. Truthful test-mode vs live-mode behavior
Required:
- test mode must be explained honestly
- replay/simulated/test-provider events must not be confused with true live carrier events
- live mode must preserve clean operational visibility
