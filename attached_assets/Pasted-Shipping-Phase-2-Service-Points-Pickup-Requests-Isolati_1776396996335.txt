==================================================
WORKSTREAM GOAL
==================================================

Add a real Service Points and Pickup Requests capability that fits the current shipping architecture for both outbound shipping and reverse logistics where appropriate.

The result should let the business:
- select a carrier service point / drop-off point where supported
- view and store service-point information linked to the shipment
- request a carrier pickup where supported
- manage pickup request lifecycle/details coherently
- keep all of this aligned with provider mode, manual mode, shipping lifecycle, and reverse-logistics flows

This workstream must also explicitly prepare the architecture for the remaining Phase 2 items:
- customs docs
- insurance rules
- carrier analytics

Do not fully implement those remaining items now.
Do make this Service Points + Pickup Requests workstream architecturally compatible with them.

==================================================
PRODUCT DIRECTION — REQUIRED
==================================================

The app should now support two new operational layers inside shipping:

1) Service Points
A shipment can optionally be tied to a carrier/service point / drop-off location where provider/carrier capability supports it.

2) Pickup Requests
A shipment can optionally request carrier pickup where provider/carrier capability supports it.

These are shipping capabilities, not separate disconnected mini-apps.
They must feel native to Shipping Center and must also support reverse-logistics flows where relevant.

==================================================
MANDATORY IMPLEMENTATION OBJECTIVES
==================================================

1) SERVICE POINTS DOMAIN / MODEL
Add a robust service-point data model that can support:
- service point identifier
- provider/carrier linkage
- service point name
- address/location details
- hours / availability fields if available
- contact details if available
- shipment linkage
- selection timestamp / operator / notes
- forward-compatible metadata for future carrier analytics and international flows

This model must be forward-compatible with:
- customs docs
- insurance rules
- carrier analytics
- future automation rules
- future SLA optimization

2) PICKUP REQUEST DOMAIN / MODEL
Add a robust pickup-request model that can support:
- pickup request reference / confirmation number
- linked shipment
- provider/carrier linkage
- request status
- requested pickup date/time window
- pickup address
- contact details
- package count / handling notes if needed
- audit timestamps / operator / notes
- failure / rejection / cancellation metadata
- forward-compatible metadata for later analytics and automation

3) SERVICE POINT SELECTION EXPERIENCE
Implement a real operator workflow for service-point selection.

Minimum expectations:
- a clear place inside shipment workflow to select/manage a service point where supported
- visible selected service-point details on the shipment
- coherent behavior for outbound and return shipments where relevant
- operator understanding of when a shipment is using:
  - direct carrier handoff
  - service point drop-off
  - pickup request

4) PICKUP REQUEST EXPERIENCE
Implement a real operator workflow for pickup requests.

Minimum expectations:
- a clear place inside shipment workflow to request/manage pickup where supported
- visible pickup request details and status on the shipment
- coherent lifecycle-aware behavior
- operator understanding of whether pickup is:
  - not requested
  - requested
  - confirmed
  - failed/rejected/cancelled
  - completed where applicable

5) PROVIDER CAPABILITY / MODE AWARENESS
This workstream must not pretend every shipment supports every operational option.

Required:
- if provider/carrier/mode does not support service points, the UI must be honest
- if provider/carrier/mode does not support pickup requests, the UI must be honest
- manual shipments remain manual
- provider-managed shipments can use service-point / pickup workflows only where the provider flow supports them
- do not create fake provider capability

6) SHIPPING / RETURNS INTEGRATION
Required:
- service points and pickup requests must fit normal outbound shipping
- they must also be coherent for reverse-logistics / return shipments where relevant
- do not break the Returns Portal
- do not create conflicting rules between outbound and return shipment handling

7) LIFECYCLE / STATE COHERENCE
Required:
- service-point and pickup state must not create impossible shipment states
- shipment status, pickup status, and service-point selection must remain understandable and auditable
- if a shipment is already too advanced in lifecycle, inappropriate edits/actions should be blocked

8) AUDITABILITY
Required:
- service-point selection and pickup-request actions must be auditable
- operator must be able to see what was selected/requested, when, and by whom where practical
- keep this lightweight but real
