1) The condition operator dropdown is too technical and unclear to normal users.
Current values like:
- eq
- neq
- in
- notIn
- gt
- gte
- lt
- lte
- truthy
- falsy
are not acceptable as the user-facing final language.

2) The rule builder does not make it clear which operators are valid for which field types.
Example:
- "Has Label" is conceptually boolean
- showing generic operators and accepting a value like "6" creates confusion
- the user expected something like "label value >= 6" but chose "Has Label eq 6" because the builder did not guide them correctly

3) Delete confirmation currently uses a browser-level confirm dialog.
This is not acceptable.
Delete confirmation must be handled with an in-app modal/dialog consistent with the app UX.

4) Rule execution is not sufficiently visible or understandable.
The operator must be able to understand:
- what the rule means in plain language
- what fields/conditions it is using
- when it fired
- why it fired or did not fire
- what action it performed

This pass must make Automation Rules clear enough for a non-technical operator to use safely.

==================================================
PART A — HUMAN-READABLE, TYPE-AWARE RULE BUILDER
==================================================

Required:
Replace technical operator labels in the UI with clear operator language appropriate for the selected field type.

Do not expose raw developer/operator codes like:
- eq
- neq
- in
- notIn
- gt
- gte
- lt
- lte
- truthy
- falsy
as the user-facing final labels.

Required typed behavior:
1) Boolean fields should only show boolean-friendly operators, for example:
- is true
- is false

2) Numeric / currency fields should show numeric-friendly operators, for example:
- equals
- does not equal
- greater than
- greater than or equal to
- less than
- less than or equal to

3) Enum / status / provider / carrier fields should show list-friendly operators, for example:
- is
- is not
- is one of
- is not one of

4) Text fields should show text-friendly operators only where appropriate, for example:
- equals
- does not equal
- contains
- does not contain
- starts with
- ends with
Only if these are truly supported by the current rules foundation.

Required:
- the available operators must change based on the selected condition field
- the value input type must also change based on field type
  - toggle or boolean selector for boolean
  - numeric/currency input for numeric
  - dropdown or multi-select for enums where possible
  - text input only where appropriate

==================================================
PART B — REMOVE MISLEADING FIELD/OPERATOR COMBINATIONS
==================================================

Required:
The builder must prevent nonsensical combinations.

Example:
- if the selected field is "Has Label", the user should not be able to create something like "Has Label eq 6"
- that condition should instead be limited to something like:
  - Has Label is true
  - Has Label is false

If the business intent is "label value >= 6", then the builder must expose the correct underlying field explicitly, such as a label/shipping cost/value field if that data truly exists and is already supported.

Required:
- audit the current condition field list
- separate boolean fields from numeric fields clearly
- do not let a boolean field masquerade as a numeric one
- do not keep redundant or confusing field choices if they create operator ambiguity

==================================================
PART C — PLAIN-LANGUAGE RULE EXPLANATION
==================================================

Required:
Every rule should present a human-readable explanation/summary sentence.

Example pattern:
- "When Label Purchased occurs, and shipping cost is greater than or equal to 6, mark review needed."
or
- "When shipment status changes, and Has Label is true, queue shipment for batch review."

Required:
- show this summary during rule creation/editing
- update it live as the user changes the rule
- make it easy for the operator to understand what the rule will do before saving

==================================================
PART D — RULE EXECUTION VISIBILITY / TRANSPARENCY
==================================================

Required:
Rule execution must be visible enough for QA and operators to trust it.

At minimum support:
- visible execution history/log entry
- which shipment triggered the rule
- which rule triggered
- what trigger event occurred
- what conditions matched
- what action was executed
- timestamp

Required:
- if a rule does not run because conditions did not match, provide enough visibility where practical to understand behavior without turning this into a full debugging console
- prefer a clean operator-friendly execution log/history model
- avoid invisible automation firing

If there is already logging:
- make it more operator-readable, not just developer-readable
