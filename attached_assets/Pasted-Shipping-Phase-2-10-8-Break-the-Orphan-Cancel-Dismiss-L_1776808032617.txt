==================================================
CURRENT QA SOURCE OF TRUTH
==================================================

Observed runtime issue:
1) Operator clicks Cancel Orphan Pickup
2) Confirmation modal appears
3) Provider responds:
   "Invalid request, the pickup's current status does not allow this operation."
4) App instructs operator to use Dismiss Orphan Locally
5) After local dismiss, another buy-failed/orphan recovery area appears again
6) Clicking cancel again loops or does nothing

This proves:
- the UI still has no terminal local-resolution state for non-cancellable orphan pickups
- the same failed provider pickup can re-enter recovery after local dismiss
- the operator is trapped in a loop

==================================================
NON-NEGOTIABLE REQUIRED FIXES
==================================================

1) DEFINE A TRUE TERMINAL LOCAL-RESOLUTION STATE
Required:
- add a terminal local-only state for a failed orphan pickup that cannot be cancelled at the carrier
- examples of acceptable internal meaning:
  - orphan_dismissed_locally
  - provider_non_cancellable_locally_resolved
- once this state is set for a given failed providerPickupId, the active recovery panel for that orphan must not reappear again

2) DISMISS ORPHAN LOCALLY MUST BE ONE-WAY
Required:
- when provider cancellation is rejected because the pickup state is not cancellable:
  - offer Dismiss Orphan Locally
- when operator confirms local dismiss:
  - resolve the orphan locally
  - hide the active failed recovery UI for that orphan
  - do not show Edit / Retry / Cancel again for that same orphan
  - do not recreate another buy-failed/orphan block for the same providerPickupId

3) DO NOT OFFER CARRIER CANCEL AFTER PROVIDER SAYS IT IS NOT ALLOWED
Required:
- if provider explicitly says current pickup state does not allow cancellation, the app must stop presenting carrier cancel for that same orphan
- do not loop the operator back into a cancel action that will predictably fail again

4) KEEP AUDITABILITY WITHOUT KEEPING THE LOOP
Required:
- after local dismiss, keep an audit/history record if needed
- but present it as resolved history, not an active recovery problem
- acceptable examples:
  - a collapsed audit note
  - a timeline event
  - a resolved badge
- not an active error block with Edit / Retry / Cancel again

5) ADD CLEAR OPERATOR EXPLANATION
Required:
- explain in plain language:
  - carrier cancellation is no longer allowed for this provider pickup state
  - the app can dismiss the orphan locally so work can continue
  - local dismiss does not mean the carrier confirmed cancellation; it means the app has closed the unresolved recovery loop locally
- keep this truthful and concise

6) ADD A TEMPORARY PROOF MARKER FOR THIS PASS
Required:
- add a small visible marker such as:
  orphan-resolve: 2.10.8
- show it when the new terminal local-resolution path is active
- QA must be able to verify that the new path is live
