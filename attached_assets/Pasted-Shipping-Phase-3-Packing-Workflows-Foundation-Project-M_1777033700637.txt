==================================================
WORKSTREAM GOAL
==================================================

Add a solid Packing Workflows foundation inside Shipping Center.

The goal is to support a practical packing process before dispatch:
- verify shipment contents
- verify packages
- record packing status
- prevent accidental dispatch/label workflows when packing is incomplete where required
- support audit trail
- prepare for future SLA optimization and carrier scorecards

Do NOT build a full warehouse management system.
Do NOT fake barcode scanning if not truly implemented.
Do NOT break existing label, pickup, return, batch label, or automation flows.

==================================================
PART A — PACKING WORKFLOW SURFACE
==================================================

Required:
Add a Packing workflow surface inside Shipping Center.

It may be:
- a Packing tab,
- a section inside shipment detail,
- or a native workflow area that fits current IA.

Required:
- must feel native to current Shipping Center design
- must not create a disconnected mini-app
- must support both outbound and return shipments where appropriate

==================================================
PART B — PACKING STATE MODEL
==================================================

Required:
Introduce a clear packing state model.

At minimum support:
- not_started
- in_progress
- packed
- exception
- not_required if applicable

Required fields:
- packingStatus
- packedBy
- packedAt
- packingStartedBy
- packingStartedAt
- packingNotes
- packingExceptions
- package verification state
- item verification state where shipment items exist
- audit/history events

Required:
- packing state must be visible in shipment detail
- packing state should be visible in shipment list where relevant
- packing state must be auditable

==================================================
PART C — ITEM / CONTENT VERIFICATION
==================================================

Required:
Packing workflow should verify shipment contents when shipment items exist.

At minimum:
- show expected shipment items / source items where available
- allow operator to mark items as packed/verified
- support quantity verification where quantity exists
- show missing / mismatch / exception state
- do not invent source items where no source data exists
- if source contents are unavailable, show truthful “contents not available from source” or manual packing note state

Required:
- preserve link to invoice / repair / return / transfer source where available
- avoid disconnected duplicate data if source items already exist

==================================================
PART D — PACKAGE VERIFICATION
==================================================

Required:
Packing workflow must verify packages.

At minimum:
- show existing shipment packages
- confirm package count
- confirm package weight/dimensions where available
- allow package-level packing notes
- flag missing package information
- prevent marking as packed if package data is incomplete, unless authorized override or exception path exists

Important:
Do not break existing package management or label purchase prerequisites.

==================================================
PART E — PACKED STATUS / SHIPPING LIFECYCLE RELATIONSHIP
==================================================

Required:
Define how Packing Workflow relates to existing shipment status.

Current shipping lifecycle already uses statuses like:
- Draft
- Ready
- Label Created
- Packed
- Dispatched
- In Transit
- Delivered
- Cancelled

Required:
- packing workflow must align with existing Packed status
- marking packing complete should move or support movement to Packed only when appropriate
- do not collapse Label Created / Packed / Dispatched
- preserve the already-fixed truth that Dispatched is not In Transit
- do not allow packing workflow to silently skip required shipping lifecycle states

Acceptance logic:
- shipment should not be considered operationally packed until packing workflow is complete
- if shipment status is already Packed from prior flows, packing UI must reflect that coherently
- if packing is incomplete, show a clear not-packed state

==================================================
PART F — PACKING EXCEPTION WORKFLOW
==================================================

Required:
Support packing exceptions.

Examples:
- missing item
- quantity mismatch
- package weight missing
- package dimensions missing
- damaged item
- source item mismatch
- cannot verify contents

Required:
- exceptions must be visible
- exceptions must be auditable
- exception should block or warn before dispatch where appropriate
- authorized users should be able to resolve packing exceptions with note
- do not hide exceptions after resolution; keep history

PART H — INTEGRATION WITH AUTOMATION + BATCH LABELS
==================================================

Required:
Packing Workflows should integrate coherently with existing Phase 3 foundations.

Automation Rules:
- allow packing-related events or conditions only if supportable now
