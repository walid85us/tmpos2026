CURRENT QA TRUTH — TREAT THIS AS SOURCE OF TRUTH
--------------------------------------------------

QA and product review have identified these required improvements:

1) Shipping provider configuration still feels too detached from Shipping Center
- users expect provider configuration to live within or directly under Shipping Center
- governance/security can remain in settings, but operational access must feel native to Shipping

2) Provider connection errors need better operational handling in all provider-backed workflows
- avoid raw/infrastructure-style messaging
- keep honest, but clearer and more operationally useful

3) Get Rates should only be available once shipment prerequisites are met
- packages must exist
- destination address must be valid and confirmed/accepted through the provider-validation flow where applicable

4) Get Rates must return a clean, selectable list of carrier/service/cost options
- user should explicitly choose one
- selected rate must drive shipment carrier, service level, and shipping cost

5) Purchase Label should become available only when shipment is truly ready
- shipment should be in Ready / Prepared
- packages present
- destination address validated/accepted where applicable
- selected rate exists OR a fully valid manual carrier/service fallback exists if manual mode is intentionally supported

6) Purchased labels should open as PDF by default
- not PNG-first behavior for the primary operational label-opening path
- keep thermal-label readiness in mind

7) After label purchase:
- tracking number should auto-populate
- tracking number should display fully for operational users, not masked with XXXXXX
- tracking number should become locked
- carrier and service should become locked
- lock should remain unless cancellation/rework rules explicitly allow reset

8) Sync Tracking behavior needs fulfillment-grade treatment
- in provider test mode, test-tracking limitations should be handled clearly and honestly
- in production design, tracking history should update automatically after carrier scans/events
- Tracking & Events should be designed to ingest provider/backend events, not rely only on manual sync clicks

MANDATORY IMPLEMENTATION REQUIREMENTS
--------------------------------------------------

1) SHIPPING CENTER SETTINGS OWNERSHIP IMPROVEMENT
Implement a native Shipping Center settings subsection or clear provider-settings entry experience.

Required:
- provider configuration must feel like part of Shipping Center operations
- governed configuration ownership/security may still live in the proper settings/admin architecture
- do not duplicate config logic into two competing sources of truth
- Shipping Center should provide a clear operational Settings / Providers entry point/subsection

2) ADDRESS VALIDATION GATING BEFORE RATES
Strengthen shipment preparation discipline.

Required:
- Get Rates must not be enabled unless package prerequisites are met
- Get Rates should also require destination address readiness
- where provider validation is part of the flow, the address must be validated/accepted before live rate retrieval
- do not silently proceed with known-invalid destination data
- if the address is unvalidated or failed validation, show a clear operational message

3) RATE SHOPPING + EXPLICIT RATE SELECTION
Implement a stronger rate workflow.

Required:
- Get Rates returns a clean list of options
- each option clearly shows carrier, service, cost, and ETA/delivery context if available
- user explicitly selects one rate
- selecting a rate updates:
  - carrier
  - service level
  - shipping cost
- selected rate persists on the shipment record
- if the user clears the selected rate, manual carrier/service fallback may become editable again only if supported

4) PURCHASE LABEL PREREQUISITES + READY-STATE LOGIC
Purchase Label must behave like a real fulfillment operation.

Required:
- Purchase Label appears only when shipment is in Ready / Prepared and all required prerequisites are satisfied
- prerequisites include:
  - packages present
  - address validation readiness
  - selected rate or valid manual carrier/service fallback
- if the shipment is not ready, the action must not appear deceptively or must explain what is missing

5) PDF-FIRST LABEL OUTPUT
Improve label output behavior.

Required:
- the main label open/print path should prefer PDF output
- structure label handling so it remains thermal-label-ready
- do not lock the system into a PNG-only operational path
- preserve label artifact metadata cleanly

6) TRACKING NUMBER VISIBILITY + LOCK RULE
Required:
- after successful label purchase, tracking number auto-populates from provider response
- tracking number is shown in full to authorized operational users
- tracking number becomes locked after label purchase
- carrier/service also become locked after label purchase
- these structural shipping fields remain locked unless cancellation/rework rules explicitly allow reset

7) TEST MODE VS LIVE MODE TRACKING UX
Required:
- if provider test mode has tracking limitations, show clear, honest, user-friendly feedback
- do not present test-environment behavior as a production defect when it is actually expected
- if Sync Tracking is attempted in test mode with unsupported/non-test tracking codes, explain that clearly
- keep the UI operationally understandable

8) AUTOMATIC TRACKING / EVENT UPDATE FOUNDATIONS
Strengthen the system for real fulfillment use.

Required:
- Tracking & Events must be ready for automatic provider/backend-driven updates after shipment events/scans
- design/extend the architecture so provider events can update:
  - tracking timeline
  - latest tracking state
  - mapped shipment lifecycle status
- this should be structured for webhook/backend ingestion and not only manual sync
- preserve auditability and avoid destructive overwrites of event history

9) TWO-WAY SHIPPING STATE INTEGRATION FOUNDATIONS
Where appropriate for current scope, improve the foundations for two-way shipping updates across:
- shipment tracking state
- linked source records
- invoice / repair / transfer / RMA shipping visibility
- status coherence between operational surfaces

Do not overbuild full automation if it belongs to a later phase, but do build the architecture and workflow correctly now.
