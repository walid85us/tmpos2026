==================================================
WORKSTREAM GOAL
==================================================

Add a main-page Bulk Sync / Reconciliation capability inside Shipping Center so operators can reconcile multiple provider-mode shipments in one controlled workflow.

This tool must:
- operate from the main Shipping Center page
- sync only eligible shipments
- exclude manual shipments by default
- avoid blindly syncing everything
- batch and process safely
- show clear results per shipment
- use the existing provider event / tracking sync architecture
- remain a recovery/reconciliation tool, not a replacement for webhook automation

==================================================
NON-NEGOTIABLE PRODUCT DIRECTION
==================================================

The intended model is:

1. Primary source of truth
- provider webhook/event automation

2. Secondary per-shipment tool
- Sync Tracking on a single shipment

3. Recovery/admin operations tool
- Bulk Sync / Reconcile Shipments from the main Shipping Center page

This workstream must preserve that design hierarchy.

==================================================
MANDATORY IMPLEMENTATION REQUIREMENTS
==================================================

1) MAIN-PAGE BULK SYNC ENTRY
Add a clearly visible main-page bulk sync / reconciliation action inside Shipping Center.

Preferred UX:
- a top-level action such as:
  - Bulk Sync
  - Reconcile Shipments
  - Sync Eligible Shipments
- located in the main Shipping Center operational surface, not buried in settings

Required:
- the action must feel native to Shipping Center operations
- it must not be the default primary operator action for every shipment
- it should clearly read as a recovery/reconciliation tool

2) ELIGIBLE SHIPMENT SCOPING
Bulk sync must not act on every shipment blindly.

Required:
- by default, target only eligible provider-mode shipments
- exclude manual shipments by default
- exclude shipments with no tracking/provider context
- exclude terminal shipments by default unless explicitly included
- support sensible filtering such as:
  - in-flight only
  - provider mode only
  - shipments with tracking number
  - shipments with sync failures
  - shipments updated before a selected cutoff / stale shipments
- do not create dangerous broad sync behavior

3) USE EXISTING UNIFIED EVENT / SYNC PIPELINE
Do not create separate fragmented logic.

Required:
- bulk sync must reuse the existing shipment-level sync/event pipeline where practical
- provider event mapping, deduplication, and status progression logic must stay consistent
- bulk sync must not invent a second contradictory source of truth

4) SAFE BATCH PROCESSING
Required:
- process shipments in controlled batches
- avoid hammering providers
- avoid freezing the UI
- show progress during the batch run
- keep results understandable
- if batching limits are configurable in code, keep them sensible and documented

5) CLEAR RESULTS PER SHIPMENT
Operators must understand what happened.

Required:
- show a per-shipment result state such as:
  - updated
  - no new events
  - duplicate only
  - skipped
  - failed
- show aggregate summary counts such as:
  - total scanned
  - updated
  - unchanged
  - skipped
  - failed
- where appropriate, preserve links/opening path back to the shipment

6) PERMISSION / ROLE GATING
This must be controlled.

Required:
- bulk sync / reconciliation must be permission-gated
- only appropriate roles/users should be able to run it
- reuse existing permission architecture coherently
- do not create a disconnected permission system

7) PREVIEW MODE SAFETY
Required:
- preview mode must not commit bulk sync changes
- bulk sync action should be blocked or clearly read-only in preview mode
- do not allow accidental multi-shipment mutation in preview mode

8) MANUAL SHIPMENTS MUST REMAIN MANUAL
Required:
- manual shipments must not incorrectly enter provider automation through bulk sync
- bulk sync must clearly ignore manual shipments unless there is an explicitly justified future path
- preserve manual mode isolation completely

9) TEST MODE VS LIVE MODE CLARITY
Required:
- if bulk sync runs in provider test mode, messages must remain clear and non-misleading
- expected test-mode limitations must not flood the operator with noisy failure output
- distinguish:
  - updated via provider/test-provider event
  - no changes
  - known test-mode limitation
  - real failure

10) AUDITABILITY
Required:
- bulk sync results should remain auditable enough for operators/admins
- if shipments were updated, operators should be able to tell what changed
- reuse the event log / sync result architecture where practical
- do not create black-box bulk behavior
