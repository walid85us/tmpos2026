Remaining corrections to implement

A) Purchase Order workflow refinement
1. PO details UX
- clicking View Details or a PO reference must open a proper popup/modal/drawer
- do not expand awkwardly inline under the list

2. PO editability before send
- if a PO is still Draft / not sent yet, it must be editable
- user must be able to edit item details while still in draft
- once the PO moves beyond the editable stage, apply clear lifecycle guardrails

3. Vendor change confirmation/reset logic
- when creating/editing a PO, if the user already added items and then changes the supplier/vendor:
  - show a warning popup
  - explain clearly that existing PO items will be removed/reset because they do not belong to the new vendor context
  - require confirmation before resetting the PO item selection

4. Multi-item same-vendor PO creation
- keep or improve support for multiple items from the same vendor in one PO

B) Transfer workflow refinement
1. Transfer details UX
- Transfer # must be clickable
- clicking it must open a proper transfer detail popup/modal/drawer
- do not render details awkwardly inline under the Inventory Transfers list

2. Transfer editability
- as long as the transfer is not received/finalized, it should remain editable where operationally appropriate

3. Confirmation flows
- lifecycle actions must continue to use confirmation prompts

4. From / To location dropdown logic
- when creating or editing a transfer:
  - From and To must come from managed location/store dropdowns
  - the same location must not be selectable in both fields at the same time
  - if one location is selected in From, it must not appear in To, and vice versa

5. Transfer receive must become a true reconciliation workflow
Do NOT treat receiving as a simple confirm button.
The Receive action must open a proper reconciliation popup/modal with:

Header:
- Transfer #
- From location
- To location
- Sent date
- Requested by / sent by
- current transfer status
- carrier/tracking fields only if they already exist; otherwise keep architecture ready for future shipping integration
- supplier/vendor context should be shown at item level where relevant, not as the main identity of the transfer

Line-item reconciliation table:
- Item
- SKU
- Vendor/Supplier if available
- Serialized / non-serialized
- Expected Qty
- Received Qty
- Variance
- Condition / Disposition
- Notes / discrepancy reason

Required reconciliation rules:
- Variance = Received Qty - Expected Qty
- zero variance = clean row
- non-zero variance = flagged row with visible discrepancy styling

Required transfer states after reconciliation:
- Received
- Partially Received
- Discrepancy Detected

State rules:
- no variance on any row → can finalize as Received
- some mismatch → must finalize into Partially Received or Discrepancy Detected
- unresolved differences must remain visible in transfer details/history

Inventory rules:
- destination stock must increase only by actual received quantities
- shortages must not be silently added
- extra received quantities must require explicit confirmation
- reconciliation event must be recorded in transfer history

Serialized item rules:
- if serialized behavior exists for the transferred item, allow explicit received-serial confirmation where cleanly supported
- missing/unexpected serials should create visible discrepancy behavior rather than being hidden

Transfer list/detail visibility:
- transfers with mismatches must show a clear discrepancy badge/indicator in the list and in detail
- include a short discrepancy summary such as mismatch count or shortage/overage visibility
