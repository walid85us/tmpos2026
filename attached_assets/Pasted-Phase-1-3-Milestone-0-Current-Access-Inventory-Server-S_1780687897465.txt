INVENTORY REQUIREMENTS
======================

Create a clear Phase 1.3 Milestone 0 inventory that includes:

1. Current Platform Permission Architecture

Document:

* platformPermissionsConfig.ts as the platform permission source.
* accessConfig.ts as the separate tenant/store permission source.
* The ranked platform levels:
  none / view / create / edit / approve / manage / full
* The existence of platformPermissionMeets.
* The fact that platform permissions and tenant/store permissions are separate systems.

2. Current Platform Feature Groups

Inventory all current platform feature groups found in the codebase.

For each feature group, document:

* key/id
* readable name if available
* current purpose
* whether it contains sensitive sub-permissions
* current UI/client-gated status
* future server enforcement recommendation

3. Current Platform Sub-Permissions

Inventory all platform sub-permissions.

For each sub-permission, document:

* sub-permission key/id
* parent feature group
* threshold level
* sensitive flag
* likely action category
* current enforcement state: UI/client-gated only
* future enforcement tier:

  * Tier 0 — UI-only acceptable for now
  * Tier 1 — future server validation recommended
  * Tier 2 — future server validation strongly required before production
  * Tier 3 — future privileged/PIM-controlled server action

4. Current Platform Roles

Document the current platform roles found in the codebase.

Include:

* system_owner
* support_admin
* billing_admin
* operations_admin
* security_admin

For each role, document:

* role purpose
* current access posture at a high level
* whether it should be eligible for future temporary elevation
* whether it should be included in future access review
* whether it should remain protected from accidental downgrade, especially System Owner

Do not change role behavior.

5. Current Resolver Model

Document the current resolver conceptually.

Include:

* system-owner allow
* explicit sub-permission override
* explicit feature override
* role default
* transitive prerequisite check
* denied_prerequisite behavior
* dependency auto-sync as write-time, direction-aware, least-privilege-preserving

Do not alter the resolver.

6. Current UI-Only / Client-Side Gating Finding

Document clearly:

* Current platform access is UI/client-gated.
* Server-side platform RBAC enforcement is not implemented yet.
* server/index.ts currently does not enforce platform RBAC boundaries.
* Any future backend enforcement must not fork permission logic away from the resolver model.

Use truthful wording such as:

“Current enforcement state: UI/client-gated only. Future server-side enforcement is recommended before production-grade privileged operations.”

7. Future Server-Side Enforcement Classification

Create a future enforcement classification table.

Classify these categories at minimum:

* platform RBAC-mutating writes
* platform role changes
* sub-permission overrides
* temporary access grants
* access review records
* privileged audit actions
* tenant management mutations
* billing/subscription mutations
* add-on governance mutations
* platform settings mutations
* provisioning lifecycle changes
* support/admin intervention actions
* sensitive audit/security actions

For each category, document:

* current state
* risk
* future enforcement need
* recommended enforcement tier
* deferred implementation phase

8. Non-Regression Guardrails

Document that future Phase 1.3 milestones must protect:

* Tenant Web Address
* Platform Settings Control Center
* Command Center Intelligence
* Audit Investigation Center
* Support Queue / SLA / Macro
* Escalation operating model
* Permission dependency auto-sync
* Add-on Governance
* Shipping
* Store Permissions Matrix
* Tenant provisioning
* paid override invoice workflow
* server PII logging

9. Milestone 0 Acceptance Criteria

Document acceptance criteria:

* all current feature groups inventoried

* sensitive flags captured

* current UI-only enforcement truthfully stated
* future server-side enforcement tiers classified
