PART A — GENERAL RULE: PLAN-DISABLED FEATURE PERMISSIONS MUST DISAPPEAR
==================================================

Required:
Implement this as a general rule, not a one-off hack.

New required rule:
- when a feature is disabled for the current tenant plan
- and a permission/sub-permission is specifically linked to that feature
- that permission row must NOT render in the Store Permissions Matrix
- do NOT render it in a "Plan locked" state
- do NOT keep it visible as disabled chrome
- remove it from the matrix view entirely

This rule must be architecture-driven and reusable for future features.

Required design expectation:
- define a clean feature-to-permission dependency mapping
- use that mapping to decide which rows/sub-permissions render in the Store Permissions Matrix
- rendering should be based on plan-available capabilities first
- only after that should role-level assignability be evaluated

Business rule:
- plan determines whether the capability exists at all
- role determines who can use/configure it
- role must never be able to expose or revive a plan-disabled feature

Important:
Use disappearance for linked feature permissions.
Do NOT keep using the current "Plan locked" pattern for this specific behavior.
