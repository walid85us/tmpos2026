==================================================
CURRENT QA SOURCE OF TRUTH — REMAINING FAILURES
==================================================

These are the exact remaining blockers and must be fixed literally:

1) Plan/feature gating is not enforced correctly
Observed QA result:
- Service Point / Drop-Off or Carrier Pickup Request can still appear/useable even when disabled in System Owner Plans & Features
Required behavior:
- if disabled at plan level, the capability must not be useable for that tenant/plan
- UI should hide/disable it coherently and not allow editing/selection

2) Store permission denial is not enforced correctly
Observed QA result:
- when store permissions are denied for Service Point / Drop-Off or Carrier Pickup Request, the user can still edit/select them
Required behavior:
- if denied at store-role/permission level, the capability must not be editable/useable for that user
- UI and action handlers must both enforce the denial

3) Service Point / Drop-Off and Carrier Pickup Request are not mutually exclusive
Required behavior:
- the shipment cannot use both at the same time
- if Service Point / Drop-Off is selected, Carrier Pickup Request must be disabled/cleared
- if Carrier Pickup Request is selected/requested, Service Point / Drop-Off must be disabled/cleared
- operator must clearly understand that these are mutually exclusive shipment handoff methods

4) Shipment Overview should show which handoff method is being used
Required behavior:
- Shipment Overview must clearly show whether the shipment is:
  - Service Point / Drop-Off
  - Carrier Pickup Request
  - neither
- this should be easy for operators to see without drilling deep into separate sections

5) Pickup cancellation / re-schedule behavior needs refinement
Required behavior:
- if Carrier Pickup Request is cancelled, show pickup cancellation confirmation/status clearly
- after cancellation, operator must be able to re-schedule a new Carrier Pickup Request
- keep lifecycle coherent and auditable

6) Provider-unavailable messaging should be consistent
Observed QA result:
- when no provider is configured, Provider & Operations validation area messaging is not aligned with Service Point / Drop-Off and Carrier Pickup Request messaging
Required behavior:
- use consistent honest messaging such as:
  “Service points not available for this shipment. No active shipping provider configured.”
  and the equivalent for pickup
- Provider & Operations should align with the rest of the shipment UX

7) Service Point search can be better
Operator improvement requested:
- add search by ZIP code for Service Point / Drop-Off where the current implementation supports it
Required behavior:
- keep this lightweight and useful
- do not redesign the full service-point UX
- if ZIP-based search is feasible in the current provider flow, implement it
- if provider capability is limited, keep the UI honest

8) Pickup tracking/history visibility
Requested behavior:
- show pickup-related tracking/event/history where carrier/provider updates are available
Required behavior for this pass:
- surface pickup status/history coherently where supported by current provider flow
- do not fake live provider pickup events
- keep the UI honest in test mode vs production mode
- at minimum, make pickup request status/history clearer and future-ready
