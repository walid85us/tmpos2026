Remaining corrections to implement

A) Transfer workflow refinement
1. Transfer editability before received
- current QA shows transfers are still not properly editable before received
- fix this so that as long as a transfer is not finalized/received, the appropriate fields and line items remain editable
- preserve lifecycle guardrails, but do not leave transfers effectively locked too early

2. Mismatch resolution after reconciliation
- current discrepancy behavior is visible, but there must be a real path to resolve mismatch and keep the transfer open until it is properly closed
- add a clear mismatch-resolution flow so authorized users can:
  - review discrepancy lines
  - adjust actual received quantities and notes
  - keep transfer open in discrepancy-related status
  - resolve/close it later when reconciliation is complete
- do not force a false final state too early
- preserve actual-received inventory logic

B) Inventory count closure refinement
1. Completed-count adjustment closure
- current QA shows adjustment is not supported once count status becomes Completed
- improve the lifecycle so discrepancy resolution and adjustment closure remain practical and operational
- user must be able to:
  - review discrepancy results
  - confirm/complete adjustments
  - understand what was adjusted and what remains unresolved
- do not leave Completed as a dead-end with no usable adjustment closure

C) Trade-in refinement
1. Customer field UX
- remove the mixed dropdown/search behavior
- use only one customer search bar/searchable field by name or phone number
- do not keep redundant dropdown behavior

2. Walk-in logic
- when Walk-in Customer is selected, the customer search field must be disabled
- when Walk-in Customer is deselected, the customer search field must be enabled again

3. ID capture/upload UX
- keep device capture/upload support
- replace the URL-style field with a proper dedicated Upload button
- add a visible preview area for the captured/uploaded ID image
- keep it practical for mobile camera capture and desktop file upload

D) RMA refinement — operational and finance-correct
1. RMA details modal/editing
- current QA shows RMA details/editing still do not behave correctly enough
- ensure RMA # opens in a proper modal/drawer
- if status is still Pending / not shipped, it must remain editable in a practical way

2. Replacement quantity controls
- current replace flow allows arbitrary replacement quantity; this is not acceptable
- implement strict per-line resolution logic

For each RMA line track and enforce:
- requestedQty
- shippedQty
- refundedQty
- replacedQty
- openQty = shippedQty - refundedQty - replacedQty

Rules:
- replacement quantity cannot exceed openQty
- refund quantity cannot exceed openQty
- once openQty reaches zero, that line is fully resolved
- the overall RMA can progress through meaningful resolution states such as:
  - Pending
  - Shipped
  - Partially Resolved
  - Closed

3. Refund confirmation logic
- refunded should represent confirmation only once supplier refund is actually confirmed to the store
- refund modal must capture:
  - refund amount
  - refund subtotal if relevant
  - tax adjustment amount
  - total refund
  - refund method such as vendor_credit / cash / bank_refund / purchase_offset
  - notes
- do not label this as sales revenue

4. Finance/reporting model for supplier refund
- supplier refund must be treated as purchase-side recovery, not customer sales revenue
- implement a lightweight operational finance record for supplier refunds / vendor credits without building a full accounting subsystem
- add a practical finance object or equivalent tracking structure linked to RMA and supplier, such as:
  - SupplierRefundEntry / VendorCreditEntry
  - linked to supplier, RMA, and PO where relevant
- this should support operational reporting on:
  - supplier refunds / vendor credits
  - net purchase recovery
  - tax adjustment from supplier-side returns
- keep this in supply-chain / procurement logic, not sales revenue logic

5. Replacement stock behavior
- replacement confirmation must add quantity back into inventory appropriately
- do not allow replacement to be cosmetic only
- quantity added back must match the confirmed replacement quantity and remain within openQty

6. RMA detail UX sections
Refine RMA details so it can clearly show:
- overview
- items
- resolution
- finance
- history
This does not need to be a huge redesign, but the information architecture should support operational clarity.
