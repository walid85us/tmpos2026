PART A — ESCALATION DEFINITIONS IN PRODUCT
==================================================

The UI and documentation must clarify these concepts.

1) Assignment
Meaning:
A support case has an owner/operator or team responsible for handling it.

Assignment does not necessarily mean escalation.

2) Escalation
Meaning:
A support case requires higher attention, a defined owner/team, and acknowledgement.

Escalation is a higher operational state on the case, not a separate incident record.

3) Acknowledgement
Meaning:
The assigned escalation owner/team has seen and accepted responsibility for the escalation.

Acknowledgement does not mean resolved.

4) Escalation Resolution
Meaning:
The elevated/escalated concern has been resolved or returned to normal handling.

Resolving escalation does not automatically close the support case.

5) Case Closure
Meaning:
The support case itself is resolved/closed.

Case closure is separate from escalation resolution.

6) Incident
Meaning:
A future coordination artifact for platform-wide or multi-tenant impact.

Do not implement incidents in this phase.
Mention future only if helpful.

Required:
Show a concise explanation in the Support Tools escalation area, such as:
“Escalation identifies ownership and acknowledgement for cases needing elevated attention. Resolving escalation does not automatically close the support case.”

PART B — ADVISORY PLATFORM SUPPORT ROLE MODEL
==================================================

Implement a lightweight advisory role model for Platform Operations support actions.

This is not full production auth/RBAC.
This is a client-side/advisory governance foundation.

Roles:
- platform_owner
- platform_admin
- platform_lead
- platform_operator
- platform_security
- platform_billing
- platform_readonly

Support-focused role aliases if useful:
- support_admin
- support_lead
- support_operator
- support_viewer

Use existing platform team/member data if available.
If not fully connected to auth, make this truthful:
“Role checks are advisory governance controls in this phase. Server-enforced Platform RBAC is planned for Platform Team Governance.”

Required permission logic:

platform_owner:
- all actions

platform_admin:
- all support escalation actions
- configure future policy placeholders if present

platform_lead:
- escalate any case
- assign escalation owner/team
- acknowledge escalation
- change level/owner/team
- de-escalate
- resolve escalation
- close case
- export audit if already allowed

platform_operator:
- escalate assigned/owned case
- acknowledge escalation if assigned to them or their team
- add notes
- resolve escalation if assigned/owned, if policy allows
- request closure but not close if not lead/admin

platform_security:
- view security/audit escalation context
- escalate security-related cases
- acknowledge security escalation if assigned to security team
- view restricted audit context if already supported
- should not automatically close non-security cases unless also lead/admin/owner

platform_billing:
- escalate billing-related cases
- acknowledge billing escalation if assigned to billing team
- should not manage unrelated security/platform cases unless also lead/admin/owner

platform_readonly:
- view only
- no mutation actions

Required:
- add a small “Current operator role” selector or internal dev/governance role context if the app lacks real auth role switching
- label it clearly as governance preview/advisory if not enforced by server
- action buttons should be enabled/disabled based on advisory role
- disabled actions should explain why

Do not confuse this with tenant employee roles or Store Permissions Matrix.

PART D — ESCALATION LIFECYCLE ACTIONS
==================================================

Implement internal lifecycle actions.

1) Escalate

Required form fields:
- reason code
- reason note
- escalation level
- target team
- owner/person if available or “Unassigned within team”
- acknowledgement due time if practical, otherwise a deterministic default
- optional related audit event/case context if already available

Rules:
- must require reason code and note
- must require target team or owner
- sets escalationStatus = escalated
- sets escalatedAt / escalatedBy
- records timeline entry
- emits audit event

2) Assign / Reassign Escalation

Required:
- change owner
- change team
- reason required
- previous/new owner/team visible
- timeline entry
- audit event

3) Acknowledge Escalation

Required:
- action visible to authorized role/assigned owner/team
- acknowledgement note optional or required depending on role; choose sensible default
- sets escalationStatus = acknowledged or in_review
- sets acknowledgedAt / acknowledgedBy
- timeline entry
- audit event

4) Change Level

Required:
- change escalation level
- reason required
- previous/new level visible
- timeline entry
- audit event

5) De-escalate

Required:
- reason required
- status becomes deescalated
- active escalation removed from active queue
- case remains open unless separately closed
- timeline entry
- audit event

6) Resolve Escalation

Required:
- resolution note required
- status becomes resolved
- active escalation removed from active queue
- case itself may remain open
- timeline entry
- audit event

7) Close Case

Do not change existing close case behavior unless needed to clarify:
- closing support case is separate from resolving escalation
- if closing an active escalated case, show in-app warning:
  “This case has an active escalation. Resolve or de-escalate escalation first, or close with escalation resolution note.”
Use a safe approach.
