==================================================
BUSINESS PROBLEM TO SOLVE
==================================================

The System Owner can create an add-on, but creating an add-on does not automatically create app functionality.

This creates a risk:
- System Owner may create an add-on that looks commercially available
- tenant may see Trial / Paid Override
- but there may be no real UI, data model, permissions, or runtime feature connected to it

We need the app to clearly distinguish:

1) Runtime-backed add-on
The add-on is linked to real implemented app functionality.

2) Commercial placeholder
The add-on exists in the catalog and matrix but does not yet unlock implemented functionality.

3) Implementation required
The add-on needs a development workstream before it can be safely sold/granted.

4) Parent-feature linked
The add-on is linked to a broad parent module and may unlock too much.

5) Runtime partially backed
The add-on is linked to a capability that exists, but not all runtime checks, permissions, or UI surfaces are fully connected.

PART A — ADD-ON IMPLEMENTATION READINESS MODEL
==================================================

Add a new readiness model for each add-on.

Required readiness statuses:

1) runtime_backed
Label:
- Runtime-backed

Meaning:
- linked to implemented functionality
- safe to offer as Trial / Paid Override if compatible and active

2) implementation_required
Label:
- Implementation Required

Meaning:
- commercial add-on exists, but real app functionality is not implemented or not connected
- should not be tenant-grantable by default

3) commercial_placeholder
Label:
- Commercial Placeholder

Meaning:
- intentionally listed for planning, roadmap, or future sale
- not tenant-grantable unless System Owner explicitly enables manual/presale handling

4) parent_feature_linked
Label:
- Parent Feature Linked

Meaning:
- linked to a broad parent feature/module
- may unlock too much
- needs warning and explicit confirmation

5) partially_backed
Label:
- Partially Runtime-backed

Meaning:
- some runtime capability exists, but missing one or more:
  - permission dependency
  - UI surface
  - plan gate
  - tenant feature mapping
  - audit behavior
  - documentation

Required:
- Store readiness status on Add-on Catalog item or derive it consistently from metadata
- Show readiness badge in Add-on Catalog
- Show readiness detail in Add-on Create/Edit modal
- Show readiness warning in Tenant Features only if needed, without cluttering the tenant feature cards

==================================================
PART B — RUNTIME BACKING CHECKLIST
==================================================

Each add-on should have an Implementation Readiness checklist.

Required fields/checks:

1) Capability key exists
- linkedFeatureId or generated cap_<addonId> exists

2) Plans & Features Matrix row exists
- no duplicate row

3) Runtime UI surface exists
- route/tab/component/surface identified where the feature appears

4) Entitlement check exists
- app checks the feature/capability before showing or allowing use

5) Permission dependency exists where needed
- Store Permissions Matrix has linked permission/sub-permission if the feature needs role-level control

6) Data model exists where needed
- Firestore/local model or state model is defined if feature needs data

7) Audit behavior exists where needed
- important tenant/admin actions are auditable if feature changes business state

8) Billing behavior exists if paid
- paid override invoice flow can link to this add-on

9) Documentation exists
- replit.md has implementation notes

Required:
Show each checklist item as:
- Complete
- Missing
- Not Required
- Unknown

Do not fake completion.
If uncertain, mark Unknown.

==================================================
PART D — GRANT SAFETY RULES
==================================================

Tenant Trial / Paid Override availability must consider readiness.

Required rules:

1) Runtime-backed
- Trial / Paid Override can be offered if:
  - Governance Status = Active
  - tenant plan is compatible or feature is included by plan logic applies
  - billing/permission rules pass

2) Partially Runtime-backed
- Trial / Paid Override can be offered only with warning:
  “This add-on may not be fully connected to runtime behavior.”
- recommended to require System Owner confirmation

3) Parent Feature Linked
- Trial / Paid Override can be offered only with warning:
  “This add-on may unlock a broad module. Consider linking a specific sub-feature.”

4) Implementation Required
- Trial / Paid Override should be disabled by default
- show:
  “Implementation required before tenant grant.”
- provide Generate Implementation Brief action

5) Commercial Placeholder
- Trial / Paid Override disabled by default
- optionally allow System Owner to mark:
  “Allow manual/presale grant”
- if allowed, show strong warning and audit it

Required:
No tenant should get access to a non-runtime-backed add-on by accident.
