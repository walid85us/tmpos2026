MANDATORY ARCHITECTURE REQUIREMENTS
--------------------------------------------------

1) PROVIDER-AGNOSTIC SHIPPING ADAPTER LAYER
Create a clean provider abstraction layer that supports future carriers/providers without reworking the shipping module.

The adapter layer must clearly separate:
- address validation
- rate retrieval
- label purchase
- tracking retrieval / tracking event sync
- carrier-specific service normalization
- provider errors and retry-safe handling

Do not bake provider-specific assumptions into the core shipment domain model.

2) FIRST PROVIDER INTEGRATION PATH
Implement the first provider integration path in a production-minded way.
Preferred first provider direction: EasyPost-style or Shippo-style adapter architecture.

Required:
- use secure configuration patterns
- no hardcoded API keys
- no fake “success” when provider calls fail
- if credentials are not available in the environment, keep the integration path real and code-complete, with safe fallback behavior and explicit UX messaging rather than pretending live booking succeeded

3) ADDRESS VALIDATION FLOW
Add address validation capability into shipment preparation where it fits operationally.

Required behaviors:
- validate destination address before final booking/purchase where appropriate
- store validation result / status in shipment context
- clearly indicate validated, corrected, or failed validation state
- do not silently overwrite user data without visibility
- allow controlled acceptance of provider-recommended address normalization where appropriate

4) RATE SHOPPING / SERVICE SELECTION
Add rate retrieval and service selection capability.

Required behaviors:
- retrieve available shipping rate/service options through the provider adapter where configured
- normalize returned services into a consistent UI/data model
- show service name, estimated delivery context if available, cost, and carrier/provider source
- allow the user to choose a service for shipment booking
- keep manual fallback behavior coherent if live provider access is unavailable

5) LABEL PURCHASE / LABEL ARTIFACT HANDLING
Add a shipment label workflow that fits the lifecycle properly.

Required behaviors:
- create/purchase label through the provider adapter where configured
- store label artifact metadata and retrieval path
- store provider shipment/buy/label identifiers safely
- transition shipment into the correct status when label creation succeeds
- do not allow label state inconsistencies
- do not mark label-created unless the label step actually succeeded

6) PRINT-LABEL READINESS
Make the app ready for real label printing.

Required behaviors:
- label artifact handling must support common shipping label formats, especially PDF and thermal-label-friendly usage paths
- provide a clear Print Label / Open Label / Download Label style operational action using the app’s current patterns
- do not overbuild print management, but do ensure label artifacts are usable for shipping operations

7) TRACKING ENRICHMENT + AUTOMATIC STATUS UPDATES
This is critical.

Shipping status should not remain only manual in the future architecture.
In this phase, implement the backend/domain structure and operational flow for provider-driven tracking updates.

Required behaviors:
- shipment keeps provider tracking identifiers and tracking number
- provider tracking events can be ingested / synced into the shipment tracking timeline
- latest carrier/provider tracking state maps into the platform shipment lifecycle carefully
- shipment status updates should be capable of automatic progression from provider/backend updates rather than relying only on manual changes
- maintain auditability and avoid destructive overwrites of historical events

Use disciplined state mapping.
Do not let noisy carrier events create broken lifecycle transitions.
