==================================================
PART C — SLA TARGET DEFINITIONS
==================================================

Required:
Support practical SLA target definitions grounded in current shipping workflow.

At minimum support targets such as:
1) Pack By
- based on shipment creation or readiness timestamp
- compares against packing completion timestamp

2) Label By
- based on packing completion / Packed timestamp or shipment readiness timestamp
- compares against label purchase timestamp

3) Dispatch By
- based on label purchase / Label Created timestamp
- compares against Dispatched timestamp

4) Deliver By
- based on Dispatched or In Transit timestamp where available
- compares against Delivered timestamp

5) Return Receive By, where relevant
- based on return shipment label/dispatch/delivery context if current return flow supports it

Important:
- only calculate actual performance where the required timestamps exist
- show unknown or insufficient data when timestamps are missing
- do not fake carrier event timing

==================================================
PART D — SLA RULE / POLICY CONFIGURATION
==================================================

Required:
Add a foundation for SLA policy configuration.

At minimum:
- default SLA windows by target type, such as:
  - pack within X hours
  - label within X hours after packed
  - dispatch within X hours after label purchase
  - deliver within X days after dispatch
- optional variation by:
  - shipment type / source type
  - outbound vs return
  - carrier/service if currently available
  - store/location if architecture supports it

Required:
- configuration should be simple and practical
- avoid overbuilding a full optimization engine
- defaults should be reasonable and editable by authorized users
- changes should be auditable where appropriate

==================================================
PART E — SLA STATUS CALCULATION
==================================================

Required:
Implement deterministic SLA status calculation.

Behavior:
- on_track when current time is before threshold and no risk condition exists
- at_risk when approaching deadline
- overdue when deadline passed and actual completion has not happened
- met when actual completion occurred on/before deadline
- missed when actual completion occurred after deadline
- paused when SLA is intentionally paused with reason
- unknown when required data is missing

Required:
- do not calculate misleading SLA status if source timestamp is missing
- expose concise explanation:
  - “Pack by 3:00 PM today”
  - “Label overdue by 2h”
  - “Dispatch met 35m before deadline”
  - “Delivery SLA unknown — no dispatch timestamp”
- support refresh/recalculation without corrupting history
