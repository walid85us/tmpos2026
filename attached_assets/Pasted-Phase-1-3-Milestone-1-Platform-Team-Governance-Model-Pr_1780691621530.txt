MODEL REQUIREMENTS
==================

Create a governance model that is additive and future-safe.

The model should support future milestones without changing current behavior.

1. Current Role Catalog

Define metadata for the current platform roles:

* system_owner
* support_admin
* billing_admin
* operations_admin
* security_admin

For each role, define:

* role id
* display label
* plain-English purpose
* governance category
* whether it is system-protected
* whether it should be included in access review
* whether it may be eligible for future temporary elevation
* recommended review cadence label
* risk posture label
* notes / warnings

Important:
System Owner must remain protected and must not become downgradeable through this milestone.

Do not introduce active behavior for new roles.

You may document future role concepts such as Read-only Auditor or Platform Admin only as future/deferred concepts, not active implemented roles.

2. Governance Categories

Define clear governance categories such as:

* system_owner
* platform_admin_ops
* support_operations
* billing_operations
* security_audit
* future_read_only_audit
* future_temporary_elevation

Use these as labels only.

They must not alter access behavior.

3. Privileged / Sensitive Action Categories

Define a typed catalog of privileged action categories based on Milestone 0, such as:

* platform_role_change
* platform_sub_permission_override
* platform_feature_level_override
* temporary_access_grant
* temporary_access_expiry
* access_review_completion
* tenant_lifecycle_change
* billing_subscription_change
* addon_governance_change
* platform_settings_change
* provisioning_lifecycle_change
* support_admin_intervention
* audit_security_sensitive_action
* paid_override_action

For each category, define:

* display label
* description
* recommended future enforcement tier
* reason required recommendation
* audit recommended
* PIM recommended
* future server validation need
* implementation status: future/deferred unless already real today

Do not wire these to live behavior yet.

4. Temporary Access Types

Define types only for future use.

Suggested shape:

TemporaryAccessGrant:

* id
* subjectUserId
* subjectName
* subjectEmail
* baseRoleId
* elevatedRoleId or elevatedPermissionScope
* requestedBy
* approvedBy
* reason
* startsAt
* expiresAt
* status
* createdAt
* updatedAt
* source
* notes

TemporaryAccessStatus:

* requested
* active
* expired
* revoked
* denied
* cancelled

Add pure helper functions such as:

* getTemporaryAccessStatus(grant, now)
* isTemporaryAccessActive(grant, now)
* isTemporaryAccessExpired(grant, now)
* getTemporaryAccessDisplayLabel(grant, now)

No persistence.
No UI.
No automatic revocation.
No scheduler.
Derived/lazy status only.

5. Access Review Types

Define types only for future use.

Suggested shape:

AccessReviewRecord:

* id
* reviewPeriod
* reviewerId
* reviewerName
* reviewedSubjectId
* reviewedSubjectName
* reviewedRoleId
* reviewStatus
* reviewedAt
* findings
* actionRequired
* notes
* createdAt
* updatedAt

AccessReviewStatus:

* pending
* reviewed_no_change
* reviewed_change_required
* escalated
* overdue
* deferred

Add pure helper functions such as:

* isAccessReviewOverdue(record, now)
* getAccessReviewStatusLabel(record, now)
* shouldIncludeRoleInAccessReview(roleId)
* getRecommendedReviewCadence(roleId)

No workflow.
No UI.
No persistence.

6. Stale Access / Risk Helpers

Add pure helper functions only.

Examples:

* getRoleRiskPosture(roleId)
* getRoleGovernanceSummary(roleId)
* shouldRequireReasonForPrivilegedAction(actionCategory)
* shouldAuditPrivilegedAction(actionCategory)
* shouldRecommendPimForAction(actionCategory)
* getFutureServerEnforcementRecommendation(actionCategory)

These helpers should return labels/recommendations only.

They must not block actions.
They must not change current permissions.

7. Server-Side Enforcement Boundary Labels

Based on Milestone 0 tiers, define safe labels such as:

* ui_only_currently
* future_server_validation_recommended
* future_server_validation_strongly_required
* future_privileged_pim_controlled_server_action

Use these labels for documentation and future planning only.

Do not implement backend enforcement.
