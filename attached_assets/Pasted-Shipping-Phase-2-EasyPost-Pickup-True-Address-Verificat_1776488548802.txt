==================================================
ROOT-CAUSE CONTEXT
==================================================

Current behavior:
- The app now correctly identifies the pickup address source as the shipment origin address
- The app pre-checks for required fields
- But EasyPost pickup still fails with invalid-address semantics on at least one carrier path
- The visible failure was observed on USPS, but the implementation fix must not be USPS-specific

This means field presence is not enough.
The actual pickup address must pass true provider/carrier validation before pickup booking is attempted.

==================================================
CURRENT QA SOURCE OF TRUTH — ISSUE TO FIX
==================================================

Observed issue:
- all required pickup-address fields can appear present
- yet EasyPost may still reject pickup create with invalid-address semantics
- failing step = pickup_create

Therefore:
- the app must stop treating “all fields present” as “pickup-ready”
- the app must verify the actual pickup address object with EasyPost before pickup booking
- this verification model must support EasyPost pickup carriers generically, not just USPS

==================================================
NON-NEGOTIABLE REQUIRED FIXES
==================================================

1) TRUE PICKUP-ADDRESS VERIFICATION
Required:
- use EasyPost address verification on the exact pickup address source used for pickup booking
- do this before enabling or sending the pickup create request
- do not rely only on local field presence checks

Implementation intent:
- verify the same address object that resolvePickupAddress(shipment) returns
- use EasyPost verification / strict verification appropriately
- the exact address sent to pickup booking must be the address that has passed verification or has been explicitly accepted by the operator after correction
- this logic must be reusable across EasyPost pickup carriers, especially USPS, UPS, and FedEx

2) DISTINGUISH FIELD PRESENCE FROM VERIFIED PICKUP READINESS
Required:
- split the concept into:
  - fields present
  - address verified for pickup booking
- Request Carrier Pickup must require true pickup-address verification readiness, not only presence

3) HANDLE CORRECTABLE VS NON-CORRECTABLE ADDRESSES
Required:
- if EasyPost returns a corrected/normalized address, present it clearly
- allow the operator to accept/update the pickup address source when appropriate
- if the address cannot be verified, keep pickup blocked and explain what to fix
- if suite/apartment/unit or ZIP/street mismatch appears to be the issue, surface that clearly
- keep the UX generic enough to work across USPS, UPS, and FedEx pickup flows while still surfacing carrier-specific details when returned

4) CLEAR OPERATOR UX
Required:
- show the pickup address verification state clearly, such as:
  - not verified
  - verified
  - corrected — review needed
  - failed verification
- if failed, show a concise operator summary first
- keep raw provider details secondary

5) DO NOT ATTEMPT PICKUP CREATE WHEN VERIFICATION HAS FAILED
Required:
- if pickup address verification fails, Request Carrier Pickup must not proceed to the EasyPost pickup create call
- block early and clearly

6) KEEP THE PICKUP ADDRESS SOURCE EXPLICIT
Required:
- preserve the current explicit source-of-truth messaging:
  - shipment origin address is being used
- but now add real verification state tied to that source

7) CARRIER-AGNOSTIC EASYPOST ERROR HANDLING
Required:
- if EasyPost still returns pickup-address failure after verification, surface that honestly
- do not hardcode USPS-only assumptions in the logic
- where carrier-specific error details exist, show them
- but keep the validation/readiness model generic for EasyPost pickup carriers
