CRITICAL RULE — ONE ESCALATION TRUTH SOURCE
===========================================

Create or consolidate a shared escalation predicate/helper.

Recommended helper names:

* isActiveEscalation(case)
* isEscalationTerminal(case)
* getEscalationStatusLabel(case)
* getEscalationSignalsForCase(case)
* getEscalatedCases(cases)
* getTenantEscalationSignals(tenantId, cases)

Use this helper across:

* Support Tools case list
* Support Tools case detail escalation card
* escalation action visibility
* escalation saved views
* Command Center escalation count
* Command Center escalation list
* Needs Attention
* Next Best Actions
* Tenant Risk Summary
* Tenant 360
* Audit escalation lifecycle references where applicable

Do not allow different pages to use different definitions.

Suggested active escalation logic:
A case is actively escalated when:

* case.escalated === true
  AND
* case.escalationStatus is one of:

  * escalated
  * acknowledged
  * in_review
  * pending_review
  * active
    or current app equivalent
    AND
* support case itself is not closed/resolved terminal.

If current data uses another status convention, adapt safely and document it.

Terminal escalation statuses:

* deescalated
* resolved
* rejected only if it refers to request rejection, not escalation status
* none
* cancelled
* closed terminal support case if applicable
