CURRENT QA SOURCE OF TRUTH — REMAINING FAILURES
==================================================

These are the exact remaining blockers and must be fixed literally:

1) Tracking number is still masked in return-linked shipment surfaces
Observed QA result:
- UPS tracking is still shown with XXXXX in return-linked shipment/shipping surfaces
Required behavior:
- full tracking number must remain visible to authorized operational users everywhere it should be visible
- no masking/truncation regression in return-linked shipment views

2) Address validation UX is too fragmented
Observed QA result:
- there are two separate buttons:
  - Validate Origin
  - Validate Destination
- this creates a clunky operator workflow
Required behavior:
- use one combined validation action for provider-mode rate readiness
- operator should have one clear button/action that handles address validation readiness for both Origin and Destination
- the flow can still:
  - correct/suggest address fixes
  - then ask for validation/confirmation if needed
- but the operator-facing entry should be one coherent action, not two separate buttons
- Get Rates must remain blocked until both Origin and Destination are validated/accepted where applicable

3) Return status still does not mirror linked shipment precisely enough
Observed QA result:
- when linked shipment is Packed, return still shows Label Created
- when linked shipment is Dispatched, return jumps to In Transit
Required behavior:
- return shipping-leg status must mirror the linked shipment status more faithfully
- expected mapping for the reverse-shipping leg should be:
  - linked shipment Ready -> return Ready (or equivalent shipping-leg state, not stale Approved)
  - linked shipment Label Created -> return Label Created
  - linked shipment Packed -> return Packed
  - linked shipment Dispatched -> return Dispatched
  - only after actual carrier/provider progression/scan should return become In Transit
  - linked shipment Delivered -> return Delivered / Awaiting Receipt (or the current equivalent pre-Received state)
- do not skip directly from Dispatched to In Transit unless carrier/provider events actually justify it
