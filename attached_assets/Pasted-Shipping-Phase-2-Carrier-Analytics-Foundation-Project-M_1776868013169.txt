==================================================
PART A — CARRIER ANALYTICS FOUNDATION
==================================================

Required concepts:
- analytics must be based on real shipment/provider/tracking/pickup data already in the system
- metrics must be truthful about source quality and coverage
- manual shipments and provider-mode shipments must not be conflated silently
- incomplete or insufficient data must be shown honestly, not guessed

Required features:

1) Carrier Analytics surface inside Shipping Center
Required:
- add a Carrier Analytics surface inside Shipping Center
- keep it consistent with the current design baseline
- it may be a tab, section, or analytics panel, but it must feel native to current Shipping Center IA
- do not create a disconnected mini-app

2) Analytics filters
At minimum support:
- date range
- store / location context if available in current architecture
- provider
- carrier
- shipment type/context where available
- outbound vs return shipment context where available
- manual vs provider-mode distinction

3) Truthful core metrics
At minimum support metrics that can be derived honestly from current data, such as:
- shipment count
- label purchased count
- provider-mode shipment count
- manual shipment count
- pickup requested count
- pickup confirmed count
- pickup cancelled count
- delivered count
- cancelled shipment count
- shipping spend totals where actual stored cost exists
- average shipping cost where actual stored cost exists
- carrier/provider distribution
- service-level distribution where available

4) Transit / delivery analytics
Required:
- only calculate transit or delivery-speed metrics when enough timestamp/state data exists
- if insufficient data exists, show a truthful unknown / insufficient-data state
- do not invent SLA performance
- if feasible from current shipment lifecycle data, support metrics such as:
  - average time from label created to delivered
  - average time from dispatched to delivered
  - delivered vs in-transit vs exception/open counts
- if exact provider event coverage is partial, reflect that honestly

5) Pickup analytics
Required:
- include pickup analytics using the completed Phase 2 pickup records
- show requested / confirmed / cancelled / failed / local-only where applicable
- preserve truthful provider-state handling
- do not overstate provider-confirmed pickups when they were only local-only intents

6) Data-quality / coverage states
Required:
- clearly distinguish:
  - actual metric
  - unavailable metric
  - insufficient-data metric
  - partial provider coverage
- show concise operator-facing explanations where a metric is not fully reliable
- do not silently mix local-only operational records with live-provider-confirmed records unless clearly labeled

7) Auditability / metric definitions
Required:
- metric definitions must be deterministic
- analytics must be traceable to current shipment records/statuses/events/cost fields
- avoid magic calculations
- use stable definitions so future scorecards can build on them
