CURRENT QA SOURCE OF TRUTH — REMAINING FAILURES
==================================================

1) Plan gating is still not enforced in live behavior
Observed QA result:
- disabling Service Point / Drop-Off or Carrier Pickup Request in System Owner Plans & Features does not remove/disable the capability in practice
Required behavior:
- if disabled at plan level, the capability must not be usable for that tenant/plan
- UI and handlers must respect the plan state
- do not show a contradictory “not included in your current plan” state when the plan actually includes it
- fix the real source-of-truth / evaluation path

2) Store permission denial is still not enforced in live behavior
Observed QA result:
- denying Service Point / Drop-Off or Carrier Pickup Request still leaves the user able to see/use the feature incorrectly
Required behavior:
- if denied in Store Permissions, the user must not be able to edit/select/use the capability
- UI and handlers must both enforce the denial
- do not show misleading plan-disabled messaging when the real issue is permission state or vice versa

3) Provider-unavailable behavior is still inconsistent
Observed QA result:
- when no provider is configured, Provider & Operations still leaves Validate Addresses enabled
Required behavior:
- in provider-dependent mode, when there is no active provider, provider-dependent actions must be disabled
- Provider & Operations should show aligned honest messaging with Service Points and Pickup Requests
- no provider-dependent actions should appear falsely active

4) Service Point / Pickup should only be active when the shipment is at the correct lifecycle stage
Required behavior:
- Service Point / Drop-Off and Carrier Pickup Request should only be active when the shipment is in the intended eligible lifecycle stage
- based on the current business rule from QA, this should be when the shipment is PACKED
- before PACKED, they should not be active/selectable
- preserve provider/manual mode logic

5) ZIP-code search should only be meaningful when service-point selection is actually eligible
Required behavior:
- ZIP-code search should be enabled only when the shipment is actually eligible to use Service Point / Drop-Off
- if provider is missing, plan disabled, permission denied, wrong lifecycle, or shipment otherwise ineligible, the search should not behave like a live usable flow
- keep the UI honest
- if current provider capability is limited, message that clearly

6) Mutual exclusivity must remain intact
Required behavior:
- if one of Service Point / Drop-Off or Carrier Pickup Request is used, the other remains disabled/cleared
- preserve this while fixing the other gating issues
