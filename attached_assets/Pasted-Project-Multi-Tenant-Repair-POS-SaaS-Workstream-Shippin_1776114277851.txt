MANDATORY CORRECTIONS
--------------------------------------------------

A) SHIPPING CENTER-NATIVE PROVIDER SETTINGS PATH
Implement an actual native provider-settings access path inside Shipping Center.

Required:
- Shipping Center must include a clear Settings / Providers / Configure path inside the module itself
- This path must feel native to Shipping Center operations
- Governance/security can still remain in the existing admin/settings architecture
- Do not duplicate provider config logic
- Do not leave Shipping Center users feeling forced to discover provider setup only through unrelated settings areas

B) PDF-FIRST LABEL OUTPUT FIX
Required:
- The primary label artifact handling/open action must use PDF-first behavior
- If the provider returns multiple label formats, prefer the PDF artifact for the main operational open/print action
- UI should reflect PDF as the primary label format
- Do not leave the main operational path opening PNG when PDF-first was required

C) FULL TRACKING NUMBER VISIBILITY
Required:
- Show the full tracking number to authorized operational users
- Remove truncation/masking in shipment detail and list views where operationally needed
- Keep lock behavior after label purchase
- Do not leak tracking data to unauthorized users, but for authorized shipping users the full value should be visible

D) MAKE TRACKING & EVENTS OPERATIONALLY PROVIDER-FED
This is the most important missing fulfillment behavior.

Current problem:
- Tracking tab still looks manual-only in practice
- Webhook foundation exists but does not create a visible provider-fed operational timeline
- QA expects that once shipment reaches dispatched and the carrier/provider has events, Tracking & Events reflects those provider events automatically

Required:
- Make provider-sourced events clearly visible in the Tracking & Events tab when available
- Ensure provider events are visually distinguishable from manual events
- Ensure the timeline is genuinely unified, not only conceptually
- Ensure provider event ingestion updates latest tracking state and mapped shipment state coherently
- Ensure provider-origin events are not all appearing as manual

Because QA is being done in test mode and real carrier scans may not naturally happen:
- add a safe, non-production test validation path for provider event ingestion
- this can be a controlled provider test-event sync path, webhook replay/test-event ingestion path, or equivalent admin/test-only validation mechanism
- the goal is to prove the provider-fed tracking pipeline actually works without waiting for real live carrier scans

Important:
- do NOT fake provider events in a misleading way
- do NOT pretend live carrier scans happened when they did not
- do provide a truthful, test-safe way to validate that webhook/provider event ingestion updates the shipment timeline correctly

E) TEST MODE VS LIVE MODE CLARITY
Required:
- In test mode, clearly explain when provider event limitations are due to the provider’s test environment
- Keep the UI honest and operationally understandable
- Distinguish:
  - manual event
  - synced provider event
  - no provider event available yet
  - provider test-mode limitation
