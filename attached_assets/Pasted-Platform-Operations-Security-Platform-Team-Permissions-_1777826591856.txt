CURRENT ISSUE
==================================================

Support Tools currently has a local “Acting as” selector/card.

This is not the correct long-term design because:

1) The app already has Dev Session role switching.
2) Platform roles should be selected through the Dev Session / active platform user context.
3) Global Permissions Matrix should define what each platform role can do.
4) Support Tools should read permissions from the active platform role, not from a local selector.
5) This should work like the tenant/store side, where role/sub-permission behavior is controlled centrally by a matrix.

The local “Acting as” selector must be removed from Support Tools.

Support Tools should only ask:
“What permissions does the active platform role have?”

==================================================
IMPORTANT CORRECTION — USE EXISTING GLOBAL PERMISSION LEVELS
==================================================

The Global Permissions Matrix already supports these permission levels:

- None
- View Only
- Create
- Edit
- Approve
- Manage
- Full Access

Do NOT replace this model with a simplified model.

Do NOT reduce it to:
- None
- View
- Manage
- Full Access

Use the existing permission hierarchy everywhere in this workstream.

Permission level meaning:

None:
- No access.
- Feature/action hidden or blocked.

View Only:
- Can open/read/view.
- Cannot create, edit, approve, manage, or perform operational mutations.

Create:
- Can create new records, such as support cases, domain records, notes, or draft items where applicable.
- Does not automatically allow editing existing records unless Edit or higher is granted.

Edit:
- Can update existing records and normal workflow fields.
- Does not automatically allow approval-sensitive actions unless Approve or higher is granted.

Approve:
- Can perform approval/authorization-sensitive workflow actions.
- Use for actions that approve, acknowledge, override, de-escalate, resolve sensitive escalations, approve exceptions, or authorize sensitive transitions.
- Approval is distinct from general management.

Manage:
- Can administer operational configuration and manage the feature area broadly.
- Can perform most Create/Edit/Approve actions unless a sub-permission is explicitly denied or protected.

Full Access:
- All actions.
- System Owner should retain Full Access and should not be reducible if owner permanence is part of the current design.

==================================================
PARENT FEATURE LEVEL VS SUB-PERMISSIONS
==================================================

Parent feature permission controls general access level.

Sub-permissions refine specific actions.

Required behavior:

- Parent feature level should provide reasonable default capability.
- Sensitive sub-permissions must still be checked directly.
- Parent Full Access grants all sub-permissions unless explicitly protected by System Owner permanence rules.
- Parent Manage should generally allow most operational actions, but approval-sensitive actions should still respect Approve / Manage / Full Access thresholds.
- Parent Edit should not automatically allow approval, escalation resolution, audit export, permission-matrix edits, delete actions, or other sensitive actions unless the matching sub-permission allows it.
- Parent Create should not automatically allow editing existing records.
- Parent View Only should never allow mutation actions.
- Explicit denied / None sub-permission should not be overridden by parent level except for System Owner Full Access if that is the existing app rule.
- Sub-permissions must use the existing hierarchy: None, View Only, Create, Edit, Approve, Manage, Full Access.
