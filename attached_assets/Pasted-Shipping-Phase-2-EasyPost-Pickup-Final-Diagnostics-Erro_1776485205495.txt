CURRENT QA SOURCE OF TRUTH — REMAINING FAILURE
==================================================

Observed QA result:
- EasyPost pickup no longer fails silently
- But the visible error is still too generic:
  "Pickup create failed, The request was well-formed but was unable to be followed due to semantic errors."
- This is not enough to diagnose or operate the feature
- We need the app to reveal the actual EasyPost failure cause

This must now become a true diagnostics pass.

==================================================
NON-NEGOTIABLE REQUIRED FIXES
==================================================

1) SURFACE THE REAL EASYPOST ERROR DETAILS
Required:
- when EasyPost pickup create/buy fails, surface the actual structured provider error details to the operator in a clean readable way
- at minimum include:
  - provider error code
  - top-level provider message
  - field-level validation details when present
  - which stage failed:
    - pickup create
    - pickup buy
    - pickup cancel
- do not stop at the generic semantic-error wrapper

2) PRESERVE STRUCTURED SERVER ERRORS
Required:
- server must preserve and return structured EasyPost error payload details instead of collapsing them into one generic sentence
- if EasyPost returns nested errors, keep those available to the client
- do not swallow useful provider diagnostics

3) CLIENT-SIDE DIAGNOSTIC DISPLAY
Required:
- shipment pickup UI must display the failure in an operator-friendly but specific way
- if there are multiple field-level issues, render them as a short structured list
- clearly indicate what the user must correct next if the provider error makes that obvious

4) IDENTIFY THE FAILING STEP
Required:
- the UI must clearly show whether the failure occurred during:
  - Create Pickup
  - Buy Pickup Rate
  - Cancel Pickup
- do not leave the operator guessing which step failed

5) SHOW THE KEY REQUEST PREREQUISITES USED
Required:
- for diagnostics, expose enough context to understand why EasyPost may reject the request, such as:
  - provider shipment reference existence
  - pickup window used
  - pickup address source
  - whether account-address mode is being used
  - whether pickup rates were returned
- keep this concise, not overly technical, but sufficient to debug

6) HONEST TEST-MODE MESSAGING
Required:
- if test mode contributes to the failure, say that specifically
- do not blame test mode unless the provider response or flow actually supports that conclusion
- distinguish:
  - true provider validation error
  - test-mode limitation
  - missing app-side prerequisite
  - provider buy-step failure

7) KEEP SUCCESS PATH READY
Required:
- if the underlying issue is fixed later, the app must still cleanly show:
  - created
  - rate returned
  - booked
  - confirmation number
  - cost
- do not regress the successful path while improving diagnostics
