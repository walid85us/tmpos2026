CRITICAL EFFECTIVE PERMISSION RULE
==================================================

Create one final platform effective permission resolver and use it everywhere.

The resolver must support:

1) Parent permission
2) Sub-permission
3) Explicit sub-permission override
4) Child-granted feature visibility
5) System Owner permanence
6) Dev Session active role
7) Matrix overrides from sessionStorage
8) No hardcoded role override unless used only as default seed

Definitions:

Parent feature permission:
- controls broad module access

Sub-permission:
- controls specific action or child capability

Effective feature visibility:
A platform feature/page/sidebar item is visible if:
- parent feature permission is View Only or higher
OR
- any child sub-permission under that feature is View Only or higher
OR
- System Owner Full Access applies

Effective page access:
A page can open if:
- its parent feature is effectively visible
AND
- the page’s required view sub-permission is View Only or higher, if the page has a required view sub-permission
OR
- at least one page-relevant child permission is View Only or higher

Effective action access:
An action is allowed if:
- the specific action sub-permission meets the required threshold
OR
- no explicit sub-permission override exists and parent feature permission meets the required threshold
OR
- System Owner Full Access applies

Critical override rule:
- If a specific sub-permission is explicitly set to None, that action is denied.
- If a specific sub-permission is explicitly set to Full Access, that action is allowed, assuming the user can access the page.
- Old hardcoded role rules cannot deny a matrix-granted action.
- Old hardcoded role rules cannot allow a matrix-denied action.

Use old role rules only as:
- default seed values
- optional advisory explanation
- not final authorization
