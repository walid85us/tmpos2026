==================================================
WORKSTREAM GOAL
==================================================

Add a solid Carrier Scorecards foundation inside Shipping Center.

The goal is to help store operators compare carrier/service performance using real recorded data only.

Carrier Scorecards should support:
- carrier/service performance summaries
- SLA performance by carrier/service where data exists
- cost and usage summaries
- delivery/transit reliability where timestamps exist
- pickup performance where pickup records exist
- exception/failure visibility
- data-quality and coverage indicators
- truthful scorecard calculations that do not overclaim precision

This is a foundation.
Do NOT fake complete carrier intelligence.
Do NOT pretend all carriers/providers have equal data coverage.
Do NOT pretend manual shipments have the same provider-event detail as provider-backed shipments.

PART B — SCORECARD DATA MODEL / DERIVATION FOUNDATION
==================================================

Required:
Build scorecards from existing recorded shipment data.

Use current available data sources such as:
- shipment records
- provider/mode fields
- carrier/service fields
- selected rate / cost fields
- label purchase timestamps
- dispatch timestamps
- tracking timestamps/states where available
- delivery timestamps where available
- pickup request/confirmed/cancelled/failed/local-only states
- SLA targets and SLA outcomes
- packing and dispatch lifecycle timestamps
- automation outcomes only where relevant and clearly labelled

Required:
Define deterministic derivation rules for every scorecard metric.
No magic calculations.
No invented carrier data.

==================================================
PART C — CORE SCORECARD METRICS
==================================================

At minimum support truthful metrics where data exists:

1) Usage / Volume
- total shipments
- provider-backed shipments
- manual shipments
- outbound shipments
- return shipments
- label purchased count
- delivered count
- cancelled count

2) Cost
- total shipping spend where stored cost exists
- average shipping cost where stored cost exists
- min / max shipping cost if useful
- cost coverage indicator if some shipments lack stored cost

3) SLA Performance
- SLA met count
- SLA missed count
- SLA at-risk count
- SLA overdue count
- SLA paused count
- SLA unknown / insufficient-data count
- SLA performance rate only where denominator is valid and documented

4) Transit / Delivery Performance
- average label-to-dispatch time where timestamps exist
- average dispatch-to-delivery time where timestamps exist
- average label-to-delivery time where timestamps exist
- delivered vs open/in-transit/exception counts where tracking data exists
- unknown/insufficient-data state where timestamps are missing

5) Pickup Performance
- pickup requested count
- pickup confirmed count
- pickup cancelled count
- pickup failed count
- local-only pickup count
- provider-confirmed pickup coverage
- pickup performance should preserve provider-state truthfulness

6) Exception / Reliability
- failed provider operation count where recorded
- tracking exception count where available
- automation review/approval/guardrail outcomes if relevant and clearly scoped
- cancelled shipment count
- unresolved SLA/packing/shipping exceptions where available

==================================================
PART D — SCORE / RATING LOGIC
==================================================

Required:
If scorecards include a numeric score or grade, it must be transparent and data-quality-aware.

Preferred:
Start with metric scorecards before introducing one combined score.

If implementing a combined score, it must:
- show exact components
- show weights
- show coverage
- exclude metrics with insufficient data instead of pretending they exist
- show “insufficient data” when minimum sample size is not met
- not rank carriers unfairly based on tiny samples

Required:
Add minimum sample-size handling.
Example:
- if carrier has fewer than N completed shipments in the selected period, show “limited data”
- do not overstate performance

Recommended foundation:
- Volume
- Cost
- SLA
- Delivery/Transit
- Pickup
- Exceptions
- Overall score only if enough data exists

==================================================
PART E — DATA QUALITY / COVERAGE STATES
==================================================

Required:
Carrier Scorecards must clearly show data quality.

At minimum distinguish:
- full provider-backed coverage
- partial provider coverage
- manual shipment data
- insufficient timestamp coverage
- insufficient SLA coverage
- insufficient sample size
- cost data unavailable
- delivery data unavailable
- pickup data local-only vs provider-confirmed

Required:
Do not silently mix:
- manual shipments with provider-backed shipments
- local-only pickup intents with provider-confirmed pickups
- dispatched shipments with in-transit shipments
- delivered shipments with assumed delivery

Use truthful labels such as:
- “Based on recorded shipment data”
- “Partial provider coverage”
- “Insufficient delivery timestamps”
- “Manual shipment coverage only”
- “Limited sample size”

PART I — RELATIONSHIP TO AUTOMATION
==================================================

Required:
Carrier Scorecards may use automation outcomes only as operational signal, not as proof of carrier failure unless directly related.

Allowed examples:
- count carrier-related automation review outcomes
- count SLA automation outcomes by carrier/service if relevant
- show unresolved shipping exceptions by carrier/service

Not allowed:
- blame carrier for internal packing delay
- blame carrier for store-created SLA pause
- treat every automation review as carrier fault

Required:
Where delay reason exists, use it to avoid misattribution.
