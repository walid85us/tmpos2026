==================================================
PHASE 2.9 REQUIRED OUTCOME
==================================================

At pickup booking time, the operator must see the real provider-returned pickup options and exact fee(s) before confirming the booking.

==================================================
NON-NEGOTIABLE REQUIRED FEATURES
==================================================

1) SURFACE pickup.create pickup_rates TO THE UI
Required:
- wire the actual pickup.create response's pickup_rates array through to the pickup-booking UI
- do not discard it
- do not continue auto-buying the first returned rate silently

2) EXACT PICKUP-RATE SELECTION UX
Required:
- when the operator opens the pickup form on a packed shipment with a label, the app should run pickup.create using the date/window/contact already in scope
- then render the returned pickup_rates as a selection UI

Required behaviors:
- one rate and rate === 0:
  - show a clear “Free pickup” indicator
  - single confirm button
- one rate and rate > 0:
  - show the exact fee prominently
  - confirm button includes the exact fee
- multiple rates:
  - show a radio/select list with carrier, service, exact rate, currency
  - confirm button stays disabled until one is selected
- if provider returns additional useful message detail or surcharges, render them clearly without clutter

3) pickup.buy MUST USE THE CHOSEN RATE
Required:
- call pickup.buy against the operator’s chosen pickup rate
- do not default implicitly to the first rate anymore

4) PERSIST EXACT BOUGHT PICKUP RATE + CONFIRMATION
Required:
- persist the exact bought pickup rate, currency, chosen carrier/service, and confirmation number on the pickup record
- render them on the shipment row / pickup summary after successful purchase
- confirmation UI must match the actual bought rate

5) EMPTY pickup_rates IS NOT partial_failed
Required:
- if pickup.create succeeds but returns an empty pickup_rates array:
  - treat this as an honest pre-booking “no rates available for this date/window/provider combination”
  - show a clear message such as:
    “No pickup rates available for this date/window through the current provider. Adjust the window or use drop-off.”
  - provide a retry/change-window path
- do NOT style this as partial_failed
- do NOT create a fake booked/requested state
- do NOT create an orphan that needs cancellation if booking never actually occurred

6) KEEP TRUE partial_failed FOR REAL CREATE/BUY FAILURES
Required:
- if pickup.create throws or pickup.buy fails after a provider pickup object exists, keep the truthful partial_failed behavior already implemented
- do not weaken honesty

7) KEEP PHASE 2.8 FORECAST/FEE BADGES ADVISORY ONLY
Required:
- the Phase 2.8 forecast and fee-class badges stay advisory
- do not back-propagate exact pickup fees into Get Rates
- do not show estimated dollar amounts on rate rows

8) UPS / USPS / FEDEX TRUTHFULNESS
Required:
- do not regress working USPS behavior
- do not regress working FedEx behavior
- keep UPS service-family gating intact
- if UPS DAP or another known unsupported service is blocked earlier, keep that block and do not attempt to force it through Phase 2.9

- Do not change Phase 2.8 forecast/fee-class logic except as strictly necessary for consistency
