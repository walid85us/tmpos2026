==================================================

- the app must not continue allowing:
  - Get Rates
  - rate selection
  - Purchase Label
  - provider-backed validation
  - provider-backed pickup operations
  - provider-backed tracking sync
  - any other provider-dependent action
when the plan no longer includes Shipping Provider capability

Critical rule:
- disabling Shipping Provider at the plan level must override any existing stored provider configuration
- if EasyPost or another provider had already been configured, the system must treat provider capability as unavailable at runtime

Implementation intent:
- do not necessarily destroy stored credentials/configuration data
- but operationally the provider must be treated as disconnected/inactive while the feature is disabled for the plan
- when the feature is disabled, the runtime should behave as though no usable provider capability is available

Required UX:
- show truthful not-included / unavailable-for-plan messaging
- do not silently keep provider-backed actions alive because configuration already exists
- do not let stale active-provider state bypass entitlement checks

Required enforcement layers:
- UI gating
- action-handler gating
- operational prerequisite evaluators
- any server/client provider invocation path that currently assumes configured provider = allowed provider
