==================================================
MANDATORY CONTEXT
==================================================

The current automation implementation is partially working, but the workflow model is still wrong.

Current issue discovered in QA:

Example rule:
- When a carrier label is purchased
- and Shipping Cost > $6.00
- mark the shipment for review

What currently happens:
- label is purchased
- rule fires afterward
- shipment gets review-needed state
- review can be resolved later

Why this is not sufficient:
- in a real workflow, this kind of rule should often act as a pre-action guardrail, not just a post-action review
- if the business rule is “labels above $6 require review/approval,” then the label purchase should not complete normally before that review is addressed
- or the operator should choose a lower-cost rate that avoids the rule
- or a user with permission should approve/override the guardrail

This means the current automation model is too flat.
It treats all rules as post-event reactions, but some must be modeled as pre-action guardrails.

Additional issues:
1) There is not yet a strong permission model for automation outcomes.
2) Review resolution is too generic — it should distinguish resolve / approve / override depending on action type.
3) Execution History should be more operational:
   - shipment number should be clickable
   - operator should be able to open the shipment from history
   - resolved items should show resolved state
4) Rule triggers, conditions, actions, and permission model must now be logically linked as a practical workflow system.

This correction pass must solve that.

PART A — INTRODUCE RULE PROCESS TYPES
==================================================

Required:
Refactor the automation foundation so supported rules are explicitly classified as one of these types:

1) Observational / Post-Event
Purpose:
- reacts after an event already happened
Examples:
- shipment created
- shipment edited
- shipment status changed
- carrier label purchased
- pickup requested
- pickup confirmed
- pickup cancelled
- tracking synced
- return shipment created

Allowed behavior:
- add internal note
- add audit/timeline event
- set priority
- queue for batch
- mark review needed
- surface recommendation / follow-up
These must remain safe and truthful.

2) Guardrail / Pre-Action
Purpose:
- evaluates before a risky or policy-sensitive action completes
Examples that should be supported where practical in current shipping architecture:
- before carrier label purchase
- before pickup request confirmation
- before rate selection finalization if architecture allows
At minimum support the pre-label-purchase guardrail path if that is the cleanest current implementation.

Allowed behavior:
- allow action
- require review/approval before action proceeds
- block action with clear reason unless approved/overridden
- allow operator to choose a different compliant option instead

Important:
Not every trigger from the current dropdown must become guardrail-capable.
Only make guardrails where they are operationally valid and supportable from current architecture.
Explain the final supported matrix clearly.

PART B — BUILD A LOGIC PROCESS MATRIX FOR EACH RULE / TRIGGER FAMILY
==================================================

Required:
Create a clear internal process matrix linking:
- trigger
- rule type
- supported conditions
- supported actions/outcomes
- whether the action is post-event, review-required, approval-required, or blocking
- who can resolve / approve / override

You must think through the logic for the current trigger families visible in the automation UI, including:
- a shipment is created
- a shipment is edited
- shipment status changes
- a carrier label is purchased
- a pickup is requested
- a pickup is confirmed
- a pickup is cancelled
- tracking is synced
- a return shipment is created

For each trigger family, define the practical action model.
Example guidance:
- Shipment created: good for tagging, routing, review flagging, batch candidacy, notes
- Shipment edited: good for re-evaluation, review flagging, note/audit, queue update
- Shipment status changes: good for queue movement, exception flagging, follow-up markers
- Carrier label purchased: good for post-event audit/flagging; but policy controls like high-cost label approval likely belong to pre-label guardrail instead
- Pickup requested/confirmed/cancelled: good for follow-up, review, note, queue, exception states
- Tracking synced: good for exception flagging, delay/follow-up, resolution prompts
- Return shipment created: good for routing, return review, note/queue state

Important:
Do not keep illogical trigger-action relationships.
For example:
- “after label purchased” should not be the only available way to enforce pre-purchase cost approval
- guardrail logic must exist for the rules that are clearly policy-enforcement rules

PART C — PRE-LABEL PURCHASE GUARDRAIL WORKFLOW
==================================================

This is mandatory because it directly addresses the QA example.

Required:
Introduce a clean guardrail workflow for label purchase where policy-sensitive rules can evaluate before purchase completes.

Example supported business outcome:
- if Shipping Cost > threshold
- require review/approval before purchase
- allow approved user to proceed
- allow normal operator to choose a cheaper rate instead
- do not let label purchase complete silently and only review afterward if the business rule is a pre-purchase control

At minimum, support:
- guardrail evaluation point before label purchase
- visible hold / approval-needed / review-needed state
- clear reason shown to operator
- action options depending on permission:
  - select another rate
  - request approval
  - approve/override if authorized
  - cancel

Do not silently downgrade this back into a post-event flag.

Required:
Think through default practical role expectations, while still supporting the app’s custom role system.
For example:
- Store Owner / Admin: can manage rules, resolve reviews, approve exceptions, override guardrails
- Manager / Operations Lead: can resolve reviews, approve exceptions, possibly override depending on policy

Important:
Permissions must be logically linked to rule outcomes.
If a rule creates approval-needed state, someone must have a defined permission to approve it.
If a rule creates review-needed state, someone must have a defined permission to resolve it.
