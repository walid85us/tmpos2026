==================================================
CURRENT QA SOURCE OF TRUTH — REMAINING FAILURES
==================================================

These are the exact remaining blockers and must be fixed literally:

1) Return creation from invoice/source still needs item-level improvement
Observed QA result:
- return can be created from invoice
- items can be included/excluded
Required behavior:
- if the same item quantity is greater than 1, the quantity selected for return must be editable
- item price must be visible in the return flow where refund/reimbursement context matters
- operator should not have to infer price manually

2) Return status is not synchronized correctly with linked return shipment
Observed QA result:
- linked return shipment can show Ready while the return still shows Approved
- linked shipment can show later shipping progress while return action buttons still allow manual progression such as Label Created / In Transit
Required behavior:
- when the return is using a linked return shipment, the reverse-shipping phase of the return must be driven by the linked shipment status, not by manual button clicks
- return must not falsely stay at Approved when the linked shipment is already Ready
- return must not allow clickable Label Created / In Transit actions in provider-managed reverse-shipping flow
- manual progression for those shipping-leg statuses is acceptable only when the return shipment is fully manual/no provider-driven shipping flow exists

3) Return should not show Label Created before actual label purchase
Observed QA result:
- operator can still move the return to Label Created manually even when provider-managed shipment has not actually purchased the label yet
Required behavior:
- if shipping provider flow is active, Label Created must only appear when the linked return shipment label is actually purchased/created
- manual Label Created action is only acceptable in manual/no-provider mode

4) Return shipment creation flow needs better field behavior
Required behavior:
- when creating a return shipment from the return section:
  - customer address must populate from customer/profile/source data
  - origin and destination must be editable before dispatch
  - Carrier, Service Level, and Tracking # must not be manually pre-entered in the return-shipment create step if those are meant to be driven later in Shipping Center
- keep the create flow clean and consistent with the shipping architecture

5) Linked shipment access should be easier
Required behavior:
- shipment number shown inside the return should be clickable
- clicking it should open the linked shipment detail/workflow, respecting role and permission rules

6) Reverse shipping progression should follow linked shipment state coherently
Observed QA result:
- when linked shipment progresses, return status can still be confusing or out of sync
- when linked shipment is Delivered, the return still shows In Transit
Required behavior:
- between shipment creation and shipment delivered, the reverse-shipping leg of the return must reflect the live linked shipment status coherently
- once the linked shipment is Delivered, the return must no longer still show In Transit
- use a clear synchronized state such as Delivered / Awaiting Receipt or equivalent that fits the current architecture
- then Received becomes the next operator step

7) Received gating needs one more refinement
Observed QA result:
- Received is enabled after linked shipment Delivered, which is good
- but the return status still showing In Transit at that moment is confusing
Required behavior:
- once linked shipment is Delivered, return must show an appropriate delivered-to-facility / awaiting-receipt state before the operator clicks Received
- Received must remain the operator action that begins intake, but the pre-Received state must be clear

8) Return label delivery to customer
Observed QA result:
- current path opens the user’s email app, which is acceptable for now
Required behavior for this pass:
- keep a lightweight operator path to give the label to the customer
- do NOT build full direct email infrastructure in this pass
- improve wording/UX if needed so the operator understands whether it opens an email draft vs sends directly
- document full direct-send email as a future workstream, not this correction pass

9) Outbound shipping regression — tracking number masking returned
Observed QA result:
- tracking number is masked again with XXXXX in shipping/return contexts
Required behavior:
- full tracking number must remain visible to authorized operational users everywhere it should be visible
- fix this regression without breaking permission safety

10) Address validation should cover both origin and destination in provider mode
Observed QA result:
- current shipping center validates destination only
Required behavior:
- for provider-mode rating, validate both origin and destination
- Get Rates should depend on the correct address-validation readiness for both sides where applicable
- if you decide on combined validation UX, make it clear and operator-friendly
- do not leave rate accuracy dependent on destination-only validation if origin materially matters
