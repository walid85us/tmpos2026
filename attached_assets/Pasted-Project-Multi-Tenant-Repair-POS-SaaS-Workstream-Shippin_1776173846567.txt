B) PDF-ONLY PRIMARY LABEL HANDLING
Required:
- The main label artifact/action must use a real PDF artifact only
- Do not use PNG for the main operational label path
- Do not label a PNG as PDF
- If a provider response includes multiple formats, choose PDF for the primary action
- Keep the workflow operationally clean and thermal-label-ready

C) FULL TRACKING NUMBER VISIBILITY
Required:
- Show the full tracking number to authorized operational users
- Remove masking/truncation in all relevant shipment detail/list surfaces where operators need the full value
- Preserve permission safety for unauthorized users
- Preserve lock behavior after label purchase

D) LIGHTWEIGHT DESTINATION ADDRESS AUTOCOMPLETE
Required:
- Add destination-address autocomplete/suggestion behavior in the shipment address flow
- When the user starts typing an address, show a suggestion dropdown
- When a suggestion is selected, prefill:
  - address line
  - city
  - state/region
  - postal code
  - other relevant location fields if available
- Keep this lightweight and operational
- Do not redesign the full form
- Do not overbuild a maps product
- Use the cleanest architecture-ready path possible
- Be truthful if a live autocomplete provider is not configured

E) CARRIER REJECTION / RETURN OUTCOME PATHS
This is a new required operational improvement.

Required:
- Add controlled shipment outcome handling for:
  - Rejected by Carrier
  - Returned
- These must fit the shipping lifecycle cleanly
- They must not behave like casual manual freeform status edits
- They should be available only where lifecycle and permissions make sense
- Require reason capture or reason selection where appropriate
- Preserve provider-driven post-dispatch authority
- Do not weaken the rule that normal post-dispatch progression comes from provider events

Reason examples may include:
- invalid address
- package refused
- carrier unable to accept
- hazardous/restricted issue
- delivery failed and returned
- customer unavailable/refused
- other operational reason

Use the current product’s style and lifecycle discipline.
