PART A — AUTOMATION RULES FOUNDATION
==================================================

Required concepts:
- Automation Rules must be deterministic, auditable, and operator-trustworthy
- Rules must not silently perform risky or irreversible actions without clear design control
- Rules must not bypass existing shipping truthfulness, lifecycle gating, or provider prerequisites
- Rules foundation should prepare the system for later Phase 3 expansion without overreaching now

Required features:

1) Automation Rules surface inside Shipping Center
Required:
- add an Automation Rules surface inside Shipping Center or Shipping settings in a way that fits the current IA
- keep it native to the current design baseline
- do not create a disconnected mini-app

2) Rule model foundation
At minimum support:
- rule name
- enabled / disabled state
- scope / store context if applicable
- trigger type
- condition set
- action set
- created by / updated by / timestamps
- execution logging / audit trail
- safe extensibility for future advanced automation

3) Trigger foundation
At minimum support clearly defined triggers such as:
- shipment created
- shipment updated
- status changed
- label purchased
- pickup requested
- pickup confirmed
- pickup cancelled
- tracking/status synced
- return shipment created
- shipment becomes Ready / Label Created / Packed / Dispatched / In Transit / Delivered / Cancelled
Only include triggers that can be implemented truthfully from the current architecture.

4) Condition foundation
At minimum support conditions grounded in current shipment data such as:
- provider mode vs manual mode
- shipment type / source type / outbound vs return
- store / location if present
- provider / carrier / service level where available
- current shipment status
- address validation state
- label existence
- pickup state
- tracking state
- shipping cost thresholds where stored cost exists
- manual vs provider-backed shipment distinctions
Do not invent conditions that are not supportable from the current data model.

5) Safe action foundation
In this pass, actions must be safe, truthful, and auditable.
Allowed action types should focus on foundation-safe behaviors such as:
- add shipment flag / priority / queue marker
- add internal operational note or timeline/audit event
- surface review-needed / exception-needed / ready-for-batch state
- assign shipment to a batch candidate queue or batch label bucket
- surface operator-facing recommendation / task / review state
- mark internal automation result state
Do NOT implement risky or misleading actions such as fake carrier operations, hidden financial changes, or silent status manipulation without explicit safe control.

6) Rule execution transparency
Required:
- when a rule runs, the result must be visible and auditable
- operators should be able to understand why a shipment was flagged / queued / marked
- avoid invisible automation firing
- avoid “magic” behavior

7) Safe activation discipline
Required:
- rules must be enable/disable-able
- foundation should support testability or safe evaluation visibility where practical
- do not allow automation to bypass existing manual checks that are still required by the current shipping architecture

==================================================
PART B — BATCH LABELS FOUNDATION
==================================================

Required concepts:
- Batch Labels must be truthful and itemized
- Batch execution must not pretend to be a provider-native bulk API if it is actually app-level orchestration of individual shipment label purchases
- Eligibility must be explicit per shipment
- Partial success / partial failure must be handled honestly

Required features:

1) Batch Labels surface inside Shipping Center
Required:
- add a Batch Labels surface inside Shipping Center in a way that fits current IA
- support multi-shipment operational workflow without redesigning unrelated shipping screens

2) Batch selection model
Required:
- allow operators to assemble a batch from eligible shipments
- support manual selection and/or rules-driven batch candidates if implemented in Part A
- preserve clear distinction between:
  - eligible shipments
  - ineligible shipments
  - partially ready shipments
- show the reason when a shipment is not eligible for batch label purchase

3) Batch eligibility evaluator
Required:
- batch eligibility must be based on the same truthful prerequisites as single-shipment label purchase
- do not invent a separate weaker path for batch mode
- preserve provider/manual distinction
- manual shipments must not be treated as provider-label-purchasable shipments
- eligibility should account for current shipping truth such as:
  - provider feature enabled
  - provider configured and available
  - shipment in valid lifecycle state
  - packages present
  - required validations satisfied
  - any selected rate or provider prerequisites that already exist in current architecture

4) Batch execution model
Required:
- batch label purchase should execute truthfully as a controlled multi-shipment operation
- if this is implemented as app-level iteration across individual shipments, say so in documentation and UI logic
- each shipment in the batch must return its own outcome:
  - success
  - failed
  - skipped / ineligible
- do not fake all-or-nothing success
- preserve current provider error truthfulness and feedback patterns

5) Batch results / auditability
Required:
- show itemized results
- preserve per-shipment audit/history updates
- allow operators to see which shipments succeeded vs failed vs skipped
- label artifacts and tracking results must remain tied to each shipment record individually
- do not collapse detailed truth into a vague batch toast

6) Batch safety / idempotency mindset
Required:
- avoid accidental double-processing
- avoid ambiguous repeated label purchases
- preserve existing label-lock behavior
- do not let batch mode bypass safeguards that single-shipment mode already enforces

7) Batch operational usefulness
Required:
- make the batch workflow operationally valuable now, not just a placeholder
- foundation should support a practical operator flow for purchasing labels across multiple eligible shipments
- however, do not broaden this into full Packing Workflows or future scorecard logic
