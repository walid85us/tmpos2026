==================================================
WORKSTREAM GOAL
==================================================

Add SLA-aware triggers and conditions to Shipping Automation so the system can react safely when SLA states change.

This pass should make SLA operationally connected without turning it into Carrier Scorecards.

Core idea:
- SLA detects state
- Automation can safely flag, prioritize, note, or queue follow-up
- No fake scorecards
- No automatic risky shipping/provider actions

==================================================
PART A — SLA AUTOMATION TRIGGERS
==================================================

Required:
Add safe SLA-related triggers to Automation Rules where supportable.

At minimum support:

1) SLA becomes At Risk
Trigger:
- fires when shipment SLA status changes to at_risk

2) SLA becomes Overdue
Trigger:
- fires when shipment SLA status changes to overdue

3) SLA is Missed
Trigger:
- fires when shipment SLA status changes to missed

4) SLA is Paused
Trigger:
- fires when SLA is paused

5) SLA is Resumed
Trigger:
- fires when SLA resumes from paused

6) Delay Reason Added
Trigger:
- fires when delay reason is added or changed

Required:
- triggers must fire from actual SLA state changes, not fake timer-only assumptions unless current architecture supports a scheduled recalculation safely
- if scheduled recalculation is not implemented, clearly document that triggers fire when SLA is recalculated/updated through existing app actions
- avoid duplicate firing for the same state transition

==================================================
PART B — SLA AUTOMATION CONDITIONS
==================================================

Required:
Add SLA-related conditions where supportable.

Examples:
- SLA status is at risk / overdue / missed / paused / unknown
- SLA target type is Pack By / Label By / Dispatch By / Deliver By / Return Receive By
- SLA delay reason exists
- SLA is paused
- SLA variance greater than X minutes/hours
- shipment source type / return vs outbound
- carrier / provider / service where already available
- shipment priority if available

Required:
- conditions must be type-aware and human-readable
- avoid raw internal tokens as final user-facing labels
- only expose conditions that current data model can support truthfully

==================================================
PART C — SAFE SLA AUTOMATION ACTIONS
==================================================

Required:
Use existing safe actions only, or add safe equivalents if needed.

Allowed actions:
- mark shipment for review
- set priority / escalate priority
- add internal note
- create follow-up marker / queue follow-up if existing foundation supports it
- queue for batch only where operationally logical
- add SLA delay review marker if supported
- add audit/timeline entry

Not allowed:
- auto-purchase label
- auto-dispatch
- auto-cancel shipment
- auto-change SLA policy
- fake carrier scorecard updates
- hidden provider operations

==================================================
PART D — DUPLICATE / LOOP PREVENTION
==================================================

Required:
SLA automation triggers must not fire repeatedly for the same unchanged SLA state.

Examples:
- if SLA is already overdue, do not repeatedly create new review outcomes every render
- if delay reason was already processed, do not duplicate the same automation outcome
- use stable event keys or transition keys where appropriate

Required:
- execution history should show one meaningful event per actual transition/change
- duplicate suppression should be auditable or deterministic

==================================================
PART E — EXECUTION HISTORY + SHIPMENT VISIBILITY
==================================================

Required:
SLA automation outcomes must be visible and understandable.

Execution History should show:
- SLA trigger name
- shipment number clickable
- old SLA state / new SLA state where available
- SLA target type
- action taken
- timestamp
- resolved state if it creates a review/outcome

Shipment detail should show:
- any review/priority/note outcome created by SLA automation
- no confusion between SLA panel and Automation Outcomes
- SLA panel remains the source for SLA facts
- Automation Outcomes remains the source for automation-created outcomes
