B) SHIPMENT DETAIL EXPERIENCE

- source document linkage

- package/parcels information
- carrier/provider section
- service/shipping method
- cost details
- tracking number and tracking timeline/events
- status history / audit-style visibility where patterns already exist
- notes/internal operational comments if supported by current architecture
- available actions based on permissions and lifecycle state

C) SHIPMENT CREATION FLOW
Users must be able to create shipments from:
- invoices
- repairs
- transfers
- RMA

Creation flow should prefill from the source document where possible:
- customer / recipient information
- relevant addresses
- item/reference information
- store/source location
- shipment purpose/type

Creation must still allow controlled editing where business logic permits.

D) SHIPMENT TYPES
Support a clear internal type model that distinguishes operational intent, such as:
- customer delivery shipment
- repair return shipment
- store transfer shipment
- RMA outbound / return shipment

This does not need to create unnecessary UI complexity, but the data model must support it cleanly.

E) PACKAGE / PARCEL MODEL
Support package entry and editing with at minimum:
- package count or parcel rows
- weight
- dimensions
- package notes or contents summary where appropriate
- optional declared value / insured value field if the current business model can support it cleanly

Avoid overengineering, but do not lock the platform into a single-package-only design.

F) PROVIDER / CARRIER ABSTRACTION
Implement a clean shipping provider abstraction layer.

If live integrations are not already present, support at least:
- manual carrier/provider selection
- manual service level entry or normalized service choice
- manual tracking number entry
- manual cost entry
- future adapter extension points

If there is already a shipping integration structure in the app, reuse and strengthen it rather than rebuilding inconsistently.

G) STATUS LIFECYCLE
Implement a disciplined shipment lifecycle aligned with the app’s lifecycle gating philosophy.

Recommended operational states:
- Draft
- Ready / Prepared
- Label Created or Booked
- Packed
- Dispatched / Handed Off
- In Transit
- Delivered
- Exception
- Cancelled

Use the most compatible naming with the existing codebase if similar status conventions already exist.
Transitions must be controlled and permission-aware.
Do not allow illogical backward editing without explicit gating.

H) TRACKING / EVENT MODEL
Support tracking visibility in a future-proof way.
At minimum allow:
- tracking number storage
- carrier/provider association
- event/timeline entries
- latest tracking state summary
- delivery/exceptions visibility

If live tracking does not exist, implement a manual event-capable structure rather than pretending live sync exists.

I) SOURCE LINKAGE + NON-DUPLICATION
A shipment must retain a durable link back to its source record.
Source records should be able to show:
- linked shipment existence
- quick access to shipment detail
- summarized shipping status if appropriate

Avoid duplicating full shipping state in multiple modules. Keep the shipping record authoritative.

J) SHIPPING DOCUMENTS / LABEL READINESS
If the application already has document handling patterns, add support for shipment-related artifacts in a compatible way.
Examples:
- label file metadata
- packing slip metadata
- shipment notes/documents

Do not force a full document management redesign.
