yet the shipment is not visibly marked in a meaningful operational way.

Current QA issue:
- the rule appears to fire
- execution history records it
- but there is no clear place where the shipment is marked for review
- no clear review queue / filter / badge / workflow exists
- no clear operator path exists to act on that rule outcome
- no clear resolution path exists once the shipment has been reviewed

Example from QA:
- rule says “When a carrier label is purchased and Has Carrier Label is true, mark the shipment for review”
- execution log shows the rule fired
- but the shipment is not visibly review-marked in a meaningful operational surface

Second example:
- rule says “When a carrier label is purchased and Shipping Cost greater than $6.00, mark the shipment for review”
- execution log shows actions: review_needed
- but there is no operational meaning for that result

This correction pass must solve that gap.

==================================================
PART A — DEFINE ACTION-TO-OUTCOME CONTRACT
==================================================

Required:
Audit each currently supported Automation Rule action and define its concrete operational outcome.

At minimum, make these actions real and visible if they already exist or are surfaced:
- mark review needed
- queue for batch
- set priority
- add internal note / audit note / timeline note
- any similar safe action currently exposed in the rules builder

Required:
For each action, define:
- what persistent shipment state/field changes
- where the operator sees that result
- how the result is cleared/resolved if applicable
- how it appears in shipment list / detail / queue surfaces
- how it appears in audit history

Do not leave any action as “execution history only” if the UI suggests it changed shipment state.

==================================================
PART B — MARK FOR REVIEW MUST BECOME A REAL OPERATIONAL STATE
==================================================

This is mandatory.

Required:
“Mark for review” / review_needed must create a real shipment review state.

At minimum support:
- visible review-needed badge/flag on the shipment
- visible review-needed state in shipment detail
- visible reason/source showing it came from automation rule
- visible rule name that caused it
- visible timestamp
- visible unresolved vs resolved state

Required operator flow:
- operator can find review-needed shipments from Shipping Center
- provide a practical surface such as:
  - filter
  - queue
  - badge
  - highlighted row state
  - dedicated section if appropriate
- operator must be able to open the shipment and understand why it was flagged
- operator must be able to resolve/clear the review-needed state with an auditable action and optional note

Important:
Do NOT overbuild a full approval engine in this pass.
The goal is a practical review/exception workflow, not enterprise BPM.

==================================================
PART C — QUEUE / FILTER / FINDABILITY
==================================================

Required:
If a shipment is flagged by automation, the operator must be able to find it easily.

At minimum:
- add meaningful filtering or queue behavior in Shipping Center for automation-created outcomes
- support finding:
  - review-needed shipments
  - ready-for-batch shipments if that action exists
  - prioritized shipments if that action exists

Required:
Do not make the user hunt through raw execution history to know what changed.

==================================================
PART D — EXECUTION HISTORY MUST MATCH REAL OUTCOME
==================================================

Required:
Execution History should remain as the audit trail, but it must now correspond to a real operational result.

Example:
If history says:
- action: review_needed
then the shipment must actually show:
- review-needed state
- rule source
- review resolution path

If history says:
- queued for batch
then the shipment must appear in the relevant batch-ready or queued state/surface

Avoid false logging where history implies an action succeeded but the shipment state did not actually change.

==================================================
PART E — RESOLUTION / CLEARING WORKFLOW
==================================================

Required:
For automation outcomes that create unresolved operator work, define a resolution path.

For review-needed:
- user with appropriate permission should be able to clear/resolve it
- resolution should be auditable
- optional resolution note should be supported if practical
- shipment should record:
  - resolved by
  - resolved at
  - resolution note if provided

Do not silently remove review state without audit trail.

==================================================
PART F — RULE ACTION EXPLANATION
==================================================

Required:
The action meaning should be clearer in the rules builder and on rule cards.

For example, “mark for review” should read/behave like:
- flag shipment for operator review
rather than a vague internal token

Where practical:
- show helper text explaining what each action does operationally
- make it clear whether the action:
  - flags the shipment
  - queues it
  - adds a note
  - changes priority
Do not leave operators guessing what the action means.
