==================================================
PART A — ADD-ON CATALOG GOVERNANCE
==================================================

Required:
Create or refine an Add-on Catalog model under System Owner controls.

At minimum, each add-on should support:

- addOnId
- name
- description
- linkedFeatureId or capability key
- default monthly price or default price
- billing cadence label if already supported
- status:
  - active
  - disabled
  - archived
- createdAt / updatedAt
- createdBy / updatedBy
- audit/history entries where appropriate

Required behavior:
- active add-ons can be offered to tenants
- disabled add-ons should not appear as available tenant add-ons
- archived add-ons should not appear in tenant feature/add-on selection surfaces
- archived/disabled add-ons should remain in System Owner catalog/history for audit, unless current app pattern permanently removes records
- do not delete historical tenant assignment/audit evidence silently

Important:
Archived or disabled add-ons must not appear under tenant Features tab as available selectable items.

==================================================
PART B — CORE FEATURE VS ADD-ON FEATURE DISTINCTION
==================================================

Required:
The UI must clearly distinguish between:

1) Plan/Core Feature
A feature included by the tenant’s plan.

2) Add-on Feature
A separately sellable feature controlled by the add-on catalog.

3) Trial Add-on
An add-on temporarily granted to a tenant.

4) Paid Override Add-on
An add-on granted to a tenant as a paid exception/override.

Required:
When a tenant feature is add-on based, the tenant feature surface must clearly indicate:
- Add-on
- Trial
- Paid Override
- Included by Plan, if applicable
- Not available, if plan/add-on status prevents it

Avoid vague or misleading labels.

==================================================
PART C — TENANT ADD-ON OVERRIDES
==================================================

Required:
Add or refine tenant-specific add-on override controls.

At minimum support:
- grant trial add-on
- grant paid override add-on
- revoke add-on override
- trial start date
- trial end date
- paid override price
- override status:
  - trial_active
  - trial_expired
  - paid_active
  - revoked
  - inactive
- notes / reason
- audit history

Required behavior:
- when granting paid override, default price should come from the Add-on Catalog default price
- System Owner must be able to edit tenant-specific paid override price
- edited override price must be persisted clearly
- tenant-specific price should not mutate the global add-on default price
- trial override should clearly show trial state and dates
- expired trial should not continue enabling the feature unless separately converted to paid override or included in plan

Do not fake billing collection.
This pass controls entitlement and commercial metadata only.

==================================================
PART D — TENANT FEATURES TAB VISIBILITY
==================================================

Required:
Fix tenant Features tab visibility behavior.

Rules:
1) If an add-on is archived, it should not appear under tenant Features tab as an available feature/add-on.
2) If an add-on is disabled, it should not appear under tenant Features tab as an available feature/add-on.
3) If a tenant already had an archived/disabled add-on assigned historically:
   - show it only where needed as historical/revoked/inactive evidence, not as an active available feature
   - do not let it continue enabling functionality
4) Active add-ons may appear as available for trial or paid override.
5) Tenant feature rows must clearly show whether the feature is:
   - Plan Feature
   - Add-on
   - Trial
   - Paid Override
   - Disabled / Not Available
   - Revoked / Expired if historical view exists

Important:
Tenant-facing/store-owner surfaces should not show unavailable add-ons as usable capabilities.

==================================================
PART E — ENTITLEMENT RESOLUTION MODEL
==================================================

Required:
Implement or refine a single entitlement resolution model.

A tenant feature should be enabled only if one of the following is true:
1) included by the tenant’s active plan
2) enabled by an active trial add-on override
3) enabled by an active paid add-on override

A tenant feature should be unavailable if:
- feature is disabled at plan level
- add-on is disabled or archived
- tenant override is expired/revoked/inactive
- plan/add-on relationship does not include it

Required:
The resolver should return a clear reason:
- included_by_plan
- enabled_by_trial_addon
- enabled_by_paid_addon
- disabled_by_plan
- addon_disabled
- addon_archived
- trial_expired
- override_revoked
- not_available

Use this reason to drive UI copy and permission filtering.

==================================================
PART F — STORE PERMISSIONS MATRIX INTEGRITY
==================================================

Required:
Store Permissions Matrix must continue to follow the accepted rule:

- plan/add-on entitlement decides whether capability exists
- role decides who can use it
- role cannot resurrect unavailable capabilities
- linked permissions disappear when the parent feature/capability is unavailable
- sub-permission counts reflect actual visible permissions

Apply this to add-on-driven features.

Examples:
- If an add-on feature is not included by plan and no active tenant trial/paid override exists, its permissions should not appear as normal assignable permissions.
- If an add-on is granted as active trial or paid override, linked permissions should appear normally, subject to role control.
- If an add-on is archived/disabled, linked permissions should disappear even if old tenant assignment exists.

PART H — PRICE DISPLAY AND EDITING
==================================================

Required:

- default price = add-on catalog price

- UI should show:

- if add-on catalog default price later changes, do not silently overwrite an already customized tenant override price unless explicitly designed

For trial overrides:
- price should not be shown as active paid amount
- trial state and dates should be clear

Do not implement payment charging or subscription billing in this pass.
