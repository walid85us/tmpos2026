==================================================
ROOT ISSUE ANALYSIS
==================================================

There are two separate remaining issues:

1) EasyPost test-mode tracking behavior
EasyPost test mode only accepts EasyPost test tracking codes such as:
EZ1000000001, EZ2000000002, ... , EZ7000000007
Ordinary shipment tracking numbers in test mode will not behave like live tracking sync.

This means the current Sync Tracking UX in test mode is still not ideal for QA.

2) Carrier-specific phone requirement for UPS/FedEx label purchase
When UPS or FedEx is selected and label purchase is attempted, the carrier/provider can fail with:
“ShipTo PhoneNumber must be at least 10 alphanumeric characters”
This should be prevented by UI preflight validation, not left to fail deep in provider/carrier response handling.
