==================================================
CRITICAL IAM / RBAC DESIGN RULE
==================================================

Implement a final least-privilege effective access model.

Do not treat feature visibility and section/action access as the same thing.

There are four levels of access checks:

1) Sidebar visibility
2) Page/route access
3) Section/widget visibility
4) Action/button/handler authorization

Each must be evaluated correctly.

--------------------------------------------------
A) Sidebar visibility
--------------------------------------------------

A sidebar item appears if:
- parent permission is View Only or higher
OR
- any child sub-permission under that feature is View Only or higher
OR
- System Owner Full Access applies

If parent is None and all children are None:
- sidebar item must disappear.

--------------------------------------------------
B) Page/route access
--------------------------------------------------

A direct page route opens if:
- parent permission is View Only or higher
OR
- at least one page-relevant child permission is View Only or higher
OR
- System Owner Full Access applies

If no effective access:
- show clean Access Denied.
- do not leak page data.

--------------------------------------------------
C) Section/widget visibility
--------------------------------------------------

This is the missing piece.

A page may open because one child permission grants access.
But each section/widget must check its own permission.

Example:
Command Center:
- view_operational_pulse controls Operational Pulse.
- view_needs_attention controls Needs Attention.
- view_tenant_360 controls Tenant 360 click-throughs.
- view_next_best_actions controls NBA list.
- act_on_nba_recommendations controls NBA action buttons.
- use_command_quick_actions controls quick actions.

If only view_needs_attention is granted:
- show only Needs Attention section.
- hide all other Command Center sections.

--------------------------------------------------
D) Action/button/handler authorization
--------------------------------------------------

Every mutation or workflow action must check its exact sub-permission.

Examples:
- change_support_status controls status dropdown/save.
- change_support_severity controls severity dropdown/save.
- assign_support_case controls assignment.
- add_internal_support_note controls adding internal notes.
- use_support_macro controls macro insertion.
- deescalate_support_case controls de-escalation.
- resolve_escalation controls resolve escalation.
- grant_trial controls Grant Trial / End Trial.
- grant_paid_override controls Paid Override.
- revoke_addon_override controls Revoke / End / Re-grant paid override.

UI button state is not enough.
Handlers must also check permission.

==================================================
PERMISSION LEVEL HIERARCHY
==================================================

Preserve exactly:

- None
- View Only
- Create
- Edit
- Approve
- Manage
- Full Access

Threshold behavior:
- View Only satisfies View Only
- Create satisfies View Only + Create
- Edit satisfies View Only + Create + Edit
- Approve satisfies View Only + Create + Edit + Approve
- Manage satisfies View Only + Create + Edit + Approve + Manage
- Full Access satisfies all

Least privilege:
- Create does not allow editing existing records.
- Edit does not allow approval-sensitive transitions.
- Approve allows approval-sensitive workflow transitions.
- Manage allows broad operational management.
- Full Access allows all actions.
- Explicit child None denies that child action/section even if parent is Manage/Full, except System Owner permanence if current design requires it.
