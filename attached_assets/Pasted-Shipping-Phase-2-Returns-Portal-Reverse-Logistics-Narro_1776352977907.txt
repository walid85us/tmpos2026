==================================================
CURRENT QA SOURCE OF TRUTH — REMAINING FAILURES
==================================================

These are the exact remaining blockers and must be fixed literally:

1) Return creation from source is not good enough
Observed QA result:
- Return creation exists, but the user wants it to work like Create Shipment using Source Reference Number
Required behavior:
- Return creation should support Source Reference Number resolution
- When the user enters/selects a valid source reference, the return form should resolve and prefill all relevant return data from that source
- This should work similarly to how shipment creation resolves source references

2) Return linkage is only partial until source resolution is improved
Required behavior:
- Return should clearly store and show the source reference number
- Source/customer/item linkage should be correct and visible after resolution
- The return detail should make the source reference easy to see and use

3) Return shipment / return status linkage is incorrect
Observed QA result:
- Return shows Label Created even while linked shipment still looks like draft / before actual label purchase
Required behavior:
- Return must not show Label Created until the linked return shipment label is actually purchased/created
- Do not advance the return lifecycle early
- Return lifecycle must follow the real linked shipment state where appropriate

4) Reverse-logistics lifecycle must synchronize with live shipment status
Required behavior:
- Once the return shipment label is created, shipment progress between Label Created and Delivered should follow the live linked shipment/tracking status
- Shipping Center and Returns Portal should stay synchronized
- Return should not require duplicate manual progression for the shipping leg where shipment status already exists

5) Received must be blocked until linked shipment is Delivered
Required behavior:
- The return status “Received” must not be enabled until the linked return shipment status shows Delivered
- Once shipment is Delivered, the operator can then move the return to Received and begin intake/inspection
- Do not allow Received prematurely

6) Return label delivery to customer needs an operational path
Required behavior:
- Add a practical operator path to provide the return label to the customer
- At minimum, support one or both of:
  - send the label to the customer by email
  - provide a usable carrier/label reference the customer can use with the carrier office to print the label
- Keep it operational and realistic
- Do not overbuild a full notification system if not needed, but there must be a real usable path

NON-NEGOTIABLE IMPLEMENTATION REQUIREMENTS
==================================================

1. Source Reference Number return creation flow
Required:
- Add Source Reference Number as a first-class return creation mechanism
- Resolve and prefill return data from the source record
- Reuse the strongest patterns from shipment source resolution where appropriate
- Prefill relevant fields such as:
  - customer
  - source type
  - source reference
  - linked shipment/invoice/repair/RMA context
  - item information where available
- Do not make the operator retype data that the source record already knows

Acceptance:
- QA can create a return by using Source Reference Number and sees the return prefilled correctly

2. Source reference visibility
Required:
- Show Source Reference Number clearly on the return
- Keep linkage visible in return detail and relevant list/detail surfaces
Acceptance:
- QA can clearly see and verify the source reference on the return

3. Correct return shipment / return lifecycle synchronization
Required:
- Return must not move to Label Created until the linked return shipment label is truly created/purchased
- If the linked shipment is still draft/not purchased, the return must not falsely show Label Created
- Keep the return and return shipment relationship truthful and state-driven

Acceptance:
- QA no longer sees return Label Created before the linked label actually exists

4. Live shipment-status-driven reverse shipping progression
Required:
- For the return shipping leg, the return should reflect the live linked shipment status where appropriate
- Between label creation and delivered, the reverse-shipment movement should be driven by the linked shipment/tracking status rather than separate conflicting manual return states
- Shipping Center and Returns Portal must stay coherent

Acceptance:
- QA can observe linked shipment status driving the reverse-logistics shipping portion coherently

5. Received gating
Required:
- Received must remain disabled until linked return shipment status is Delivered
- After Delivered, Received becomes available
- Intake/inspection then follows normally

Acceptance:
- QA cannot set Received before shipment Delivered
- QA can set Received after shipment Delivered

6. Return label delivery to customer
Required:
- Add an operator path to provide the return label to the customer
- Acceptable implementations:
  - Send Label by Email action
  - or a clean customer-facing reference/instructions path using shipment/label/tracking/carrier details
- Keep it lightweight and operational
- If email is implemented, keep it consistent with the app’s current mail/send patterns if any exist
- If not, provide a clearly usable carrier/label reference path

Acceptance:
- QA can see a real operator option to get the return label to the customer
