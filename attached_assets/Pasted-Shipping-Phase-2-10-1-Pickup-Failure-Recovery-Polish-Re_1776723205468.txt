CURRENT QA SOURCE OF TRUTH
==================================================

Phase 2.10 now passes.

However, the remaining polish need is:
- when pickup is not confirmed / booking failed, the UI should not leave Cancel Pickup as the main recovery path
- operator should be guided to edit date/window or other blocking details and retry
- FedEx "Pickup Date too far in future" should have clearer recovery guidance
- remaining untouched state / ZIP / phone fields across the app should be audited and normalized if any gaps remain

==================================================
NON-NEGOTIABLE REQUIRED IMPROVEMENTS
==================================================

1) FAILED / UNCONFIRMED PICKUP RECOVERY UX
Required:
- for pickup objects that exist but are not confirmed, make the primary recovery path:
  - Edit pickup date/window
  - Edit pickup details
  - Retry booking
- keep Cancel Pickup available when an orphan provider pickup exists, but make it a secondary recovery action
- do not present the failed/unconfirmed state as if the pickup is booked

2) FUTURE-DATE / PROVIDER-REFUSAL GUIDANCE
Required:
- when provider returns a specific date/window-style reason such as:
  - PICKUPDATE.TOO.FAR
  - no pickup rates for the selected date/window
- show a clear operator hint such as:
  - "Try an earlier pickup date"
  - "Try a different pickup window"
  - "Carrier/account/date combination may not support that future date"
- keep the provider reason visible, but add a clear operator action path

3) FAILED / UNCONFIRMED PICKUP ACTION HIERARCHY
Required:
- for failed/unconfirmed pickup states:
  - primary CTA = edit date/window/details and retry
  - secondary CTA = cancel orphan pickup
- do not make cancellation the only obvious next action when the better next step is correction + retry

4) REMAINING APP-WIDE INPUT AUDIT
Required:
- audit remaining state / ZIP / phone inputs across the app
- identify any remaining structured address/contact inputs not yet normalized
- apply the same normalization rules where gaps still exist
- if some inputs are intentionally excluded, document why

5) DO NOT CHANGE UNDERLYING TRUTHFULNESS
Required:
- do not weaken the truthful distinction between:
  - no-rates
  - booking failed
  - confirmed pickup
- do not weaken USPS/FedEx/UPS truthfulness already achieved
