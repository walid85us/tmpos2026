==================================================
CURRENT QA SOURCE OF TRUTH
==================================================

Observed QA follow-up issues:

1) FedEx future-window behavior
- next-day pickup can succeed
- a pickup for 2 days ahead can fail
- this should not be treated as a blanket “test mode only” rule without evidence
- the app must surface this honestly as a provider/account/window outcome, not a vague failure

2) Pickup cancellation UX/integrity
- clicking Cancel Pickup currently gives no confirmation modal
- in some cases it appears nothing happens
- operator needs explicit confirm/cancel interaction and truthful final status

3) Input hardening across the app
- state fields should be 2-letter state code only
- ZIP code fields should accept numeric input only
- phone number fields should accept numeric input only
- this should be applied consistently wherever those inputs appear in the app

4) Shipment cancel dependency on confirmed pickup
- currently a shipment can be cancelled while a pickup remains confirmed
- this is operationally wrong
- shipment cancellation must either:
  A) be blocked until pickup is cancelled first
  OR
  B) explicitly cascade and cancel the pickup first, with user confirmation
- the app must choose one consistent, truthful rule

==================================================
NON-NEGOTIABLE REQUIRED FIXES
==================================================

1) PICKUP CANCELLATION CONFIRMATION + OUTCOME
Required:
- when operator clicks Cancel Pickup, show a confirmation modal
- modal must explain what will happen
- on success, show a clear success state/message
- on failure, show a clear failure state/message
- do not leave the operator with “nothing happened”

2) FUTURE-WINDOW / NO-RATES / PROVIDER-REFUSAL TRUTHFULNESS
Required:
- distinguish clearly between:
  - no pickup rates for this date/window/account/provider combination
  - provider created pickup object but booking failed
  - provider rejected/could not confirm the booking
- do not imply a blanket carrier rule if the result is actually date/window/account-specific
- FedEx two-day failure should be surfaced as a truthful provider/window outcome

3) STATE CODE INPUT HARDENING
Required:
- all U.S. state inputs across the app must use 2-letter state code format only
- normalize to uppercase
- reject longer free-text state names in structured address inputs
- apply consistently wherever shipment/customer/store/etc. addresses are entered

4) ZIP + PHONE INPUT HARDENING
Required:
- ZIP code fields: numeric characters only
- phone number fields: numeric characters only
- if formatting/masking is used, underlying stored value must stay valid and normalized
- apply consistently across the app

5) SHIPMENT CANCEL DEPENDENCY ON CONFIRMED PICKUP
Required:
- choose and implement one correct operational rule:
  OPTION A: Disable shipment cancellation while pickup is confirmed/scheduled until pickup cancellation succeeds
  OR
  OPTION B: Shipment cancel flow becomes a confirm dialog that explicitly says pickup will also be cancelled first, and shipment cancellation only completes if pickup cancellation succeeds
- recommend and implement the better option for operational integrity
- do not allow shipment to show cancelled while pickup remains confirmed

6) KEEP EXISTING TRUTHFULNESS
Required:
- preserve USPS success
- preserve FedEx truthful success/failure
- preserve UPS unsupported-service gating
- preserve no-rates vs partial_failed distinction
