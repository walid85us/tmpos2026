==================================================
WORKSTREAM GOAL
==================================================

Add a manual backfill option for SLA Automation Rules so an authorized user can intentionally apply an SLA rule to shipments that already match its conditions.

Example:
- A shipment is already overdue for Pack By SLA
- User creates rule: “When SLA becomes Overdue, mark shipment for review”
- Default behavior should still not fire retroactively
- But user can click:
  “Apply this rule to existing matching shipments”
- The app scans matching shipments, shows count/result, applies the rule safely, and audits the backfill

==================================================
PART A — EXPLICIT BACKFILL ACTION
==================================================

Required:
Add an explicit backfill action on applicable SLA automation rules.

Suggested label:
- “Apply to Existing Matching Shipments”
or
- “Backfill Existing Matches”

Required:
- show this action only for SLA automation rules where backfill is meaningful
- do not show it for unrelated non-SLA automation rules unless there is a clear reason
- do not auto-run it when a rule is created
- do not auto-run it when a rule is enabled
- user must click it intentionally

==================================================
PART B — CONFIRMATION STEP
==================================================

Required:
Before running backfill, show an in-app confirmation modal.

The modal must explain:
- this action will evaluate existing shipments
- it may create automation outcomes for multiple shipments
- it will not purchase labels, dispatch shipments, or perform risky provider actions
- it will be audited

Required:
Show a preview/count if practical:
- number of matching shipments found
- rule name
- SLA trigger/condition being evaluated
- action that will be applied

If exact count cannot be calculated before confirmation, explain that clearly and show itemized results after run.

Do not use browser-native confirm.

==================================================
PART C — MATCHING LOGIC
==================================================

Required:
Backfill must evaluate current shipment state against the selected SLA automation rule.

Supported examples:
- current SLA status is at_risk
- current SLA status is overdue
- current SLA status is missed
- SLA is paused
- delay reason exists
- SLA target type matches selected condition
- return/outbound context matches selected condition
- carrier/provider/service/source conditions match where available

Required:
- use the same condition logic as the rule engine where possible
- do not use a weaker or separate inconsistent matcher
- if a rule cannot be safely backfilled, disable the backfill action and show why

==================================================
PART D — SAFE ACTION EXECUTION
==================================================

Required:
Backfill can only execute safe SLA automation actions already allowed by the SLA Automation Linkage scope, such as:
- mark shipment for review
- set priority
- add internal note
- create follow-up marker / queue follow-up if supported
- add audit/timeline entry

Not allowed:
- auto-purchase label
- auto-dispatch
- auto-cancel shipment
- auto-change SLA policy
- fake scorecard update
- hidden provider operations

==================================================
PART E — ITEMIZED RESULTS
==================================================

Required:
After running backfill, show itemized results.

Each evaluated shipment should show:
- shipment number
- matched / skipped / already processed / failed
- reason
- action applied
- link/open action to shipment where possible

Required summary:
- total evaluated
- total matched
- total applied
- total skipped
- total failed

==================================================
PART F — DUPLICATE / RE-RUN SAFETY
==================================================

Required:
Backfill must be duplicate-safe.

If a shipment already has an automation outcome from:
- the same rule
- the same SLA target/context
- the same backfill or transition key

then do not create a duplicate.

Show result as:
- skipped — already processed
or
- already has active outcome

Required:
- generate stable backfill keys / context keys
- do not create duplicate review outcomes on repeated backfill clicks
- repeated backfills should be safe and predictable

==================================================
PART G — AUDITABILITY
==================================================

Required:
Backfill run must be auditable.

Record:
- ruleId
- ruleName

- runAt
- number evaluated
- number applied
- number skipped
- number failed
- per-shipment results where practical
- reason/context

Shipment-level history should also show:
- SLA automation backfill applied
- rule name
- action applied
- timestamp
- actor
