==================================================
WORKSTREAM GOAL
==================================================

Add a real Returns Portal / Reverse Logistics capability that fits this app’s existing repair/POS/shipping architecture.

The result should let the business:
- create and manage customer/product returns in a structured way
- initiate returns from appropriate sources such as invoices, repairs, shipments, and RMA contexts
- generate and manage return shipments/return labels where provider mode is used
- track return progress through a coherent reverse-logistics lifecycle
- capture reasons, condition, requested outcome, and intake decisions
- keep return statuses synchronized with shipping/tracking activity where appropriate

This workstream must also explicitly prepare the architecture for remaining Phase 2 items:
- service points
- customs docs
- pickup requests
- insurance rules
- carrier analytics

Do not fully implement all remaining Phase 2 items now.
Do make this returns workstream architecturally compatible with them.

==================================================
PRODUCT DIRECTION — REQUIRED
==================================================

The app should now support a first-class return flow with two complementary surfaces:

1) Returns Portal / Return Intake Experience
A dedicated return initiation/intake flow where returns can be created and managed clearly.

2) Reverse Logistics Operational Management
Shipping Center and related operational modules should understand and manage return shipments, return status, and intake outcomes coherently.

This workstream must feel native to the product and not like a disconnected mini-app.

==================================================
MANDATORY IMPLEMENTATION OBJECTIVES
==================================================

1) RETURNS OBJECT / DOMAIN MODEL
Add a robust returns domain model that can support:
- return reference number
- source linkage
- customer linkage
- original shipment/order/invoice/repair/RMA linkage where applicable
- return reason
- return category/type
- requested resolution/outcome
- item-level return contents where applicable
- return shipment linkage
- intake/inspection results
- disposition outcome
- audit timestamps / status history metadata

The model must be forward-compatible with future:
- service points
- customs docs
- pickup requests
- insurance rules
- carrier analytics

2) RETURN SOURCES / ENTRY POINTS
Support returns initiation from the most relevant operational sources.

At minimum, evaluate and implement the correct entry points where they fit the current system:
- invoice-related customer return
- shipment-related return
- repair-related return / send-back / reverse flow
- existing RMA/reverse service context

Do not force every source if the current app structure does not support it cleanly.
But the app must have a coherent path to initiate and manage returns, not just a hidden admin record.

3) RETURNS PORTAL / INTAKE EXPERIENCE
Implement a dedicated returns interface.

Minimum expectations:
- a dedicated Returns area or Returns Portal surface inside the product
- ability to create / review / edit / process returns
- clear statuses
- return reason selection
- requested resolution selection
- visible linkage to source documents and shipments
- clean operator workflow for reverse logistics

If you decide to include a customer-facing or external-share return request surface, keep it minimal and secure.
Do not overbuild a public portal unless it clearly fits the current app.
A strong internal portal/workspace is acceptable for this pass.

4) RETURN SHIPMENT / REVERSE SHIPPING INTEGRATION
Returns must integrate with shipping cleanly.

Required:
- return shipments must fit the Shipping Center architecture
- provider mode/manual mode rules must remain coherent
- return labels should be supported where appropriate in provider mode
- reverse shipment tracking must be visible and lifecycle-aware
- return shipping must not break the existing outbound shipping model

5) RETURN LIFECYCLE / REVERSE LOGISTICS STATUS MODEL
Implement a coherent reverse-logistics lifecycle.

At minimum, define and support statuses such as:
- Draft
- Requested / Initiated
- Approved / Authorized
- Label Created
- In Transit Back
- Received
- Inspecting / Processing
- Completed / Resolved
- Rejected / Closed / Cancelled where appropriate

Do not force these exact names if your app architecture needs slightly different wording.
But the lifecycle must be structured, audit-friendly, and operationally clear.

6) RETURN REASONS / RESOLUTION OUTCOMES
Returns are not just shipments.
They need business meaning.

Required:
- structured return reason selection
- optionally categorized reasons if that fits
- requested outcome / resolution capture, such as:
  - refund
  - exchange
  - repair
  - store credit
  - inspection only
  - send back to customer
- clear final disposition/outcome after intake

7) INTAKE / INSPECTION / DISPOSITION
Reverse logistics must not end at label creation.

Required:
- support receiving the return
- support intake inspection state
- support condition/result capture
- support final disposition
- keep this lightweight but real
- do not overbuild warehouse software, but do not stop at “return label created”

8) SHIPPING / TRACKING / EVENT SYNCHRONIZATION
Required:
- return shipment tracking must stay synchronized where provider mode is used
- webhook/event automation should be compatible with return shipments
- bulk sync/reconciliation should not break return shipments
- reverse-logistics statuses should update coherently when return shipment events arrive

9) AUDITABILITY
Required:
- return creation, status changes, labels, intake results, and disposition outcomes must be auditable
- source linkage and return-shipment linkage must remain clear
- operators must understand what happened and why
