==================================================
PART A — PICKUP FORECAST AT GET RATES
==================================================

PRODUCT GOAL

At Get Rates time, the operator should be able to see which returned carrier/service options are:
- likely pickup-capable
- pickup-capable but require a later live pickup check
- account/setup dependent
- not suitable for pickup
- unknown until pickup check

IMPORTANT:
Do NOT present this as final pickup eligibility.
This is a Pickup Forecast / Pickup Readiness indicator at rating time.
The real pickup booking flow remains the final source of truth.

NON-NEGOTIABLE REQUIRED FEATURES

1) ADD PICKUP FORECAST TO RATE RESULTS
Required:
- when rates are displayed, each rate row/card must show a pickup forecast indicator
- it must be tied to the actual returned carrier/service level
- it must help the operator compare rates without clutter

Suggested states:
- Likely pickup-capable
- Pickup-capable — final check after label purchase
- Account/setup required
- Not pickup-capable
- Unknown until pickup check

2) CARRIER-AWARE FORECAST ENGINE
Required:
- build a carrier-aware pickup forecast model using:
  - active provider
  - returned carrier
  - returned service level
  - provider capability model
  - known carrier/provider setup constraints
  - current provider settings
- do not use random cosmetic tags

At minimum, account for:
- USPS
- UPS
- FedEx
- DHL if currently surfaced in this flow

3) DO NOT OVERSTATE CERTAINTY
Required:
- do not label a rate as final “pickup eligible” at Get Rates time unless there is a real basis for certainty
- use forecast/probability/readiness wording instead
- later pickup flow remains the final truth

4) CLEAR HELP TEXT
Required:
- add a concise explanation:
  "Pickup forecast is an estimate based on carrier, service, and provider setup. Final pickup eligibility is confirmed when the carrier pickup request is created."

==================================================
PART B — TWO-STAGE ADDRESS READINESS WORKFLOW
==================================================

PRODUCT GOAL

The address workflow in Shipping must become an explicit two-stage process:

Stage 1: Check Address
- this marks the address as corrected / reviewed
- this is not yet fully ready for rating

Stage 2: Validate Address
- this marks the address as validated
- only validated addresses may unlock Get Rates

This must apply cleanly to the shipping address workflow and must not reintroduce prior regressions.

NON-NEGOTIABLE REQUIRED FEATURES

1) ADDRESS STATE MACHINE
Required:
- implement an explicit address-state model for the relevant shipping addresses
- at minimum support:
  - unchecked
  - corrected
  - validated
  - stale_after_edit

Apply this to the shipping address workflow in a way that is clear in the UI.

2) TWO-STAGE ACTION FLOW
Required:
- Check Address must run first
- once the address is corrected/reviewed, Validate Address becomes the next step
- once the address is validated:
  - Validate Address must be disabled
  - Get Rates must be enabled
- if the address changes later:
  - validation state must be invalidated
  - Get Rates must be disabled again
  - Check/Validate flow must be required again

3) GET RATES GATING
Required:
- Get Rates must only be enabled when the required shipping addresses are in validated state
- do not enable Get Rates merely because an address was corrected
- corrected is not enough; validated is required

4) RATE INVALIDATION AFTER ADDRESS CHANGE
Required:
- if the operator changes any rating-relevant address after validation and/or after rate selection:
  - previously selected rate must be invalidated
  - Get Rates must be disabled until the address goes through Check Address + Validate Address again
  - downstream steps that depend on the selected rate must not remain falsely valid

5) VALIDATE ADDRESS BUTTON BEHAVIOR AFTER RATE SELECTION
Required:
- once a valid rate is selected and the address remains unchanged:
  - Validate Address stays disabled
- if the address changes after a rate is selected:
  - Validate Address becomes relevant again through the required Check → Validate sequence
  - selected rate becomes stale and must not remain trusted

6) CLEAR UI WORDING
Required:
- the operator must clearly understand the difference between:
  - Check Address
  - Validate Address
  - Get Rates
- corrected and validated must not be visually or semantically conflated

7) DO NOT BREAK EXISTING SHIPPING FLOW
Required:
- preserve current flow integrity for:
  - rate selection
  - label purchase
  - pickup
  - tracking
  - reverse logistics

PART C — HOW BOTH PARTS MUST WORK TOGETHER
==================================================

1) Pickup Forecast appears only after valid rate results exist
Required:
- no pickup forecast should appear before Get Rates returns actual results

2) If address changes and rates are invalidated
Required:
- pickup forecast must also clear/reset with the stale rates
- do not let old pickup forecast remain attached to stale rates

3) Later pickup flow remains final source of truth
Required:
- the forecast can guide the operator
- the real pickup create/buy flow remains authoritative
