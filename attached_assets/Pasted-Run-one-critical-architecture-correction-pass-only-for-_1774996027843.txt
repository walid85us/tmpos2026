Required global permission levels:

- Manage
- Approve

These levels must become the standard permission model across the system.

Global behavior model to enforce for every permission domain:
- None:
  no access, no visibility of restricted actions, no execution
- View only:
  can view allowed module/list/detail surfaces only
  cannot create, edit, manage, approve, or execute restricted actions
- Create:
  can create/initiate new records/transactions in that domain
  does NOT automatically gain edit/manage/approve/full powers
- Edit:
  can edit/update allowed existing records in that domain
  does NOT automatically gain approve/full powers
- Manage:
  can perform operational management actions in that domain
  such as assignment, workflow handling, queue handling, state movement where appropriate
  does NOT automatically imply Approve unless explicitly granted
- Approve:
  can approve/reject/final-authorize actions in that domain where approval applies
  does NOT automatically imply unrelated manage/edit/full powers unless explicitly intended
- Full Access:
  includes all capabilities in that domain

Important:
- Lower levels must NOT silently behave like higher levels
- Create must not imply Approve
- View only must not imply Create
- Manage must not automatically imply Approve unless explicitly designed and documented
- Approve must not automatically imply unrelated management/edit powers unless explicitly designed
- Full Access may include all lower capabilities for that domain
- Apply this same hierarchy pattern to the entire permissions system, not selectively
